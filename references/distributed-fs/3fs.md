## 架构设计

### 整体架构

3FS 是一个面向 AI 训练场景设计的**高性能共享分布式文件系统**，采用经典的 **分离式架构（Disaggregated Architecture）**，将元数据服务与数据服务彻底解耦，以支持大规模并发 I/O 和高吞吐量。

```
┌─────────────────────────────────────────────────────────┐
│                    Client Nodes                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐               │
│  │ Trainer  │  │ Trainer  │  │ Trainer  │  ...           │
│  │ (FUSE)   │  │ (FUSE)   │  │ (FUSE)   │               │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘               │
│       │              │              │                     │
└───────┼──────────────┼──────────────┼─────────────────────┘
        │              │              │
   ┌────▼─────┐  ┌────▼─────┐  ┌────▼─────┐
   │ Meta RPC │  │ Meta RPC │  │ Meta RPC │  ← 元数据请求
   └────┬─────┘  └────┬─────┘  └────┬─────┘
        │              │              │
┌───────▼──────────────▼──────────────▼─────────────────────┐
│                    Meta Service Cluster                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐           │
│  │  MetaNode  │  │  MetaNode  │  │  MetaNode  │           │
│  └────────────┘  └────────────┘  └────────────┘           │
└───────────────────────────────────────────────────────────┘
        │              │              │
   ┌────▼─────┐  ┌────▼─────┐  ┌────▼─────┐
   │ Data RPC │  │ Data RPC │  │ Data RPC │  ← 数据请求
   └────┬─────┘  └────┬─────┘  └────┬─────┘
        │              │              │
┌───────▼──────────────▼──────────────▼─────────────────────┐
│                    Data Service Cluster                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐           │
│  │ DataNode   │  │ DataNode   │  │ DataNode   │  ...      │
│  │ (SSD/HDD)  │  │ (SSD/HDD)  │  │ (SSD/HDD)  │           │
│  └────────────┘  └────────────┘  └────────────┘           │
└───────────────────────────────────────────────────────────┘
```

### 节点角色

| 角色 | 职责 | 特点 |
|------|------|------|
| **Client（FUSE 客户端）** | 通过 FUSE 将 3FS 挂载为本地文件系统，拦截 POSIX 调用并转发到服务端 | 轻量级，无状态；每个训练节点运行一个 Client 实例 |
| **MetaNode（元数据节点）** | 管理目录树、文件属性、权限、文件布局（layout）等元数据 | 无状态设计 + 后端持久化存储；可水平扩展 |
| **DataNode（数据节点）** | 实际存储文件数据块，处理数据读写请求 | 每个 DataNode 管理本地磁盘；支持多盘并发 |

### 数据流

1. **写路径**：Client 向 MetaNode 请求文件分配 → MetaNode 返回文件布局（chunk 分布） → Client 直接与目标 DataNode 建立连接写入数据 → DataNode 落盘并返回 ACK → Client 通知 MetaNode 更新文件元数据
2. **读路径**：Client 向 MetaNode 查询文件布局 → Client 并行从对应 DataNode 读取数据 → 支持预读和缓存
3. **关键优化**：Client 与 DataNode 之间**直接通信**（Data Path），绕过 MetaNode，避免了控制路径成为瓶颈

### 控制流

- MetaNode 负责所有元数据操作（lookup、create、unlink、rename、setattr 等）
- 控制流与数据流分离，确保元数据操作不影响数据吞吐
- 客户端元数据缓存减少 RPC 次数，提升小文件操作性能

## 核心技术

### 1. Chunk-based 数据布局

3FS 将文件划分为固定大小的 **Chunk**（默认 128MB），每个 Chunk 作为一个独立的数据分配和调度单元：

- **大 Chunk 设计**：减少了元数据开销，适合 AI 训练场景中的大文件顺序读写
- **Chunk 分布**：文件的不同 Chunk 可以分布在不同的 DataNode 上，天然支持并行读写
- **Chunk 内分片**：大 Chunk 内部进一步细分为更小粒度的 slice/segment，支持细粒度随机访问

### 2. 无状态 MetaNode 设计

3FS 的 MetaNode 采用**无状态架构**：

- MetaNode 本身不持久化数据，所有元数据存储在独立的 **持久化存储层**（如基于 RocksDB 的嵌入式存储）
- 无状态特性使得 MetaNode 可以像无状态微服务一样快速扩缩容和故障恢复
- 任何 MetaNode 实例都可以处理任意元数据请求，便于负载均衡

### 3. 多副本与纠删码

3FS 支持两种数据冗余策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **多副本（Replication）** | 每个 Chunk 存储 2-3 个副本，分布在不同的 DataNode 上 | 热数据、频繁访问的训练数据集 |
| **纠删码（Erasure Coding）** | 将 Chunk 编码为 k 个数据块 + m 个校验块，容忍任意 m 个块丢失 | 冷数据、归档数据，存储空间利用率高 |

### 4. RDMA 与高性能网络

3FS 针对高性能网络进行了深度优化：

- 支持 **RDMA（RoCE v2）** 作为主要网络传输协议，大幅降低延迟、提升带宽
- 同时支持 TCP 作为备选传输方式
- RDMA 零拷贝技术减少了 CPU 开销和上下文切换
- 利用 NIC 的硬件能力进行数据搬运，释放 CPU 资源

### 5. 面向 AI 训练场景的优化

3FS 的核心设计目标是服务 AI/ML 训练工作负载：

- **大吞吐顺序读**：训练时所有 GPU 节点并行读取训练数据，3FS 通过 Chunk 分布和并行 I/O 提供聚合带宽
- **Checkpoint 快速写入**：训练过程中周期性保存模型状态（checkpoint），要求极高的并发写带宽
- **小文件优化**：对于包含大量小文件的场景（如图像数据集），通过客户端缓存和批量元数据操作降低延迟
- **多租户隔离**：支持不同训练任务间的资源隔离和 QoS 保障

### 6. 客户端缓存与预读

- **元数据缓存**：客户端缓存目录条目和文件属性，减少对 MetaNode 的 RPC 调用
- **数据缓存**：支持本地页面缓存，热数据优先从本地内存读取
- **自适应预读**：根据 I/O 模式动态调整预读策略，提升顺序读性能

## 元数据管理

### 元数据模型

3FS 的元数据管理涵盖：

- **目录树结构**：标准的层级目录组织
- **文件属性**：大小、权限、时间戳、所有者等
- **文件布局（File Layout）**：记录文件 Chunk 的分布信息（哪些 Chunk、每个 Chunk 在哪些 DataNode 上、副本位置等）
- **租约管理（Lease）**：用于保证元数据操作的分布式一致性

### 存储引擎

- 元数据存储在 **RocksDB** 中，利用其 LSM-Tree 结构提供高吞吐写入
- 使用合理的 key 设计（目录前缀 + 文件名）来优化目录列表操作
- 支持元数据的快照和回滚

### 分片与扩展

- 元数据按**目录前缀**或**命名空间**分片到不同的 MetaNode
- 支持在线元数据分片迁移，实现 MetaNode 的动态扩缩容
- 无状态设计使得添加新 MetaNode 几乎不需要数据搬迁

## 一致性模型

### CAP 取舍

3FS 在 CAP 三角中的取舍偏向 **CP（一致性与分区容忍性）**：

- **强一致性**：元数据操作保证强一致性，避免训练过程中的数据冲突
- **分区容忍**：在网络分区情况下，系统继续服务但保证已提交操作的一致性
- **最终一致性**：部分客户端缓存场景下允许短暂的最终一致性

### 共识机制

- 元数据的一致性通过**租约（Lease）机制**保障
- 写入操作通过**版本号/序列号**实现乐观并发控制
- 对于关键的元数据操作（如 rename），使用分布式锁来保证原子性
- 多副本写入采用**主从复制**模式，确认大多数副本成功后才返回

### 客户端一致性

- 客户端缓存通过租约失效机制保持一致性
- 当文件被其他客户端修改时，通过 MetaNode 通知持有租约的客户端使缓存失效
- 支持强制刷新（fsync）确保数据持久化

## 容错与高可用

### MetaNode 容错

| 机制 | 说明 |
|------|------|
| **无状态设计** | MetaNode 崩溃后可快速重启，无需数据恢复过程 |
| **负载均衡** | 请求自动路由到健康的 MetaNode 实例 |
| **多活部署** | 多个 MetaNode 同时服务，避免单点故障 |
| **元数据持久化** | RocksDB 本地持久化 + 远程复制，防止数据丢失 |

### DataNode 容错

| 机制 | 说明 |
|------|------|
| **多副本** | 数据在多个 DataNode 上有冗余副本，单节点故障不影响可用性 |
| **自动修复** | 检测到副本不足时自动触发重新复制（Re-replication） |
| **健康检查** | 心跳机制监测 DataNode 状态，超时标记为不可用 |
| **磁盘故障隔离** | 单个磁盘故障不影响同一 DataNode 上的其他磁盘 |
| **纠删码恢复** | EC 模式下通过剩余数据块重建丢失的块 |

### 故障恢复流程

1. **检测**：通过心跳和超时检测故障节点
2. **隔离**：将故障节点从服务列表中移除
3. **修复**：后台启动数据复制/重建任务
4. **恢复**：新节点加入后自动接收数据分配

### Client 容错

- 客户端支持自动重试和故障节点跳过
- 写入操作使用原子性保证，避免部分写入导致数据损坏
- 断线重连后自动重新获取文件布局信息

## 性能特征

### 典型基准数据

> 以下数据基于论文和公开报告中披露的测试环境，实际性能取决于硬件配置和 I/O 模式。

| 场景 | 指标 | 说明 |
|------|------|------|
| **聚合读取带宽** | 数十至数百 GB/s | 多客户端并行顺序读取大文件 |
| **聚合写入带宽** | 数十至数百 GB/s | 多客户端并行写入（如 Checkpoint） |
| **元数据操作延迟** | 亚毫秒级 | 缓存命中场景下的 lookup/create |
| **IOPS** | 数十万级 | 小文件随机读场景 |

### 与同类系统的对比

| 系统 | 优势对比 |
|------|----------|
| **vs NFS** | 3FS 提供原生分布式并行 I/O，NFS 受限于单服务器瓶颈 |
| **vs Lustre** | 3FS 的无状态 MetaNode 设计使运维更简单，容错恢复更快 |
| **vs CephFS** | 3FS 在 AI 训练场景下（大吞吐、大文件）性能更优，延迟更低 |
| **vs GPFS/Spectrum Scale** | 3FS 更轻量，面向云原生和容器化部署优化 |
| **vs JuiceFS** | 3FS 自研底层协议栈而非依赖对象存储，端到端延迟更低 |

### 性能瓶颈分析

- **元数据热点**：大量并发小文件操作可能使单个目录成为热点
- **网络带宽**：聚合 I/O 带宽受限于网络拓扑（通常使用 Spine-Leaf 架构）
- **磁盘 IOPS**：单盘性能限制单个 DataNode 的 IOPS 上限
- **CPU 开销**：RDMA 场景下 CPU 开销较低，但 TCP 模式下 CPU 可能成为瓶颈

## 适用场景

### 最佳实践

| 场景 | 适配度 | 说明 |
|------|--------|------|
| **AI/ML 模型训练** | ⭐⭐⭐⭐⭐ | 核心设计目标，大吞吐并行读 + 快速 Checkpoint |
| **大规模数据处理** | ⭐⭐⭐⭐⭐ | 批处理任务的中间数据存储和交换 |
| **深度学习数据集存储** | ⭐⭐⭐⭐⭐ | 支持 ImageNet 等大规模数据集的高效读取 |
| **模型推理服务** | ⭐⭐⭐⭐ | 需要加载大型模型文件时提供高带宽 |
| **日志收集与分析** | ⭐⭐⭐ | 适合大文件写入，小文件场景需优化 |
| **通用 POSIX 文件共享** | ⭐⭐ | 可用但非最优，传统 NFS/SMB 更轻量 |

### 局限性

1. **部署复杂度**：需要专门的运维知识，MetaNode/DataNode 的规划和调优
2. **硬件依赖**：最优性能需要 RDMA 网络支持，非标准以太网环境下性能降级
3. **小文件场景**：大量小文件的元数据操作仍可能成为瓶颈，需配合缓存策略
4. **生态集成**：相比成熟的 NFS/Ceph，与第三方工具的集成度较低
5. **社区成熟度**：作为较新的开源项目，社区规模和文档丰富度仍有提升空间

## 历史影响与定位

### 在分布式文件系统演进中的位置

```
早期 NFS/CIFS → Lustre/GPFS → HDFS → Ceph/GlusterFS → 3FS / JuiceFS / Alluxio
                                                          (面向 AI 训练)
```

3FS 代表了分布式文件系统演进的**最新方向**——从通用存储向**场景特化**转变：

1. **第一代（1980s-1990s）**：NFS、CIFS 等网络文件系统，解决跨机器文件共享
2. **第二代（2000s）**：Lustre、GPFS 等 HPC 文件系统，面向超算并行 I/O
3. **第三代（2010s）**：HDFS（大数据）、Ceph（通用对象+块+文件），面向云和大数据
4. **第四代（2020s）**：3FS、JuiceFS 等，面向 **AI/ML 训练**和**云原生**场景

### 与前代系统的关系

- **继承自 HPC 文件系统理念**：数据-元数据分离、并行 I/O 等来自 Lustre/GPFS
- **吸收了 Ceph 的教训**：无状态设计避免了 CRUSH 算法和 Ceph Monitor 的运维复杂度
- **超越传统对象存储**：不依赖 S3/MinIO 作为后端，而是自建高性能数据路径
- **面向 AI 原生**：从第一天就为 GPU 训练集群设计，而非事后适配

### 行业影响

- **推动了 AI 基础设施专用化**：证明了通用文件系统在 AI 训练场景下的局限性
- **开源贡献**：将字节跳动内部大规模生产实践开源，丰富了分布式存储生态
- **设计范式影响**：无状态 MetaNode + RDMA + Chunk-based 布局的组合被后续项目借鉴
- **与 Alluxio/Fluid 等缓存层互补**：在 AI 训练栈中，3FS 可作为底层存储与计算缓存层协同

### 未来趋势

- **与计算存储分离架构融合**：在云原生环境下与 Kubernetes、容器存储接口（CSI）深度集成
- **智能分层**：利用 ML 预测 I/O 模式自动进行热/冷数据分层
- **持久内存支持**：利用 Intel Optane / CXL 等新型存储介质进一步降低延迟
- **多模态训练优化**：支持视觉、语言、语音等多模态训练场景的混合 I/O 模式
