## 1. Context & Motivation（背景与动机）

### 解决的问题

腾讯云 CBS 是腾讯云为其 CVM（云服务器）实例提供的 **云原生块存储服务**，核心解决以下问题：

- **虚拟机的持久化存储:** CVM 实例本质上是虚拟化的计算节点，需要可靠、高性能的虚拟磁盘作为系统盘和数据盘
- **存储与计算解耦:** 传统本地磁盘与计算节点绑定，CVM 迁移/伸缩时数据无法随之迁移。CBS 将存储抽离为独立服务，实现计算与存储的生命周期独立
- **弹性供给:** 按需创建、扩容、缩容块设备，无需预先采购物理磁盘
- **数据保护:** 快照、克隆、备份等能力，满足企业级容灾需求

### 前代系统的局限

| 系统 | 局限 |
|------|------|
| **CVM 本地磁盘** | 与计算节点绑定，CVM 迁移/销毁时数据丢失；容量固定，无法在线扩容 |
| **自建 SAN/NAS** | 需要专用硬件和运维团队；在云环境下无法按需弹性扩展 |
| **传统云硬盘 (第一代)** | 基于集中式存储阵列架构，容量和 IOPS 受单阵列控制器限制，扩展能力有限 |
| **副本冗余方案** | 早期云盘多用 3 副本，存储利用率仅 33%，在大规模场景下成本过高 |

### 硬件与网络背景

CBS 的架构演进受益于以下趋势：

- **NVMe SSD 的大规模部署:** SSD 提供了远超 HDD 的 IOPS 和低延迟，使云硬盘能达到数十万级 IOPS
- **RDMA 网络 (RoCE v2):** 腾讯内部数据中心大规模部署 RDMA，存储节点间的复制流量和前端 I/O 路径受益于 RDMA 的低延迟
- **分布式存储中间件成熟:** 腾讯内部自研的分布式存储架构（基于盘古/自研架构理念）支持大规模集群管理
- **虚拟化技术演进:** KVM + 自研虚拟化加速，virtio-blk 半虚拟化驱动减少 I/O 开销
- **纠删码 (EC) 工程实践:** EC 在大规模生产环境中成熟，可在保证耐久性的同时显著提升存储利用率

CBS 的核心思路：**构建在腾讯云自研分布式存储基础设施之上，通过多层缓存 + NVMe SSD + 纠删码，为 CVM 提供高性能、高耐久、弹性可扩展的云原生块存储。**

### 数据模型与命名空间

CBS 的数据模型是一个 **逻辑卷 → 数据分片 → 纠删条带 → 物理磁盘** 的四级映射：

```
Cloud Disk (逻辑卷, e.g. /dev/vda)
    │
    ▼ 按固定大小分片 (Chunk/Extent)
    │
Logical Chunks (逻辑分片)
    │
    ▼ 纠删码编码 (k+m)
    │
EC Stripes (k 个数据块 + m 个校验块)
    │
    ▼ 分布映射
    │
NVMe SSD (物理存储, 跨多个 Storage Node)
```

**关键概念:**

| 概念 | 说明 |
|------|------|
| **云硬盘 (Cloud Disk / Volume)** | 一个块设备，绑定到 CVM 实例使用，可独立创建、扩容、快照 |
| **逻辑分片 (Chunk)** | 卷数据按固定大小切分（通常几十到几百 MB），作为数据迁移和恢复的基本单位 |
| **纠删条带 (EC Stripe)** | 每个 Chunk 编码为 k 个数据块 + m 个校验块，分布到不同 Storage Node |
| **Storage Node** | 部署 NVMe SSD 的物理存储服务器，构成存储池 |
| **CBS Gateway** | 前端 I/O 代理，接收 CVM 的块 I/O 请求，转发到 Storage Node |
| **AZ (Available Zone)** | 可用区，CBS 支持在同一 Region 内跨多个 AZ 部署 |

### 元数据架构归类

**集中式分区 (Partitioned Centralized) — 分布式 K-V 元数据服务**

CBS 作为闭源商用产品，其元数据架构基于腾讯自研的分布式 K-V 存储系统（推测基于 TcaplusDB 或类似架构），核心特征：

- **集中管理但分区部署:** 元数据服务按 Volume/Chunk 分区，每个分区由独立的元数据节点管理
- **元数据与数据路径分离:** 元数据操作（创建卷、快照、扩容）走控制面，数据 I/O 走数据面（Gateway → Storage Node）
- **缓存优化:** CBS Gateway 层缓存热点元数据（卷配置、Chunk 映射），减少元数据查询延迟

### 数据架构分析

#### 分块方式 (Chunking)

- **基于 Extent/Chunk 的固定分块:** 卷数据按固定大小的 Chunk 切分（推测 64MB-256MB 级别）
- **Chunk 内部:** 数据进一步按 4KB 块对齐，以匹配 NVMe SSD 的页大小
- **增量管理:** 快照操作基于 Chunk 级别的 COW，仅记录修改的 Chunk

#### 分布策略 (Distribution)

- **CRUSH 类算法或一致性哈希:** Chunk 映射到 Storage Node 采用分布式哈希策略，确保均匀负载
- **故障域隔离:** 同一 Chunk 的 k+m 个 EC 分片分布到不同的 Storage Node、不同的物理机架
- **多 AZ 部署:** Premium SSD 支持跨 AZ 部署，数据分布在多个可用区的存储池中

#### 冗余方式 (Redundancy)

- **纠删码 (Erasure Coding):** CBS 的核心冗余机制，典型配置如 k=4, m=2（可容忍 2 个分片丢失）或 k=8, m=4
  - 存储利用率 = k/(k+m)，如 4+2 = 66.7%，8+4 = 66.7%，显著优于 3 副本的 33.3%
  - 写放大：k+m 次写入（vs 副本的 3 次），但在 NVMe SSD + RDMA 下可接受
- **多副本模式 (部分场景):** 某些低延迟场景或早期版本可能保留 3 副本选项
- **校验和:** 每个数据块和校验块携带 CRC 校验，防止静默数据损坏
- **跨 AZ 复制 (可选):** 支持将数据同步复制到另一个 AZ，实现 AZ 级容灾

#### I/O 路径 (IO Path)

**完整数据路径 (写操作):**

```
CVM Instance
    │
    ▼
┌─────────────────┐
│  virtio-blk     │  或 NVMe controller (新一代 CVM)

└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│              CBS Client (内核态驱动)          │
│  - 块请求队列管理                             │

         │
         ▼

│    或直接内核态) │
└─────────────────┘
```

**读操作优化:**
- **本地缓存:** CBS Gateway 缓存热点 Chunk 元数据，减少查表延迟
- **预读 (Read-ahead):** 对顺序读模式预取相邻数据块
- **EC 读优化:** 正常读只需 k 个数据分片中的任意一个即可返回，无需等待所有分片

**写操作优化:**
- **EC 并行写:** k+m 个分片通过 RDMA 并行写入，延迟取决于最慢的一个
- **条带写入:** 对齐 EC 条带边界的写操作避免读-改-写 (RMW)
- **写缓冲:** 小写操作可能先聚合到缓冲区，再批量刷盘

#### 生命周期管理

- **动态创建:** 秒级创建空卷（thin provisioned），实际空间按需分配
- **在线扩容:** 支持在线增大卷容量（无需卸载），通过追加新 Chunk 实现
- **缩容:** 一般不支持缩容（防止数据丢失）
- **卷迁移:** 卷可在 CVM 间迁移（先 Detach 再 Attach）
- **垃圾回收:** 删除卷时，标记对应 Chunk 为空闲，后台异步回收

## 3. Core Technical Innovations（核心技术创新）

### 一致性模型

**单节点强一致性 (Per-Volume Strong Consistency)**

- **单 Writer 语义:** 每个 CBS 卷在同一时刻只能被一个 CVM 实例以读写模式挂载（Multi-attach 场景除外，见下文）
- **写后读一致:** 写操作完成（ACK 返回）后，后续读操作必定读到最新数据
- **实现方式:**
  - CBS Gateway 对每个卷维护串行化 I/O 队列
  - Chunk 级别的写操作保证原子性（写 k+m 分片全部成功或全部失败）
  - 读操作从最新版本的分片读取

### 共识协议

CBS 的共识机制体现在多个层次：

| 层次 | 共识机制 | 说明 |
|------|----------|------|
| **元数据服务** | Raft / 类 Raft | 分布式 K-V 元数据存储通过 Raft 保证元数据的强一致性和高可用 |
| **Gateway HA** | 主备切换 | 多个 Gateway 实例组成主备组，通过心跳检测实现自动故障切换 |
| **EC 分片** | 无需共识 | EC 分片的分布通过确定性算法计算，不依赖运行时共识 |
| **跨 AZ 复制** | 同步/异步 | 可选同步复制（强一致但增加延迟）或异步复制（最终一致但延迟更低） |

### 容错机制

| 机制 | 说明 |
|------|------|
| **纠删码容错** | k+m 配置下可容忍 m 个分片同时丢失，数据不丢失 |
| **Chunk 级别恢复** | 单个分片丢失时，通过 k 个存活分片重建丢失分片（在线 EC 重建） |
| **Storage Node 故障** | Node 故障后，其上的分片自动在其他节点重建 |
| **Gateway 高可用** | 多个 Gateway 实例主备/负载分担，单 Gateway 故障不影响服务 |
| **数据校验 (Scrubbing)** | 定期校验数据块完整性，通过 CRC 检测静默数据损坏 |
| **跨 AZ 容灾** | 可选跨 AZ 数据复制，单 AZ 故障时切换到备用 AZ |
| **快照隔离** | 快照创建时锁定当前 Chunk 版本，保证快照的数据一致性 |

### 性能优化技术

#### 多级存储架构

```
┌─────────────────────────────────────────────┐
│  L1: CBS Gateway 元数据缓存 (DRAM)           │
├─────────────────────────────────────────────┤
│  L2: Storage Node 写缓冲区 (DRAM/SSD SLC)    │
├─────────────────────────────────────────────┤
│  L3: NVMe SSD 主存储 (TLC/QLC)               │
└─────────────────────────────────────────────┘
```

#### I/O 路径优化

- **RDMA (RoCE v2):** 存储节点间复制和前端 I/O 路径使用 RDMA，绕过内核协议栈，降低延迟至微秒级
- **SPDK 用户态 I/O:** Storage Node 可能使用 SPDK 框架，实现零拷贝、无中断的 NVMe I/O 路径
- **I/O 调度:** CBS Gateway 对 I/O 进行合并、排序和优先级调度，减少随机 I/O 的放大效应
- **QoS 隔离:** 每个卷可配置 IOPS 和带宽上限，防止 "吵闹邻居" (noisy neighbor) 问题

#### 快照优化

- **增量快照:** 仅记录与上次快照之间的差异 Chunk，大幅节省快照空间
- **即时快照:** 快照创建是元数据操作（标记 Chunk 版本），秒级完成，不影响 I/O 性能
- **快照链管理:** 维护快照之间的父子关系，支持从任意快照恢复或创建克隆

```
快照链示例:

  [Base Volume]
       │
       ├── Snap-2024-01-01 (完整快照)
       │      │
       │      └── Snap-2024-01-15 (增量, 仅记录差异 Chunk)
       │             │
       │             └── Clone-From-Snap (从快照创建的克隆卷)
       │
       └── Snap-2024-02-01 (增量, 仅记录差异 Chunk)
              │
              └── Clone-For-Testing (克隆用于测试环境)
```

### Multi-Attach 支持

CBS 支持 **多挂载 (Multi-Attach)** 模式，允许多个 CVM 实例同时挂载同一个 CBS 卷：

- **读写共享模式:** 多个 CVM 可同时读写同一卷（需要上层文件系统支持集群文件系统如 OCFS2/GFS2）
- **只读共享模式:** 一个 CVM 读写，多个 CVM 只读
- **典型场景:**
  - 数据库共享存储 (如 Oracle RAC 类似架构)
  - 容器共享持久卷
  - 高可用集群 (Active-Active 或 Active-Passive)
- **实现机制:** CBS Gateway 层通过分布式锁管理多客户端的并发访问，确保数据一致性

### 扩展方式

- **横向扩展 (Scale-out):** 增加 Storage Node 即可线性增加容量和 IOPS
- **垂直扩展 (Scale-up):** 单个卷支持在线扩容（增大容量），性能随之提升
- **性能等级选择:** 用户可根据需要选择不同性能等级的 CBS 卷

## 4. Trade-offs (CAP / PACELC)（权衡取舍）

### CAP 定理定位

**CBS 是 CP 系统（单节点强一致性）**

| 维度 | 选择 | 说明 |
|------|------|------|
| **Consistency** | ✅ 强一致 | 每个卷保证写后读一致性，EC 分片写操作原子性 |
| **Availability** | ⚠️ 部分可用 | 单 Storage Node 故障时，EC 冗余保障数据可用；但跨 AZ 同步复制场景下，AZ 故障可能短暂影响可用性 |
| **Partition Tolerance** | ✅ | 分布式架构，容忍网络分区 |

**CAP 结论: CP 系统** — 在网络分区或节点故障时，优先保证数据一致性。这符合块存储的核心需求：文件系统要求块设备提供一致的读写语义，否则会导致文件系统损坏。

### PACELC 定位

| 场景 | 选择 | 说明 |
|------|------|------|
| **Partition (P)** | **C**onsistency | 网络分区时优先保证一致性，可能短暂拒绝服务直到恢复 |
| **Else (E-L)** | **C**onsistency | 正常运行时也优先一致性，不牺牲一致性换取延迟 |

PACELC 结论: **CC** — 无论是否有分区，都选择一致性。

**细化理解:** 在正常场景下，CBS 通过以下手段在一致性和延迟之间取得平衡：
- **EC 并行写:** k+m 分片并行写入，不串行化，减少延迟
- **Gateway 本地缓存:** 减少元数据查询延迟
- **RDMA 低延迟网络:** 传输延迟微秒级，掩盖了 EC 的写放大

### 复杂度 vs 运维简易性

| 优势 | 代价 |
|------|------|
| 全托管服务，用户无需运维 | 闭源架构，用户无法自行调优底层参数 |
| 自动故障检测和恢复 | 问题排查依赖腾讯云工单支持 |
| 自动数据均衡和 EC 重建 | EC 重建期间性能可能下降 |
| 弹性扩容，按需付费 | 跨 AZ 复制增加成本和延迟 |

### 成本 vs 性能

#### 性能等级 (Performance Tiers)

| 性能等级 | 介质 | IOPS | 吞吐量 | 延迟 | 适用场景 |
|----------|------|------|--------|------|----------|
| **Premium SSD (增强型 SSD)** | NVMe SSD + 低延迟网络 | 最高数十万 | 最高数 GB/s | 亚毫秒级 | 数据库、高性能 OLTP |
| **High-Performance SSD (高性能 SSD)** | SSD + 标准网络 | 数万~十万 | 数百 MB/s | 毫秒级 | 通用计算、Web 应用 |
| **Standard SSD (普通 SSD)** | SSD | 数千~数万 | 数百 MB/s | 数毫秒 | 开发测试、轻量应用 |

#### 存储利用率与成本

| 冗余方案 | 存储利用率 | 典型配置 | 适用场景 |
|----------|-----------|----------|----------|
| **3 副本** | 33.3% | 每份数据写 3 份 | 早期版本/低延迟场景 |
| **EC k=4, m=2** | 66.7% | 容忍 2 分片丢失 | 主流配置 |
| **EC k=8, m=4** | 66.7% | 容忍 4 分片丢失 | 高耐久要求 |

EC 相比 3 副本：**存储成本降低 50%，但写入放大约 1.5x-2x**。在 NVMe + RDMA 下，写入延迟增加可接受。

#### 快照成本

- **增量快照:** 仅存储差异数据，空间占用小
- **快照保留策略:** 用户可配置快照保留周期，避免快照空间无限增长

### Multi-Attach 的一致性权衡

Multi-attach 场景下，CBS 提供共享块设备，但**不解决文件系统级别的并发控制**：

- **CBS 层:** 保证块级别的写一致性（每个扇区的写是原子的）
- **文件系统层:** 需要集群文件系统 (OCFS2, GFS2) 或上层分布式文件系统协调
- **风险:** 普通 ext4/xfs 在 Multi-attach 下可能导致文件系统损坏

## 5. Influence & Legacy（影响与遗产）

### 在腾讯云生态中的角色

CBS 是腾讯云 IaaS 层的核心存储服务，与以下产品深度集成：

| 产品 | 集成关系 |
|------|----------|
| **CVM (云服务器)** | CBS 是 CVM 的系统盘和数据盘后端 |
| **TDSQL-C (云原生数据库)** | CBS 为 TDSQL-C 提供存储后端（存算分离架构） |
| **CBS 快照服务** | 支持手动/自动快照，快照可跨地域复制 |
| **CBS 备份服务** | 定期备份到 COS (对象存储)，实现长期归档 |
| **弹性伸缩 (AS)** | CBS 卷可随 CVM 自动创建和挂载 |
| **容器服务 (TKE)** | CBS 作为 Kubernetes PV/PVC 的后端存储 (CSI 驱动) |

### 成为标准的设计模式

腾讯云 CBS 代表了**云厂商块存储的通用架构模式**，与 AWS EBS、阿里云 ESSD 等共同定义了以下行业标准：

1. **存储与计算解耦:** 块存储作为独立于计算实例的托管服务
2. **性能等级分层:** 用户按 IOPS/吞吐量需求选择不同性能等级
3. **快照 + 克隆 + 备份:** 三位一体的数据保护方案
4. **纠删码替代副本:** 在保证耐久性的同时降低存储成本
5. **多 AZ 部署:** 跨可用区容灾，满足企业级 SLA
6. **按需弹性:** thin provisioning + 在线扩容

### 与同类产品的对比

### 当前相关性和使用场景

**腾讯云 CBS 在今天依然是:**

1. **腾讯云 CVM 的核心存储后端** — 几乎所有 CVM 实例依赖 CBS 提供系统盘和数据盘
2. **云原生数据库的存储层** — TDSQL-C 等存算分离架构依赖 CBS 提供低延迟持久化
3. **容器持久化存储** — 通过 CSI 驱动为 TKE 提供 PV/PVC
4. **企业级容灾方案** — 快照 + 跨 AZ 复制满足金融/医疗等行业合规要求

**典型用户场景:**
- **电商:** Premium SSD 支撑大促期间的高并发数据库 I/O
- **游戏:** 高性能 SSD 提供低延迟的玩家数据存储
- **金融:** 跨 AZ 复制 + 快照满足监管合规
- **AI/ML:** 大容量 CBS 卷存储训练数据集

### 局限与竞争

- **闭源架构:** 用户无法审计或调优底层实现，依赖腾讯云技术支持
- **网络延迟天花板:** 虽然使用 RDMA，但跨 AZ 场景的 RTT 仍在 0.1-1ms 级别，对极致低延迟场景仍有局限
- **EC 重建性能:** 大规模 Storage Node 故障时，EC 重建消耗大量网络和 I/O 资源
- **与对象存储的竞争:** 某些场景下（如数据湖、备份），COS (对象存储) 是更经济的选择
- **Multi-attach 的局限性:** 共享块设备需要上层文件系统配合，限制了多写场景的普适性
