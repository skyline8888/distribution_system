## 1. Context & Motivation（背景与动机）

### 解决的问题

字节跳动在 2018-2020 年经历了爆炸性增长——TikTok 全球用户从数亿飙升至超 10 亿。支撑这一增长的核心挑战是：

1. **海量 VM/容器存储需求**：内部云平台需要为数十万台虚拟机和容器实例提供高性能块存储，传统 SAN 阵列在成本和扩展性上完全无法满足。
2. **AI/推荐训练的低延迟 IO**：模型训练需要高 IOPS、低延迟的数据加载，通用 NFS/对象存储在随机读写场景下表现不足。
3. **多业务线统一存储底座**：不同业务（抖音、TikTok、西瓜视频、头条、飞书）各自为战建设存储，导致运维复杂度高、资源利用率低。
4. **NVMe 硬件红利释放**：NVMe SSD 大规模采购部署后，需要新的软件栈来充分发挥其 IOPS 潜力，传统内核 I/O 栈成为瓶颈。

### 前代系统的局限性

| 前代方案 | 局限 |
|----------|------|
| 传统 SAN 阵列 | 成本极高、扩展受限、单阵列 PB 级即触顶 |
| Ceph RBD | 内核 I/O 路径延迟高、CRUSH 计算在大规模 NVMe 下效率不足、社区版对多 AZ 部署支持有限 |
| 直接挂载本地 NVMe | 无快照/克隆能力、数据不可恢复、VM 迁移困难 |
| 基于 NFS 的 VM 磁盘 | 随机读写 IOPS 低、延迟高 |

### 硬件/网络背景

- **NVMe SSD 成熟**：PCIe 4.0/5.0 NVMe 单盘可达 1M+ IOPS、7+ GB/s 吞吐
- **25G/100G RDMA 网络**：RoCE v2 在数据中心内部署，为 NVMe-oF 提供网络基础
- **SPDK 生态成熟**：用户态 I/O 框架绕过内核瓶颈，实现亚微秒级 I/O 延迟
- **多 AZ 基础设施**：字节跳动自建数据中心支持跨可用区部署

### 元数据架构：原生分布式元数据服务

| 维度 | 设计 |
|------|------|
| **模式** | 原生分布式元数据服务 + 外部强一致 KV 存储 |
| **存储引擎** | 基于 Raft 的强一致 KV（类似 TiKV/FoundationDB 模式） |
| **元数据内容** | Volume 拓扑、放置策略、快照树、权限、QoS 配置 |
| **数据面元数据** | 每个 BS-Node 维护本地 extent 映射表（数据块 → 物理位置） |
| **缓存策略** | 客户端缓存元数据热路径，减少控制面交互 |

> **推断标注：** 具体元数据存储引擎为推断。基于字节跳动技术栈（广泛使用 Raft/KV 模式）和业界通用做法推断。

### 数据架构

#### 分块方式 (Data Chunking)

- **固定大小 Chunk/Extent**：每个 Volume 被切分为固定大小的逻辑块单元（推测为 64MB–1GB 级 extent，类似 Ceph 的 RADOS object 或盘古的 chunk）
- **逻辑块映射**：Volume → Extent Group → Chunk → 物理存储位置
- **动态分裂**：热点 chunk 可分裂以分散 I/O 压力

#### 分布策略 (Data Distribution)

- **一致性哈希 + 拓扑感知放置**：数据在 BS-Node 间分布时考虑机架/机房拓扑
- **跨 AZ 放置**：支持多可用区数据分布策略（如 3AZ 各一份、或纠删码跨 AZ 分布）
- **负载均衡**：Placement Engine 监控各节点负载，动态迁移热点数据

#### 冗余方式 (Data Redundancy)

| 模式 | 适用场景 | 说明 |
|------|----------|------|
| **三副本复制** | 高 IOPS 热数据 | 同步写入 3 个 BS-Node，强一致性 |
| **纠删码 (EC)** | 温/冷数据、大容量场景 | 类似 RS(4+2) / RS(8+2) 编码，降低存储开销 |
| **EC + 局部重建** | 单盘/单节点故障 | 支持快速局部重建，无需全条带重建 |

#### I/O 路径 (IO Paths)

```
Client (VM)                    Data Path
    │
    ▼
┌───────────────────────┐
│  BS-Client (用户态)    │  ← SPDK / DPDK 绕过内核网络栈
│  ├─ NVMe-oF Initiator  │
│  ├─ I/O Scheduler      │
│  └─ Connection Pool    │
└───────────┬───────────┘
            │  RDMA/RoCE v2 或 TCP
            ▼
┌───────────────────────┐
│  BS-Node Target       │
│  ├─ SPDK NVMe Target  │  ← 用户态 NVMe Target，零拷贝
│  ├─ I/O Dispatcher    │
│  ├─ EC Encoder        │  ← 纠删码编码/解码
│  └─ Replication Engine│  ← 副本同步
└───────────┬───────────┘
            │  PCIe 直连
            ▼
    ┌───────────────┐
    │  NVMe SSD     │
    │  (用户态直连)  │
    └───────────────┘
```

**关键 I/O 路径优化：**
- **全用户态 SPDK 栈**：BS-Client 和 BS-Node 均运行在用户态，避免内核上下文切换
- **零拷贝**：I/O 数据路径零拷贝，减少内存带宽消耗
- **轮询模式 (Polling)**：SPDK 轮询模式替代中断驱动，降低延迟
- **多队列并行**：利用 NVMe 多队列特性，充分利用 CPU 核心

#### 生命周期管理 (Data Lifecycle)

| 阶段 | 操作 |
|------|------|
| **创建** | 在线创建，秒级就绪，预分配或瘦分配 |
| **在线扩容** | 容量热扩容，无需停机 |
| **快照** | 写时复制 (CoW) 或重定向写 (Row) 快照，支持快照链 |
| **克隆** | 快照可克隆为独立可写 Volume，秒级克隆 |
| **迁移** | 跨 AZ/跨节点数据迁移（存储级 VM 热迁移） |
| **删除/回收** | 异步安全擦除，空间回收 |
| **QoS 调整** | 在线调整 IOPS/吞吐限制 |

### 快照/克隆/备份集成

```
┌─────────────────────────────────────────────────────┐
│                   快照/克隆/备份体系                   │
│                                                     │
│  Volume ──(写时复制)──► Snapshot ──(克隆)──► Clone  │
│    │                           │                     │

│    │                               │                 │
│    │                               ▼                 │

│    │                         或归档存储               │

- **快照**：增量快照，仅记录变更块，快照链支持回溯

- **与火山引擎集成**：快照/克隆作为云控制台一等公民功能暴露

## 3. Core Technical Innovations（核心技术创新）

### 一致性模型

| 维度 | 设计 |
|------|------|
| **单 Volume 级别** | **强一致性** — 对单个 Block Device 的所有 I/O 操作提供线性一致性（Linearizable） |
| **多 Volume 级别** | 无跨 Volume 一致性保证（标准块存储语义） |
| **写路径** | 同步写入多数派（副本模式）或完成 EC 编码后确认，确保写入持久化 |
| **读路径** | 可从本地副本直接读取（减少延迟） |
| **故障恢复** | 副本自动重建，EC 条带局部重建 |

> **CAP 解读：** 对单个 Volume 的 I/O 操作选择 **CP**（强一致性优先于可用性）。当节点/网络分区时，部分 Volume 可能暂时不可写，但确保不返回陈旧数据。

### 共识与复制

| 机制 | 说明 |
|------|------|
| **副本模式** | 主从复制 + 多数派确认（类似 Raft 的 Write Quorum） |
| **EC 模式** | 纠删码编码后分布式放置，重建时从幸存块恢复 |
| **主节点选举** | 推测基于 Raft 或自研 Paxos 变种（字节跳动内部有成熟的 Raft 实现经验） |
| **故障切换** | 主节点故障时自动切换，客户端通过连接池自动重连 |

### 容错机制

| 故障场景 | 处理方式 |
|----------|----------|
| **单 NVMe SSD 故障** | 副本自动重建 / EC 局部重建，分钟级恢复 |
| **单 BS-Node 宕机** | 元数据服务检测并重新分配数据，客户端自动切换连接 |
| **单 AZ 不可用** | 跨 AZ 副本提供持续服务，EC 模式容忍 AZ 级故障 |
| **网络分区** | 副本模式：分区侧可能暂停写入以保证一致性；EC 模式：依赖多数派数据可用 |
| **静默数据损坏** | 端到端校验和（推测为 CRC32/XXH3），读取时校验并自动修复 |

### 性能优化技术

| 优化 | 效果 |
|------|------|
| **SPDK 用户态 I/O 栈** | 延迟从内核态的 ~50μs 降至 ~5-10μs |
| **NVMe SSD 直连 PCIe** | 消除 RAID/HBA 卡开销，单盘 1M+ IOPS |
| **零拷贝数据路径** | 减少内存带宽消耗 30-50% |
| **I/O 合并与批处理** | 小 I/O 合并为大 I/O，提高 NVMe 利用率 |
| **客户端连接池** | 复用 RDMA/网络连接，降低建连开销 |
| **QoS 隔离** | 多租户场景下 IOPS/吞吐隔离，防止 noisy neighbor |
| **预取/缓存优化** | 客户端侧可选的读缓存层，加速重复读 |

### 扩展方式 (Scale-Out)

| 维度 | 策略 |
|------|------|
| **水平扩展** | 增加 BS-Node 自动加入集群，数据自动再平衡 |
| **容量扩展** | 在线添加 NVMe 磁盘，自动纳入存储池 |
| **性能扩展** | IOPS 和吞吐随节点数线性增长 |
| **跨 AZ 扩展** | 支持 3AZ 部署，单 AZ 独立扩容 |
| **元数据扩展** | 分布式元数据服务可独立扩容 |

## 4. Trade-offs (CAP / PACELC)

### CAP 定位

```
                    分区容忍 (P)
                        ╱
                       ╱
                  ────●────
                 ╱    │    ╲
                ╱     │     ╲
               ╱  强一致 │     高可用
              ╱   (C)    │    (A)
             ╱           │           ╲
            ●───────────●───────────●
          一致性 (C)     │          可用性 (A)
```

**ByteStore 在 CAP 中的位置：CP 系统（单 Volume 级别）**

- **分区时 (P)**：优先选择 **强一致性 (C)** 而非可用性 (A)
- 当网络分区或 AZ 间连接中断时，部分 Volume 可能暂时只读或不可写
- 绝不牺牲数据一致性来换取可用性——这对数据库 VM、AI 训练等场景至关重要

### PACELC 分析

```
PACELC: [P]artition → [C]onsistency or [A]vailability?
        [E]lse (正常) → [L]atency or [C]onsistency?

ByteStore:
  Partition 时 → 选 C（强一致性）
  Else (正常时) → 选 L（低延迟）
  
  结论：CP + EL = 正常时追求低延迟的强一致系统
```

| 场景 | 选择 | 说明 |
|------|------|------|
| **网络分区时** | **C** — 强一致性优先 | 部分 Volume 可能降级，但不返回陈旧数据 |
| **正常运行时** | **L** — 低延迟优先 | 在强一致保证下，通过 SPDK/NVMe 实现亚毫秒延迟 |

### 复杂度 vs 运维简洁性

| 维度 | 取舍 |
|------|------|
| **架构复杂度** | **高** — 用户态 I/O 栈、多 AZ 编排、EC 管理增加系统复杂度 |
| **运维简洁性** | **中等** — 通过统一管控面抽象复杂度，但仍需专业存储运维团队 |
| **故障排查** | **有挑战** — 用户态栈使传统内核诊断工具不适用，需自建可观测性体系 |

### 成本 vs 性能

| 维度 | 权衡 |
|------|------|
| **副本模式** | 存储效率 ~33%（三副本），但 IOPS 高、延迟低 |
| **EC 模式** | 存储效率 ~67-80%（取决于 EC 参数），I/O 路径有编解码开销 |
| **NVMe 硬件** | 初始硬件成本高，但单位 IOPS 成本低于传统 SAN |
| **SPDK 开发** | 研发成本高，但生产性能回报显著 |

### 技术取舍总结

| 选择 | 放弃了什么 | 获得了什么 |
|------|-----------|-----------|
| SPDK 用户态 | 内核兼容性、通用性 | 极致低延迟 |
| CP 一致性 | 分区时部分可用性 | 数据绝不丢失/错乱 |
| 多 AZ 部署 | 延迟增加（跨 AZ RTT） | AZ 级容灾能力 |
| 闭源自研 | 社区生态、人才吸引力 | 深度定制、业务适配 |

## 5. Influence & Legacy（影响与遗产）

### 启发的后续系统

| 系统 | 关系 |
|------|------|
| **火山引擎 CBS** | ByteStore 的对外商业化版本（火山引擎云块存储） |
| **火山引擎 ECS 磁盘** | ByteStore 作为底层存储，支撑火山引擎计算实例 |

### 成为标准的设计模式

ByteStore 体现的设计模式与业界趋势一致：

1. **全用户态存储栈** — 与 SPDK、3FS、DAOS 等系统共享"绕过内核"的设计理念
2. **存算分离** — 存储层与计算层解耦，支持独立扩展
3. **统一存储底座** — 同一套基础设施服务多种业务线（字节跳动内部"统一存储层"策略）
4. **NVMe-first 设计** — 从第一天就针对 NVMe 特性优化，而非在 HDD/SSD 架构上修补

### 技术谱系位置

```

    FC-SAN ──┐
             ├──► iSCSI ──► 集中式存储阵列 ──► AWS EBS (初代)
    iSCSI ──┘

                                        ├──► 云原生块存储通用范式

                                        │
    ┌───────────────────────────────────┤

### 与同类系统的横向对比

| 维度 | ByteStore | Ceph RBD | 阿里云 Pangu(ESSD) | CurveBS |
|------|-----------|----------|---------------------|---------|
| **I/O 路径** | 用户态 SPDK | 内核态 | 用户态+内核混合 | 用户态 |
| **协议** | NVMe-oF/自定义 | RBD Protocol | 自定义 | 自定义 |
| **延迟** | 亚毫秒级 (~100-500μs) | 毫秒级 (~1-5ms) | 亚毫秒级 | 亚毫秒级 |
| **冗余模式** | 副本 + EC | 副本 + EC | 副本 + EC | 副本 |
| **多 AZ** | ✅ | 有限支持 | ✅ | 有限 |
| **快照/克隆** | ✅ 增量+链式 | ✅ CoW | ✅ | ✅ |
| **开源** | ❌ | ✅ Apache 2.0 | ❌ | ✅ Apache 2.0 |
| **典型场景** | 内部云 VM + AI 训练 | 通用云/私有云 | 阿里云 ECS | K8s 低延迟 OLTP |

### 当前相关性和使用场景

ByteStore 作为字节跳动内部关键基础设施：

1. **火山引擎 ECS 底层存储**：所有火山引擎虚拟机实例的系统盘和数据盘
2. **AI/ML 训练集群**：为推荐系统、大模型训练提供高吞吐数据加载
3. **数据库服务**：托管数据库实例（RDS/Redis/Kafka）的持久化层
4. **TikTok/Douyin 业务**：视频渲染、转码、内容处理等重 I/O 工作负载
5. **飞书等办公套件**：企业级持久化存储

### 演进趋势

| 方向 | 说明 |
|------|------|
| **CXL/SCM 集成** | 未来可能引入 CXL 存储扩展器和 SCM（持久内存）进一步降低延迟 |
| **AI 原生优化** | 针对大模型训练的检查点 I/O 模式进行专项优化 |
| **Serverless 块存储** | 按需分配、自动缩容的弹性块存储（类似 AWS EBS gp3 演进路径） |
| **开放接口** | 可能逐步开放更多 NVMe-oF 原生接口，增强第三方兼容性 |
