# Amazon DynamoDB

## 基本信息

| 维度 | 值 |
|------|-----|
| **发布年份** | 2012 |
| **开发方** | Amazon Web Services (AWS) |
| **类型** | 商用 / 闭源 / 托管 NoSQL |
| **核心论文** | 无单一论文 — 设计源自 2007 *Dynamo* 论文 + Bigtable 概念 + 内部演进 |
| **官方文档** | <https://docs.aws.amazon.com/dynamodb/> |

> **重要区分**：DynamoDB ≠ 2007 年 Amazon Dynamo 论文中的系统。DynamoDB 是 AWS 2012 年推出的**托管 NoSQL 数据库服务**，借用了 Dynamo 的部分思想（最终一致性、一致性哈希），但内部架构已大幅演进。

---

## 1. Context & Motivation（背景与动机）

### 解决的问题

2006 年 AWS 推出 S3 和 EC2 后，云原生应用开始涌现。开发者面临一个核心矛盾：

- **RDBMS（如 RDS）功能丰富但弹性受限** — 垂直扩展为主，自动分片困难，高并发下写扩展差
- **自建分布式 KV 运维成本极高** — 需要处理分区管理、故障恢复、监控告警、容量规划

AWS 需要一个**完全托管、无限水平扩展、单数位毫秒延迟**的 NoSQL 服务，让开发者只关注数据模型，不关心运维。

### 前代系统的局限

| 前代 | 局限 |
|------|------|
| **Amazon Dynamo (2007)** | 纯内部系统，非托管服务；无 SQL-like 查询；Gossip + Vector Clock 实现复杂；最终一致性是唯一选项 |
| **Cassandra (2008)** | 开源但需自运维；运维复杂度高；多数据中心配置繁琐 |
| **HBase (2008)** | 依赖 Hadoop 生态；RegionServer 运维重；延迟波动大（GC pause） |
| **SimpleDB (2007)** | AWS 早期 NoSQL 尝试，但 2GB 大小限制、查询能力弱、无法满足大规模场景 |

### 硬件/网络背景

- SSD 在 2010 年代初开始普及，为亚毫秒级 IO 提供了基础
- AWS 全球区域（Region）扩展，多区域复制成为刚需
- 多租户云架构成熟，资源隔离技术完善

## 2. Architecture（架构设计）

### 设计哲学：Dynamo + Bigtable + 多租户隔离

DynamoDB 的内部架构融合了三条技术血脉：

```
                    ┌─────────────────────────────────────────────┐
                    │            DynamoDB 内部融合                 │
                    ├──────────────┬──────────────┬───────────────┤
                    │   Dynamo     │   Bigtable   │ 多租户隔离    │
                    │  (2007)      │  (2006)      │               │
                    ├──────────────┼──────────────┼───────────────┤
                    │ 最终一致性   │ 排序映射表   │ 资源配额/隔离 │
                    │ 一致性哈希   │ SSTable 存储 │ 自动分区/     │
                    │ 去中心化     │ 范围查询     │ 再平衡        │
                    │ 可调一致性   │ Bigtable API │ 多租户安全    │
                    └──────────────┴──────────────┴───────────────┘
```

### 数据模型与命名空间

```
DynamoDB 层级结构:

  Account
    └── Table
          ├── Partition Key (Hash Key) — 必需
          │     └── Sort Key (Range Key) — 可选
          │           └── Item = {PK, SK, Attribute1, Attribute2, ...}
          │
          ├── Primary Index (主索引)
          ├── Global Secondary Index (GSI) — 最多 20 个
          ├── Local Secondary Index (LSI)  — 最多 5 个
          └── TTL, Streams, Backup 等特性
```

**核心概念**：
- **Partition Key (PK)**：决定数据分配到哪个物理分区。相同 PK 的数据位于同一分区。
- **Sort Key (SK)**：在分区内排序，支持范围查询（`BETWEEN`, `>`, `<`, `begins_with`）。
- **Item**：最大 400KB，Schema-free 的键值对集合。

### 元数据架构归类

DynamoDB 属于 **集中式分区元数据** 模式：

- **Partition Manager** 作为集中式控制服务，维护分区到物理节点的映射
- 分区元数据按表/分区粒度分片管理
- 客户端不直接感知分区位置，通过 API 层透明路由

### 数据架构分析

#### 分块方式
- 以 **Partition** 为基本数据单元（非固定大小块）
- 单个分区容量上限约 10GB（早期），后动态调整
- 分区内数据按 Sort Key 有序存储，底层使用 **B-Tree / LSM-Tree 混合**结构

#### 分布策略
- 原始设计：**一致性哈希环**（继承自 2007 Dynamo 论文）
- 现代实现：**自动分区管理** — Partition Manager 根据容量和吞吐自动分裂分区并重新分配
- 分裂阈值：当分区达到 10GB 或吞吐上限时自动分裂

#### 冗余方式
- **同步副本复制**（非 Gossip）：每个分区在多个可用区（AZ）维护副本
- 默认 3 副本跨 AZ 分布
- 写入需多数副本确认后返回（但客户端可配置一致性级别）

#### I/O 路径
- 存储节点使用 **SSD** 作为持久化介质
- **Write-Ahead Log (WAL)** 保证持久性
- 数据页缓存在内存中
- 通过 AWS 内部 RPC 框架进行节点间通信

#### 生命周期管理
- **TTL（Time To Live）**：自动过期删除
- **Point-in-Time Recovery (PITR)**：连续备份，支持 35 天内任意时间点恢复
- **On-Demand Backup**：按需全量备份
- **DynamoDB Streams**：变更数据捕获（CDC）

## 3. Core Technical Innovations（核心技术创新）

### 3.1 自动分区与再平衡

这是 DynamoDB 区别于 2007 Dynamo 的核心进化：

| 特性 | 2007 Dynamo | DynamoDB |
|------|-------------|----------|
| 分区管理 | Gossip 协议，客户端感知 | Partition Manager 集中调度 |
| 分裂触发 | 手动或固定阈值 | 自动（容量 + 吞吐双阈值） |
| 再平衡 | Gossip 传播 | 集中式调度，零停机迁移 |
| 运维 | 需人工介入 | 完全自动化 |

```
分区生命周期:

  [分区 P: 5GB] ─── 写入增长 ───▶ [分区 P: 10GB] ─── 触发自分裂 ───▶
                                                      │
                                        ┌─────────────┴─────────────┐
                                        ▼                           ▼
                                [分区 P1: 5GB]              [分区 P2: 5GB]
                                (PK 范围: 0-M)            (PK 范围: M-end)
                                        │                           │
                                        ▼                           ▼
                              分配给不同物理节点          分配给不同物理节点
```

### 3.2 Adaptive Capacity（自适应容量）

**问题**：在预置容量（Provisioned）模式下，如果某个分区出现热点（hot partition），该分区会触达其分配的吞吐上限，而其他空闲分区无法贡献其闲置容量。

**解决方案**：

- DynamoDB 自动检测热分区
- 在短时间内为该分区**临时分配额外容量**（burst capacity）
- 从其他空闲分区借调容量
- 此过程对应用完全透明

```
Adaptive Capacity 原理:

  预置: 1000 RCU / 1000 WCU (均匀分配到 10 个分区)
  每分区: 100 RCU / 100 WCU

  场景：分区 7 成为热点，需要 250 RCU

  普通行为: 分区 7 在 100 RCU 处节流 → 客户端报错
  Adaptive:  分区 7 获得额外 150 RCU 借用
             从空闲分区（1,3,5,8,10）借调
             持续时间: 短窗口（分钟级）

  最终效果: 热点分区不节流，用户无感知
```

### 3.3 可配置的一致性模型

DynamoDB 在**读操作**级别提供一致性选择：

| 读类型 | 行为 | 延迟 | 适用场景 |
|--------|------|------|----------|
| **强一致读** (Strongly Consistent) | 读取最新写入（等待所有副本同步） | 略高 | 金融交易、库存扣减 |
| **最终一致读** (Eventually Consistent) | 可能读到旧数据（秒级延迟） | 更低 | 社交 feed、商品浏览 |

```
一致性模型对比:

  写入: Client ──W──▶ [主副本 AZ-1] ──同步复制──▶ [副本 AZ-2]
                           │                         │
                           │                         │
  强一致读: Client ────────▶ [主副本 AZ-1]  (保证最新)
  最终一致读: Client ──────▶ [任意副本]   (可能陈旧)

  写入本身总是强一致的（同步复制到多数副本）
  差异仅在于读操作的行为
```

**写操作一致性**：DynamoDB 的写入**始终是强一致的** — 写入操作会同步复制到同一区域的多个副本，多数确认后返回成功。

### 3.4 DynamoDB Streams（变更数据捕获）

- 按时间顺序记录表中的项变更（INSERT / MODIFY / REMOVE）
- 保留 24 小时
- 支持 Lambda 触发器、Kinesis 集成
- 底层实现类似数据库的 Binlog / WAL 回放

```
DynamoDB Streams 数据流:

  ┌──────────┐    ┌───────────┐    ┌─────────────┐    ┌──────────┐
  │ 应用写入  │───▶│ DynamoDB  │───▶│   Stream    │───▶│ Lambda   │
  │          │    │  Table    │    │  (24h 保留) │    │  函数    │
  └──────────┘    └───────────┘    └──────┬──────┘    └──────────┘
                                          │
                                          ▼
                                   ┌─────────────┐
                                   │ Kinesis /   │
                                   │ 自定义消费者 │
                                   └─────────────┘

  Stream Record 内容:
  {
    "eventID": "...",
    "eventName": "INSERT|MODIFY|REMOVE",
    "Keys": { ... },
    "NewImage": { ... },
    "OldImage": { ... }
  }
```

### 3.5 Global Tables（全局表 — 多区域复制）

- 跨多个 AWS 区域自动复制数据
- **主动-主动（Active-Active）** 架构：每个区域都可读写
- 基于 DynamoDB Streams 的异步复制
- **最终一致性**跨区域（通常 < 1 秒）
- 冲突解决：**最后写入者胜出**（Last Writer Wins, 基于时间戳）

```
Global Tables 架构:

  ┌───────────────┐      ┌───────────────┐      ┌───────────────┐
  │  us-east-1    │      │  eu-west-1    │      │  ap-northeast │
  │  ┌─────────┐  │      │  ┌─────────┐  │      │  ┌─────────┐  │
  │  │ Table A │──┼─────▶│  │ Table A │  │      │  │ Table A │  │
  │  │ (读写)  │  │      │  │ (读写)  │  │      │  │ (读写)  │  │
  │  └────┬────┘  │      │  └────┬────┘  │      │  └────┬────┘  │
  │       │       │      │       │       │      │       │       │
  │  ┌────┴────┐  │      │  ┌────┴────┐  │      │  ┌────┴────┐  │
  │  │ Stream  │──┼─────▶│  │ Stream  │──┼─────▶│  │ Stream  │  │
  │  │ (CDC)   │  │      │  │ (CDC)   │  │      │  │ (CDC)   │  │
  │  └─────────┘  │      │  └─────────┘  │      │  └─────────┘  │
  └───────────────┘      └───────────────┘      └───────────────┘

         ◀───────────── 双向异步复制 ────────────▶
         延迟: 通常 < 1 秒
         一致性: 最终一致（跨区域）
         冲突: Last Writer Wins
```

### 3.6 DAX（DynamoDB Accelerator）

- 完全托管的**进程内内存缓存**
- 与 DynamoDB API 兼容（代码无需修改）
- 读取延迟：从毫秒级降至**微秒级**
- 写穿透（Write-Through）缓存
- 集群化部署，自动失效和故障转移

```
DAX 缓存架构:

  Client ───▶ DAX Cluster (3-10 节点)
                    │
              ┌─────┴─────┐
              │  缓存命中  │ ──▶ 返回 (< 1ms)
              │  缓存未命中│ ──▶ 回源 DynamoDB ──▶ 写入缓存 ──▶ 返回
              └───────────┘
                    │
              ┌─────┴─────┐
              │ Write-Through:
              │ 写入同时到 DAX + DynamoDB
              │ 保证缓存与源一致
              └───────────┘

  适用: 读多写少、热点数据访问
  不适用: 需要强一致读的场景（DAX 最终一致）
```

## 4. Trade-offs (CAP / PACELC)

### CAP 定理定位

DynamoDB 的独特之处在于**可配置性**：

| 场景 | 配置 | CAP 选择 |
|------|------|----------|
| 区域内强一致读 | `ConsistentRead = true` | **CP**（分区时选一致性） |
| 区域内最终一致读 | `ConsistentRead = false` | **AP**（分区时选可用性） |
| 跨区域 Global Tables | 异步复制 | **AP**（分区时选可用性） |

```
        分区发生 (P)
             │
    ┌────────┴────────┐
    │                 │
  强一致读          最终一致读
    │                 │
    ▼                 ▼
  [C] 一致性        [A] 可用性
  拒绝读/写         继续服务
  或返回最新        可能返回旧数据
```

### PACELC 定理定位

| 条件 | 选择 | 说明 |
|------|------|------|
| **Partition 时** | **EL** (默认) / **EC** (强一致模式) | 默认可选：最终一致+低延迟 或 强一致+可能不可用 |
| **Else (正常时)** | **L** | 正常时优化延迟 — 最终一致读延迟低于强一致读 |

**结论**：DynamoDB = **EL / EC 可切换 + L** — 将一致性选择权交给开发者。

### 复杂度 vs 运维

| 维度 | DynamoDB | 自建 (Cassandra/HBase) |
|------|----------|----------------------|
| 运维复杂度 | **极低** — 无服务器管理、自动分区 | **高** — 节点管理、监控、扩容 |
| 灵活性 | **受限** — 固定数据模型，查询模式受限 | **高** — 自定义 schema、任意查询 |
| 扩展方式 | 自动/按需 | 手动规划、测试、执行 |
| 调试难度 | 黑盒 — 内部机制不可见 | 白盒 — 可调参、可观测 |

### 成本 vs 性能

| 计费模式 | 适用场景 | 成本特征 |
|----------|----------|----------|
| **预置容量 (Provisioned)** | 可预测负载 | 成本低但需容量规划 |
| **按需容量 (On-Demand)** | 不可预测/突发负载 | 方便但单价高 2-3 倍 |
| **预置 + Auto Scaling** | 有周期性波动 | 平衡成本与弹性 |

**热分区成本陷阱**：在预置模式下，如果数据分布不均（单一 PK 值占比过高），会出现"有容量但用不上"的情况 — 整体吞吐未满但单分区节流。自适应容量缓解了这个问题，但并非无限。

## 5. Influence & Legacy（影响与遗产）

### 设计模式成为行业标准

DynamoDB 开创了多个云原生数据库范式：

| 设计模式 | DynamoDB 的贡献 | 后续系统 |
|----------|----------------|----------|
| **Serverless KV 服务** | 首个大规模成功的托管 NoSQL | Azure Cosmos DB, GCP Firestore |
| **按需计费** | 按请求付费，无容量规划 | MongoDB Atlas, Fauna |
| **内置 Streams** | CDC 作为一等公民 | Cosmos DB Change Feed |
| **多区域主动-主动** | Global Tables | Cosmos DB Multi-Region |
| **托管内存缓存** | DAX | Redis Enterprise Cloud |
| **自适应容量** | 热分区自动缓解 | 各云的 burst capacity 机制 |

### 受 DynamoDB 影响的具体设计

1. **Azure Cosmos DB** — 多模型、全球分布、多一致性级别 — 直接对标 DynamoDB
2. **Google Cloud Firestore** — 从 Datastore 演进，加入更灵活的查询，面向无服务器
3. **阿里云表格存储 (Table Store)** — 类似的数据模型，分区 + 排序键
4. **腾讯云 TcaplusDB** — 游戏场景优化，但架构思路相似
5. **Apache Cassandra** — 虽然与 DynamoDB 同期独立发展，但共享 Dynamo + Bigtable 的思想根基

### 当前相关性与使用场景

| 场景 | 适配度 | 说明 |
|------|--------|------|
| Serverless 应用后端 | ⭐⭐⭐⭐⭐ | Lambda + DynamoDB 是经典组合 |
| 游戏玩家数据 | ⭐⭐⭐⭐⭐ | 低延迟、高并发写 |
| 会话/购物车 | ⭐⭐⭐⭐⭐ | 简单 KV 模型、TTL 自动过期 |
| IoT 遥测数据 | ⭐⭐⭐⭐ | 大量写入、DAX 缓存热点 |
| 复杂关联查询 | ⭐⭐ | 不适合 — 需要关系型或图数据库 |
| 全文搜索 | ⭐ | 不适合 — 应配合 OpenSearch/Elasticsearch |
| 分析/报表 | ⭐⭐ | 应导出到 Redshift/Athena |

### 生态定位总结

DynamoDB 在 AWS 生态中是**无服务器应用的事实标准存储**。与以下服务形成互补：

- **RDS/Aurora** — 关系型工作负载
- **ElastiCache/Redis** — 纯缓存场景
- **OpenSearch** — 搜索和日志分析
- **Redshift/Athena** — 分析和 OLAP
- **S3** — 对象/归档存储

DynamoDB 的价值不仅在于技术先进性，更在于**将分布式数据库的复杂度彻底隐藏在托管服务之下**，让开发者用简单的 KV/文档模型就能获得全球扩展能力。这一范式影响了整个云数据库行业的发展方向。
