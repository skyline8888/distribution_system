# Dynamo (2007)

## 基本信息
- **发布时间**：2007 年 10 月
- **开发方**：Amazon (Werner Vogels, Giuseppe DeCandia, et al.)
- **开源/商用**：商用（Amazon 内部系统，未开源）
- **核心论文**：*Dynamo: Amazon's Highly Available Key-value Store* — SOSP 2007
- **当前状态**：已演进为 DynamoDB（2012 年发布的 AWS 托管服务）；原始 Dynamo 作为内部基础设施仍在运行
- **⚠ 区分**：Dynamo ≠ DynamoDB。Dynamo 是 2007 论文描述的自研 KV 存储；DynamoDB 是 2012 年推出的 AWS 全托管 NoSQL 数据库服务，受 Dynamo 启发但架构完全不同

---

## 5-Dim 分析框架

### 一、数据模型 (Data Model)

**类型**：分布式键值存储（Key-Value Store）

- **Key**：不透明字符串，由客户端或系统生成
- **Value**：不透明二进制 blob，通常 ≤ 1 MB
- **操作**：`get(key)` / `put(key, value)` / `delete(key)`
- **无结构化查询**：无 SQL、无 JOIN、无二级索引（原始 Dynamo 层面）
- **元数据**：每个对象携带 vector clock 和版本号，用于冲突检测
- **设计哲学**：极简化接口，将复杂性推向应用层

```
  ┌──────────────────────────────────────────────────┐
  │  Dynamo Data Model                               │
  │                                                  │
  │  Key ─────────────────────────► Value (opaque)   │
  │  "cart:user:12345"             [binary blob]     │
  │  "session:abc"                 [binary blob]     │
  │  "product:SKU:789"             [binary blob]     │
  │                                                  │
  │  无 schema，无类型约束，无查询语言               │
  └──────────────────────────────────────────────────┘
```

#### 核心设计特征

- **完全对称（Symmetric）**：每个节点既是协调者也是副本存储者，无主从角色
- **增量可扩展**：节点增删仅影响 ~1/N 的数据迁移量
- **去中心化**：无单点故障，无中心化元数据服务
- **NWR 可调一致性**：N（副本数）、R（读 quorum）、W（写 quorum）可 per-request 配置
- **Sloppy Quorum**：首选节点不可用时，自动路由到次优节点（hinted handoff），牺牲一致性保可用性

### 三、核心技术 (Core Technologies)

#### 1. 一致性哈希 + 虚拟节点 (Consistent Hashing with Virtual Nodes)

**问题**：传统哈希取模 `hash(key) % N`，增删节点导致几乎全部数据迁移。

**解决方案**：
- 将哈希空间组织为一个环（0 到 2^128 - 1）
- 每个节点负责环上的一段区间
- 数据 key 的 hash 值顺时针路由到第一个负责的节点

**虚拟节点（vnode）**：
- 每个物理节点在环上拥有多个位置（通常 100-200 个）
- 解决负载不均问题（heterogeneous clusters）
- 节点增删时，负载分散到多个物理节点，而非单个
- 加速数据重新平衡

```
  哈希环示意 (简化，实际为 2^128 空间):

  0° ─────────────────────────────────────────────────── 360°
       │                                                    │
       │   h(cart:1)  h(session:5)     h(product:10)        │
       │      ○          ○                  ○               │
       │       \         │                  /               │
       │        \        │                 /                │
       │     ┌───┴───┐┌──┴──┐┌───────────┴───┐             │
       │     │ A₁    ││ B₂  ││     C₃        │             │
       │     │       ││     ││               │             │
       │     │ A₂    ││ B₁  ││     C₁        │             │
       │     │       ││     ││               │             │
       │     │ A₃    ││ B₃  ││     C₂        │             │
       │     └───────┘└─────┘└───────────────┘             │
       │                                                    │
       │  物理节点 A: A₁, A₂, A₃ (3 个 vnode)               │
       │  物理节点 B: B₁, B₂, B₃                           │
       │  物理节点 C: C₁, C₂, C₃                           │
       └────────────────────────────────────────────────────

  数据路由规则:
    hash(key) 顺时针找到第一个 vnode → 该 vnode 所属物理节点
    复制: 顺时针找下 N-1 个不同物理节点上的 vnode
```

#### 2. 数据复制策略

**偏好列表（Preference List）**：
- 对每个 key，通过一致性哈希确定一个"偏好节点列表"
- 列表长度为 N（通常 N=3），按环上顺时针顺序排列
- 列表上前 N 个**不同物理节点**承担副本职责

**复制位置选择**：
- 不仅考虑哈希环位置，还考虑机架/数据中心分布
- 确保副本跨故障域分布

#### 3. Vector Clock（向量时钟）

**用途**：检测并发写入冲突，而非确定唯一顺序。

```
  Vector Clock 结构:
  [ (Node_A, 2), (Node_B, 1), (Node_C, 3) ]
   每个元组: (节点标识, 该节点累计写入次数)

  因果比较规则:
  - VC₁ ≥ VC₂: 若 VC₁ 在每个维度上 ≥ VC₂ → VC₁ 更新
  - VC₁ ≤ VC₂: 若 VC₁ 在每个维度上 ≤ VC₂ → VC₂ 更新
  - 否则: 并发冲突 (conflict / siblings)

  示例 - 无冲突:
    VC₁: [A:2, B:1]  vs  VC₂: [A:1, B:1]
    → VC₁ ≥ VC₂ (A 的计数更大，B 相同)
    → VC₂ 是旧版本，可安全丢弃

  示例 - 并发冲突:
    VC₁: [A:2, B:0]  vs  VC₂: [A:1, B:1]
    → A 维度: VC₁ > VC₂
    → B 维度: VC₂ > VC₁
    → 互不支配 → 并发冲突 → 返回多个 siblings 给客户端

  客户端解决策略:
    - 最后写入胜出 (LWW)
    - 业务语义合并（如购物车：取并集）
    - 自定义冲突解决函数
```

**注意**：Dynamo 的 vector clock 只记录协调者节点，而非客户端。这意味着同一客户端的不同请求若被不同协调者处理，可能产生不必要的冲突。实践中 vector clock 大小受限（通常截断到最近的 10 个条目），可能导致假阳性冲突。

#### 4. Sloppy Quorum & Hinted Handoff

**严格 Quorum**：写入/读取必须在偏好列表的前 N 个节点上完成。

**Sloppy Quorum**（Dynamo 的创新）：
- 当偏好列表中某个节点不可达时
- 协调者将请求路由到环上**下一个健康节点**（非偏好节点）
- 这个节点临时存储数据，并标记为"hint"（提示：实际归属哪个偏好节点）
- 读操作同理：从可达的任何 N 个节点读取

```
  正常情况:
    key → 偏好列表 [A, B, C]
    写入到 A, B, C → 等待 W 个确认

  节点 B 宕机:
    key → 偏好列表 [A, B, C]
    A 正常 → 直接写入 A
    B 宕机 → 写入到 D（B 顺时针下一个健康节点）← hinted handoff
    C 正常 → 直接写入 C
    D 临时持有该数据，标记 hint: "实际归属 B"

  好处: 写入仍然成功（高可用）
  代价: 数据不在"正确"的位置 → 需要后续修复
```

**NWR 权衡**：
- `W = 1, R = N`：快速写入，强一致读取（写优化）
- `W = N, R = 1`：强一致写入，快速读取（读优化）
- `W + R > N`：保证至少一个节点读到最新值
- `W + R ≤ N`：允许读取到旧值（牺牲一致性，追求可用性）
- Dynamo 典型配置允许 `W + R < N`，这是 AP 取舍的体现

#### 5. Gossip 协议

**用途**：成员管理、故障检测、元数据传播。

- 每个节点定期（每秒）与随机选择的节点交换状态信息
- 传播节点存活状态、负载信息、向量时钟元数据
- 最终一致传播，无全局视图
- 无需中心化注册表

```
  Gossip 协议工作流程:

  节点 A 每 1 秒:
    1. 随机选择节点 B
    2. 交换各自的成员视图（谁活着、谁死了）
    3. 合并视图（取最新信息）
    4. 将新信息传播给其他节点

  信息传播速度: O(log N) 轮次传播到全集群
  集群规模: 通常数百节点
```

#### 6. 反熵（Anti-Entropy）与 Merkle Tree

**问题**：Hinted handoff 和 sloppy quorum 导致数据不一致，需要后台修复。

**Merkle Tree 反熵**：
- 每个节点为数据范围构建 Merkle Tree（哈希树）
- 树根哈希代表整个数据集的完整性
- 节点间交换 Merkle Tree，快速定位差异
- 只同步不一致的部分，减少网络传输

```
  Merkle Tree 反熵同步:

  节点 A                        节点 B
  ┌────────────┐                ┌────────────┐
  │  Root: H₁  │                │  Root: H₂  │
  │   /    \   │                │   /    \   │
  │  H₃    H₄  │  ──比较──►     │  H₅    H₆  │
  │  / \   / \ │                │  / \   / \ │
  │ D₁ D₂ D₃ D₄│                │ D₁ D₂ D₃ D₄│
  └────────────┘                └────────────┘

  比较过程:
  1. 比较根哈希 H₁ vs H₂
  2. 不同 → 递归比较子节点 H₃ vs H₅, H₄ vs H₆
  3. 定位到差异叶子节点
  4. 只同步差异数据

  优势: O(log N) 比较即可定位差异，无需全量扫描
  开销: 构建 Merkle Tree 需要遍历数据（I/O 密集）
```

#### 7. 故障检测与副本同步

**临时故障**：Hinted handoff 处理（目标节点短暂不可达）
**永久故障**：从其他副本恢复数据到新节点
**读时修复（Read Repair）**：读取时发现版本不一致，自动用最新版本覆盖旧版本

### 四、CAP / PACELC 分析

#### CAP 定理定位：**AP 系统**

| CAP 维度 | Dynamo 的选择 | 说明 |
|----------|--------------|------|
| **Consistency** | ✗ 牺牲 | 不保证强一致性，允许脏读/旧读 |
| **Availability** | ✓ 保证 | 任何存活的节点都能处理请求 |
| **Partition Tolerance** | ✓ 保证 | 网络分区时系统继续运行 |

**核心逻辑**：Dynamo 的设计目标是"始终可写"。在亚马逊的电商场景中，购物车和会话数据在黑色星期五等高峰期**绝不能拒绝写入**——哪怕这意味着短暂的不一致。不一致可以在事后修复，但丢失的订单无法挽回。

#### PACELC 分析

| 场景 | Dynamo 行为 |
|------|-------------|
| **Partition (P)** | 选择 **Availability** — 继续接受读写，使用 sloppy quorum |
| **Else (E) — 正常运行时** | 选择 **Latency** — 可调一致性，允许 R+W<N |

```
  PACELC 矩阵:

  ┌──────────────┬──────────────────────────────────┐
  │  网络正常 (E)│  网络分区 (P)                     │
  ├──────────────┼──────────────────────────────────┤
  │  EL: 选延迟  │  PA: 选可用性                     │
  │              │                                   │
  │  Dynamo 允许 │  Dynamo 通过 sloppy quorum        │
  │  R+W < N     │  和 hinted handoff                │
  │  换取低延迟  │  保证写入不拒绝                    │
  │              │                                   │
  └──────────────┴──────────────────────────────────┘

  结论: Dynamo = PA/EL
```

#### 一致性级别总结

| 配置 | 一致性保证 | 延迟 | 可用性 |
|------|-----------|------|--------|
| R+W > N | 单调读一致 | 较高 | 降低 |
| R+W = N | 可能读到旧值 | 中等 | 中等 |
| R+W < N | 最终一致 | 最低 | 最高 |
| R=1, W=1 | 最终一致（最弱） | 最低 | 最高 |

### 五、历史影响与遗产

#### 直接衍生系统

| 系统 | 年份 | 关系 |
|------|------|------|
| **Cassandra** | 2008 | 融合 Dynamo (去中心化架构) + BigTable (数据模型)，Apache 开源 |
| **Riak** | 2010 | Basho 开发，最忠实遵循 Dynamo 论文精神的实现 |
| **Voldemort** | 2009 | LinkedIn 开发，明确基于 Dynamo 设计的 KV 存储 |
| **DynamoDB** | 2012 | AWS 全托管服务，受 Dynamo 启发但架构不同（集中式控制平面） |
| **Couchbase** | 2010 | 部分采用 Dynamo 的一致性哈希和复制策略 |

#### 对分布式系统的深远影响

1. **AP 系统的标杆**：证明了在电商核心业务场景中，最终一致性是可行的
2. **去中心化架构范式**：Gossip + 一致性哈希 + 无主节点成为无中心设计的标准模式
3. **可调一致性**：NWR 模型启发了后续系统的 tunable consistency（如 Cassandra 的 consistency level）
4. **Vector Clock 普及**：将分布式系统中的向量时钟从理论带入工业实践
5. **Hinted Handoff**：成为异步修复机制的标准模式
6. **Merkle Tree 反熵**：成为分布式存储后台数据修复的标准技术
7. **"Always Writable" 哲学**：影响了整个 NoSQL 运动的设计哲学

#### Dynamo vs. DynamoDB 区别

| 维度 | Dynamo (2007, 论文) | DynamoDB (2012, AWS 服务) |
|------|---------------------|---------------------------|
| 性质 | Amazon 内部自研 KV 存储 | AWS 全托管 NoSQL 数据库 |
| 架构 | 完全去中心化，对等节点 | 集中式控制平面 + 数据节点 |
| 共识 | 无共识协议，最终一致 | 内置复制与强一致选项 |
| 开源 | 否（仅论文描述） | 否（商业服务） |
| 数据模型 | 纯 KV | KV + 文档，支持二级索引 |
| Vector Clock | 是 | 否（使用版本号） |
| 影响 | 学术论文，思想影响业界 | 商业产品，直接使用 |

#### 论文的历史地位

- **SOSP 2007** 最具影响力的工业论文之一
- 与 Google 的 BigTable (2006)、MapReduce (2004) 并称 Web 2.0 三大基础设施论文
- 开启了 "NoSQL 运动" 的序幕（"NoSQL" 一词在 2009 年才正式出现，但 Dynamo 是其思想源头）
- 2020 年 ACM 获得 **SOSP 时间检验奖**（Test of Time Award）

## 局限与教训

1. **冲突处理推给应用层**：Vector clock 检测冲突但不解决，客户端必须实现冲突解决逻辑
2. **Vector clock 膨胀**：高并发场景下 clock 可能变得很大，需要截断，导致假阳性冲突
3. **Sloppy quorum 的安全性**：hinted handoff 期间数据不在"正确"位置，如果源节点永久故障，需要手动干预
4. **无认证/授权**：原始 Dynamo 论文未涉及安全问题（内部系统假设可信网络）
5. **仅支持 KV**：无查询能力，无二级索引，应用需要自己维护索引
6. **Merkle Tree 开销**：构建 Merkle Tree 需要扫描全部数据，对 I/O 有压力
7. **Gossip 收敛延迟**：成员信息传播有延迟，可能导致短暂的路由错误
8. **不保证线性一致性**：即使在正常运行时（无分区），也可能读到旧值
