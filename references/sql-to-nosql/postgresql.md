# PostgreSQL

## 5-Dim Analysis Framework

### 一、Context & Motivation — 背景与驱动力

#### 1.1 历史起点

PostgreSQL 的历史可追溯至 **1986 年** 加州大学伯克利分校的 **POSTGRES** 项目（Michael Stonebraker 领导）。它是 Ingres 的学术后继者，目标是探索"后关系型"（post-relational）数据模型。

**名称演变：**
- **1986** — Ingres 项目结束，POSTGRES 开始（"Post-Ingres"）
- **1994** — Andrew Yu 和 Jolly Chen 添加 SQL 解释器 → **Postgres95**
- **1996** — 正式更名为 **PostgreSQL**，强调 SQL 能力与血统

> PostgreSQL 的名字中 "SQL" 是官方命名的一部分（发音 "post-gres-Q-L"），而非社区附加。

**时代背景：**
- 1990s，关系数据库被 Oracle、DB2、Sybase 等商业厂商垄断
- 与 MySQL 的"轻量快速"路线不同，PostgreSQL 选择的是 **功能完整性 + 学术严谨性**
- 目标用户：需要复杂查询、自定义类型、事务完整性的高级应用（GIS、金融、科研）

**核心设计哲学：**
> 功能正确性 > 极致性能 — 宁可慢一点，也要做对。可扩展性是一等公民。

#### 1.2 问题空间

| 问题 | 早期方案 | 痛点 |
|------|---------|------|
| 复杂查询 | MySQL 不支持子查询/CTE/窗口函数 | 分析能力弱 |
| 数据类型 | 固定内置类型 | 无法扩展自定义类型 |
| 并发控制 | 表级锁（MyISAM）| 写操作完全串行化 |
| 全文搜索 | 需要外部引擎 | 架构复杂 |
| GIS/空间数据 | 无原生支持 | 需专用系统 |
| 扩展性 | 封闭架构 | 无法添加新功能 |

### 二、Architecture — 架构设计

#### 2.1 整体架构

PostgreSQL 采用 **进程-per-连接** 模型（而非 MySQL 的线程-per-连接），这是其最显著的架构差异之一。

**进程模型 vs 线程模型对比：**

| 维度 | PostgreSQL (进程) | MySQL/InnoDB (线程) |
|------|-------------------|---------------------|
| 连接开销 | 高 (fork + 独立内存) | 低 (共享线程池) |
| 隔离性 | 强 (进程崩溃不影响其他) | 弱 (线程崩溃可能影响进程) |
| 连接数上限 | 通常数百 (受进程数限制) | 可达数千 |
| 内存使用 | 每连接独立 work_mem | 线程共享 Buffer Pool |
| 扩展方式 | 连接池 (PgBouncer) | 线程池插件 |

#### 2.2 查询处理管道

```
SQL 语句进入 PostgreSQL 后的完整处理流程
══════════════════════════════════════════════════════════

  SQL: SELECT u.name, SUM(o.amount)
       FROM users u JOIN orders o ON u.id = o.user_id
       WHERE u.status = 'active'
       GROUP BY u.name
       ORDER BY SUM(o.amount) DESC
       LIMIT 10;

  第 1 步: Parser (词法 + 语法分析)
  ─────────────────────────────────
  输入: SQL 字符串
  输出: Parse Tree (原始语法树)
  工具: bison + flex (词法分析器生成器)

  第 2 步: Analyzer / Transformer (语义分析)
  ─────────────────────────────────
  输入: Parse Tree
  输出: Query Tree (查询树)
  工作:
    - 名称解析 (表名、列名、函数名)
    - 类型检查与隐式类型转换
    - 权限校验
    - 规则系统 (Rule System) 重写

  第 3 步: Rewriter (规则系统重写)
  ─────────────────────────────────
  输入: Query Tree
  输出: 重写后的 Query Tree(s)
  工作:
    - 视图展开 (VIEW → 子查询)
    - 规则重写 (RULE)
    - 注意: PostgreSQL 的视图实现为"宏展开"，非物化

  第 4 步: Planner / Optimizer (规划器/优化器)
  ─────────────────────────────────
  输入: Query Tree(s)
  输出: Plan Tree (执行计划树)
  工作:
    - 路径枚举 (扫描方式: Seq Scan / Index Scan / Bitmap Scan)
    - 连接顺序优化 (动态规划, GEQO 用于多表)
    - 成本估算 (基于 pg_statistic 统计信息)
    - 并行计划决策 (何时启用并行执行)
    - JIT 编译决策 (何时编译表达式)

    成本模型:
      cpu_tuple_cost        (每行 CPU 成本)
      cpu_index_tuple_cost  (索引扫描每行 CPU 成本)
      random_page_cost      (随机页 I/O 成本, 默认 4.0)
      seq_page_cost         (顺序页 I/O 成本, 默认 1.0)

  第 5 步: Executor (执行器)
  ─────────────────────────────────
  输入: Plan Tree
  输出: 结果元组流
  工作:
    - 计划树遍历 (火山模型 / tuple-at-a-time)
    - 执行节点: Scan → Join → Aggregate → Sort → Limit
    - 锁管理 (行级锁、表级锁、谓词锁)
    - MVCC 可见性判断
    - WAL 记录生成 (写操作)

══════════════════════════════════════════════════════════
```

#### 2.3 MVCC 实现 — 死元组 + VACUUM

PostgreSQL 的 MVCC 与 MySQL/InnoDB 的根本区别在于：**版本存储在数据页内（行内多版本），而非 undo log**。

```
PostgreSQL MVCC — 行内多版本模型
══════════════════════════════════════════════════════════

  数据页内的行结构 (Heap Tuple Header, 23 字节):
  ┌─────────────────────────────────────────┐
  │ t_xmin: 插入此版本的事务 ID              │
  │ t_xmax: 删除/更新此版本的事务 ID (0=活跃) │
  │ t_cid:  命令 ID (同事务内多命令区分)      │
  │ t_ctid: 当前版本的物理位置 (ctid)        │
  │ t_infomask: 状态标志位 (已提交/回滚等)    │
  └─────────────────────────────────────────┘

  版本链示例 (UPDATE 操作):
  ═══════════════════════════════════════════════════════

  数据页 (Block 5):
  ┌────────────────────────────────────────────┐
  │  元组 #1 (死元组 / Dead Tuple)              │
  │  xmin=100, xmax=200, ctid=(5,1) → 新位置   │
  │  data: "Alice", age=25                      │
  │  状态: xmax 事务已提交 → 此版本已死           │
  ├────────────────────────────────────────────┤
  │  元组 #2 (活跃版本 / Live Tuple)            │
  │  xmin=200, xmax=0, ctid=(5,2)              │
  │  data: "Alice", age=30                      │
  │  状态: xmax=0 → 此版本当前活跃               │
  └────────────────────────────────────────────┘

  可见性判断 (对当前事务 T):
    1. t_xmin 对 T 可见? (已提交且 < T 的快照)
    2. t_xmax 对 T 可见? (如果可见则此版本已被删除)
    3. 如果 xmin 可见 且 xmax 不可见 → 此版本对 T 可见

  快照 (Snapshot):
    snapshot {
      xmin: 150,    // < 150 的事务已提交，可见
      xmax: 210,    // ≥ 210 的事务是未来，不可见
      xip: [155, 160, 180, 200],  // 活跃事务列表
    }

══════════════════════════════════════════════════════════

  vs MySQL/InnoDB MVCC:
  ┌──────────────────┬──────────────────────────────────┐
  │ 维度             │ PostgreSQL                      │ MySQL/InnoDB                    │
  ├──────────────────┼──────────────────────────────────┤
  │ 版本存储位置      │ 数据页内 (行内多版本)             │ Undo Log (独立存储)            │
  │ 更新操作          │ 写新行 + 标记旧行 xmax            │ 原地更新 + undo 记录旧值        │
  │ 死元组清理        │ 需要 VACUUM                      │ Purge 线程自动清理              │
  │ 读取放大          │ 扫描时跳过死元组                  │ 遍历 undo 版本链               │
  │ 写放大            │ 每次 UPDATE = INSERT + DELETE     │ 原地修改 + undo               │
  │ HOT 优化          │ 无索引列更新可复用页面槽位         │ 无对应优化                    │
  │ 空间回收          │ VACUUM (标记可用) + VACUUM FULL   │ 页面内自然复用                 │
  └──────────────────┴──────────────────────────────────┘
```

#### 2.4 VACUUM — 空间回收机制

```
VACUUM 工作机制
══════════════════════════════════════════════════════════

  VACUUM (普通):
    1. 扫描数据页，识别死元组 (xmax 已提交)
    2. 将死元组占用的空间标记为"可复用"
    3. 更新页内的空闲空间映射 (FSM - Free Space Map)
    4. 更新可见性映射 (VM - Visibility Map)
       - 标记"所有元组对所有事务可见"的页
       - 支持仅索引扫描 (Index-Only Scan)
    5. 不释放空间给操作系统 (文件大小不变)

  VACUUM FULL:
    1. 重建整个表 (写入新文件)
    2. 仅保留活跃元组
    3. 释放空间给操作系统
    4. 需要独占锁 (阻塞所有访问!)
    5. 替代方案: pg_repack (在线重建, 无需排他锁)

  可视性映射 (Visibility Map):
  ┌─────────────────────────────────────────────────────┐
  │ VM 每页 2 个标志位:                                  │
  │   bit 0: all_visible (所有元组对所有事务可见)         │
  │   bit 1: all_frozen (所有元组已冻结, 防事务 ID 回卷) │
  │                                                      │
  │ all_visible = true 的作用:                            │
  │   - 仅索引扫描可跳过堆表 (Index-Only Scan)           │
  │   - VACUUM 可跳过此页                                │
  │                                                      │
  │ all_frozen = true 的作用:                             │
  │   - 该页元组不再参与事务 ID 可见性比较                │
  │   - 防止事务 ID 回卷导致的"全表可见"错误              │
  └─────────────────────────────────────────────────────┘

  Autovacuum (自动清理):
    - 后台守护进程，根据表的变更阈值自动触发
    - 触发条件 (默认):
      autovacuum_vacuum_threshold = 50
      autovacuum_vacuum_scale_factor = 0.2
      → 当 dead tuples > 50 + 20% × 总行数时触发
    - 动态调整优先级 (基于表活跃度)
    - 同时执行 ANALYZE (更新统计信息)

  事务 ID 回卷保护 (Transaction ID Wraparound):
    - 事务 ID 是 32 位计数器，约 40 亿后回卷
    → 回卷后"旧"事务 ID 会被误认为"新"的
    - VACUUM 将老元组的 xmin 替换为 FrozenTransactionId (2)
    → 冻结后的元组对所有未来事务永远可见
    - autovacuum_freeze_max_age = 200M 触发强制冻结
══════════════════════════════════════════════════════════
```

#### 2.5 存储结构 — Heap Table + TOAST

```
PostgreSQL 存储结构
══════════════════════════════════════════════════════════

  数据目录层次:
  $PGDATA/
    base/
      1/         (template0 数据库)
      4/         (template1 数据库)
      5/         (用户数据库, OID=5)
        16384    (表文件, relfilenode)
        16384.1  (分段文件, 每段 1GB)
        16384_fsm  (空闲空间映射)
        16384_vm   (可见性映射)

  堆表存储 (Heap Table) — 核心设计:
  ┌─────────────────────────────────────────────────────┐
  │  数据页 (Page / Block, 默认 8KB)                     │
  │  ┌───────────────────────────────────────────────┐  │
  │  │ PageHeaderData (24 字节)                       │  │
  │  │   lsn, checksum, flags, lower, upper, special │  │
  │  ├───────────────────────────────────────────────┤  │
  │  │                                               │  │
  │  │  ItemId 数组 (行指针, 从页头向下增长)            │  │
  │  │  ┌─────┐ ┌─────┐ ┌─────┐                     │  │
  │  │  │ LP1 │ │ LP2 │ │ LP3 │                     │  │
  │  │  │offset│ │offset│ │offset│                   │  │
  │  │  └──┬──┘ └──┬──┘ └──┬──┘                     │  │
  │  │     │       │       │                         │  │
  │  │  ┌──┴───┐ ┌─┴────┐ ┌┴──────┐                 │  │
  │  │  │Tuple1│ │Tuple2│ │Tuple3 │                 │  │
  │  │  │      │ │      │ │       │                 │  │
  │  │  └──────┘ └──────┘ └───────┘                 │  │
  │  │           ↑ (从页尾向上增长)                    │  │
  │  │                                               │  │
  │  │  Special Space (索引用, 堆表为空)              │  │
  │  └───────────────────────────────────────────────┘  │
  │                                                      │
  │  特点:                                                │
  │  - 无序存储 (插入位置由空闲空间决定)                   │
  │  - 行指针间接寻址 (VACUUM 可移动元组而不改指针)        │
  │  - 页内空间双向增长 (ItemId ↓, Tuples ↑)             │
  └─────────────────────────────────────────────────────┘

  TOAST (The Oversized-Attribute Storage Technique):
  ┌─────────────────────────────────────────────────────┐
  │  问题: 行数据超过一页大小 (通常 > 8KB - 页头)        │
  │  解决: 将大值拆分存储到单独的 TOAST 表中              │
  │                                                      │
  │  触发条件 (默认):                                     │
  │    - 单个行超过 2KB (TOAST_TUPLE_THRESHOLD)          │
  │    - 目标压缩到 2KB 以下 (TOAST_TUPLE_TARGET)        │
  │                                                      │
  │  策略 (按优先级):                                     │
  │    1. 内联压缩 (PG_TOAST 列不压缩, 其他列压缩)        │
  │    2. 将最大的可 TOAST 列移出到 TOAST 表              │
  │    3. 压缩 TOAST 表中的值                             │
  │    4. 继续移出更多列直到行 ≤ 目标大小                  │
  │                                                      │
  │  TOAST 表结构:                                        │
  │    pg_toast.pg_toast_16384                           │
  │    ┌─────────────────────────────────────────┐      │
  │    │ chunk_id  (UUID, 关联原始行)             │      │
  │    │ chunk_seq (序列号, 排序重组)             │      │
  │    │ chunk_data (实际数据, 2KB/块)            │      │
  │    └─────────────────────────────────────────┘      │
  │                                                      │
  │  适用类型: TEXT, VARCHAR, BYTEA, JSONB, ARRAY,       │
  │           自定义大类型                                │
  │  不适用: 固定长度类型 (INTEGER, TIMESTAMP 等)         │
  └─────────────────────────────────────────────────────┘

  Heap-Only Tuples (HOT) — 关键优化 (8.3+):
  ┌─────────────────────────────────────────────────────┐
  │  场景: UPDATE 只修改非索引列                         │
  │  问题: 正常 UPDATE 会产生新元组 + 更新所有索引        │
  │  优化: HOT 链 — 新元组链接到旧元组，不更新索引        │
  │                                                      │
  │  索引条目 ───► HeapTuple1 ───► HeapTuple2 (HOT链)    │
  │                (旧版本)         (新版本)              │
  │                                                      │
  │  条件:                                                │
  │  - 更新不涉及任何索引列                               │
  │  - 同一页面内有足够的空闲空间                         │
  │  - 新元组通过 t_ctid 链接到前一个版本                  │
  │                                                      │
  │  收益:                                                │
  │  - 避免索引更新 (显著降低写放大)                      │
  │  - 减少 WAL 记录                                      │
  │  - 提升 UPDATE 吞吐量                                 │
  └─────────────────────────────────────────────────────┘
```

#### 2.6 Write-Ahead Log (WAL) — 预写式日志

```
PostgreSQL WAL 架构
══════════════════════════════════════════════════════════

  WAL 基本概念:
    "在数据页修改写入磁盘之前，必须先将修改记录写入 WAL"
    → 保证崩溃恢复时的持久性 (Durability)

  WAL 记录结构:
  ┌─────────────────────────────────────────────────────┐
  │ WAL Record (XLogRecord):                            │
  │  - 头部: xl_tot_len, xl_xid, xl_info, xl_rmid       │
  │  - 数据: 资源管理器特定的修改描述                      │
  │  - 备份块 (Full Page Image, FPI):                   │
  │    首次修改页时记录完整页镜像 (防部分写)               │
  └─────────────────────────────────────────────────────┘

  WAL 写入流程:
  ┌─────────────────────────────────────────────────────┐
  │  1. 事务修改数据页 (在 Shared Buffer 中标记脏页)      │
  │     ↓                                                 │
  │  2. 生成 WAL 记录 → 写入 WAL Buffer                   │
  │     ↓                                                 │
  │  3. WAL Buffer 刷盘 → pg_wal/ 目录下的 WAL 段文件     │
  │     - wal_writer_flush_after = 1MB (触发刷盘阈值)     │
  │     - commit_sync = on (提交时强制 fsync)             │
  │     ↓                                                 │
  │  4. 返回客户端 (commit acknowledged)                  │
  │     ↓                                                 │
  │  5. 后台: BgWriter 将脏页刷盘 (非立即)                │
  └─────────────────────────────────────────────────────┘

  WAL 段文件管理:
  ┌─────────────────────────────────────────────────────┐
  │  pg_wal/                                             │
  │  ├── 000000010000000000000001  (段文件, 默认 16MB)   │
  │  ├── 000000010000000000000002                        │
  │  ├── 000000010000000000000003                        │
  │  └── ...                                             │
  │                                                      │
  │  段文件命名: TLI + 逻辑段号 (各 8 位十六进制)          │
  │  TLI (Timeline ID): 区分恢复后产生的新 WAL 分支        │
  │                                                      │
  │  回收策略:                                             │
  │  - 超过 checkpoint 的旧段文件可被回收                  │
  │  - 可配置归档 (archive_command → cp 到安全位置)       │
  │  - wal_keep_size: 保留最近 N MB 的 WAL                │
  └─────────────────────────────────────────────────────┘

  Checkpoint (检查点):
  ┌─────────────────────────────────────────────────────┐
  │  检查点 = 将 Shared Buffer 中所有脏页强制刷盘的瞬间   │
  │  目的: 崩溃恢复时只需回放检查点之后的 WAL              │
  │                                                      │
  │  触发方式:                                             │
  │  - 定时: checkpoint_timeout (默认 5 分钟)              │
  │  - 容量: max_wal_size (默认 1GB)                      │
  │  - 手动: CHECKPOINT; / pg_start_backup()              │
  │  - 关机: 快速关机或智能关机时自动触发                   │
  │                                                      │
  │  检查点开销:                                           │
  │  - 刷盘峰值 I/O (checkpoint_completion_target 控制)   │
  │  - 默认 0.9: 将刷盘分散到 90% 的检查点间隔内           │
  │  - 过短 → 频繁 I/O 峰值                               │
  │  - 过长 → 恢复时间长 (更多 WAL 要回放)                 │
  └─────────────────────────────────────────────────────┘

  崩溃恢复流程:
  ┌─────────────────────────────────────────────────────┐
  │  1. 读取控制文件 (pg_control) → 定位最后一个检查点    │
  │     ↓                                                 │
  │  2. 从检查点开始重放 WAL (REDO 阶段)                   │
  │     - 按 LSN 顺序重放每个 WAL 记录                     │
  │     - FPI (完整页镜像) 直接恢复页面                     │
  │     - 增量记录重放到页上                               │
  │     ↓                                                 │
  │  3. 遇到 shutdown checkpoint → 恢复完成                │
  │     遇到在线检查点 → 进入恢复模式 (recovery mode)     │
  │     - 继续重放归档 WAL                                 │
  │     - 应用 restore_command                             │
  │     - 可恢复到指定时间点 (PITR)                       │
  └─────────────────────────────────────────────────────┘

  时间点恢复 (PITR — Point-In-Time Recovery):
  ┌─────────────────────────────────────────────────────┐
  │  前提:                                               │
  │  1. 基础备份 (pg_basebackup / 物理备份)               │
  │  2. WAL 归档 (archive_command 持续保存 WAL 段)        │
  │                                                      │
  │  恢复:                                               │
  │  1. 恢复基础备份到数据目录                             │
  │  2. 配置 restore_command (从归档取 WAL)               │
  │  3. 指定 recovery_target_time / recovery_target_xid   │
  │  4. 启动 → 重放 WAL 到目标时间点 → 停止               │
  │  5. 数据库处于指定时间点的状态                         │
  │                                                      │
  │  应用场景:                                             │
  │  - 误删表/数据后回滚                                   │
  │  - 合规审计 (恢复到历史状态)                           │
  │  - 克隆生产库到指定时间点                              │
  └─────────────────────────────────────────────────────┘
```

  │  4. 消费端:                                           │

  │ 冲突处理         │ 不适用 (只读)     │ 需配置策略      │
  │ 用途             │ HA / 读扩展       │ 数据集成/迁移   │

  │                │                                     │
  │                └──► Standby-3                        │
  │                                                      │
  │  Standby-1 既接收 Primary 的 WAL, 又转发给下游        │
  │  减少 Primary 的 walsender 连接数                    │
  └─────────────────────────────────────────────────────┘
```

#### 3.2 索引系统

```
PostgreSQL 索引类型详解
══════════════════════════════════════════════════════════

  B-Tree (默认索引):
  ┌─────────────────────────────────────────────────────┐
  │  适用: =, <, >, <=, >=, BETWEEN, LIKE 'prefix%'     │
  │  实现: Lehman-Yao 并发 B-Tree (高并发友好)           │
  │  特性:                                               │
  │    - 9.6+: 并行 B-Tree 构建                          │
  │    - 13+: 索引行去重 (deduplication)                 │
  │    - 支持 INCLUDE 列 (覆盖索引)                      │
  │    - NULL 值可索引                                   │
  │  PostgreSQL B-Tree vs InnoDB:                        │
  │    - PG: 索引与数据分离 (堆表 + 独立索引)            │
  │    - InnoDB: 聚簇索引 (数据即索引)                   │
  └─────────────────────────────────────────────────────┘

  GIN (Generalized Inverted Index):
  ┌─────────────────────────────────────────────────────┐
  │  适用: 数组、JSONB、全文搜索、hstore                  │
  │  原理: 倒排索引 (值 → 行位置列表)                     │
  │                                                      │
  │  示例:                                                │
  │  CREATE INDEX idx_gin ON articles                    │
  │    USING GIN (to_tsvector('english', body));         │
  │                                                      │
  │  CREATE INDEX idx_jsonb ON docs                      │
  │    USING GIN (metadata jsonb_path_ops);              │
  │                                                      │
  │  特性:                                               │
  │    - 支持 @>, ?, && 等运算符                         │
  │    - fastupdate 缓冲 (延迟合并)                      │
  │    - 构建较慢但查询极快                               │
  │    - 存储开销: 通常 2-3x 原始数据                     │
  └─────────────────────────────────────────────────────┘

  GiST (Generalized Search Tree):
  ┌─────────────────────────────────────────────────────┐
  │  适用: 几何类型、范围类型、全文搜索 (tsearch)         │
  │  原理: 可扩展的搜索树，每个节点存储"摘要" (predicate)  │
  │  后现代 B-Tree: 支持非一维数据的索引                  │
  │                                                      │
  │  内置 OpClass:                                       │
  │    - 几何: box, circle, point, polygon               │
  │    - 范围: int4range, tsrange                        │
  │    - 全文: tsvector (tsearch2)                       │
  │    - 网络: inet (IP 地址范围)                        │
  │    - 自定义: PostGIS 地理空间索引                    │
  │                                                      │
  │  vs GIN:                                             │
  │    GiST → 有损索引 (可能假阳性, 需回表验证)           │
  │    GIN  → 无损索引 (精确匹配, 无需回表验证)           │
  └─────────────────────────────────────────────────────┘

  BRIN (Block Range Index) — 9.6+:
  ┌─────────────────────────────────────────────────────┐
  │  适用: 大数据集上的范围查询 (十亿级行)                 │
  │  原理: 每 N 页 (默认 1 页 = 128 页) 存储 min/max 摘要│
  │                                                      │
  │  CREATE INDEX idx_brin ON big_table                  │
  │    USING BRIN (created_at)                           │
  │    WITH (pages_per_range = 32);                      │
  │                                                      │
  │  优势:                                               │
  │    - 索引极小 (B-Tree 的 1-5%)                       │
  │    - 适合时间序列、日志类数据                         │
  │    - 仅支持范围查询 (<=, >=, <, >)                   │
  └─────────────────────────────────────────────────────┘

  SP-GiST (Space-Partitioned GiST):
  ┌─────────────────────────────────────────────────────┐
  │  适用: 非平衡数据结构 (四叉树、kd 树、基数树)          │
  │  用于: 地理空间、IP 前缀、电话号码等                  │
  │  特点: 内存效率高于 GiST, 适合稀疏数据                │
  └─────────────────────────────────────────────────────┘

  Hash 索引:
  ┌─────────────────────────────────────────────────────┐
  │  仅支持等值查询 (=)                                  │
  │  10+ 支持 WAL (之前崩溃不安全)                        │
  │  通常 B-Tree 更快, 不推荐使用                         │
  └─────────────────────────────────────────────────────┘
```

  │        Parallel B-Tree Index Build                  │
  │                                                      │
  │  11:   Parallel Index Scan (B-Tree)                 │
  │        Parallel Hash Join (内存哈希表共享)            │
  │                                                      │
  │  12:   Parallel CREATE INDEX (部分操作)              │
  │        Incremental Sort                             │
  │                                                      │
  │  13:   Parallel VACUUM (多 Worker 并行清理)          │
  │                                                      │
  │  14+:  更多操作并行化, 优化并行计划成本估算            │
  └─────────────────────────────────────────────────────┘

  控制参数:
    max_worker_processes = 8       (后台 Worker 总数上限)
    max_parallel_workers_per_gather = 2  (每查询 Worker 数)
    max_parallel_workers = 8      (并行 Worker 总数上限)
    parallel_tuple_cost          (并行传输成本, 影响计划)
    parallel_setup_cost          (并行启动成本, 影响计划)
    min_parallel_table_scan_size = 8MB  (最小并行扫描大小)

  成本模型中的并行决策:
    规划器会估算:
      - 并行启动开销 (parallel_setup_cost)
      - 数据传输开销 (parallel_tuple_cost)
      - 加速比 (通常 < 线性, 因为合并开销)
    只有当并行计划总成本 < 串行计划时才选择并行

```
锁系统
══════════════════════════════════════════════════════════

  锁层次 (从轻到重):
  ┌─────────────────────────────────────────────────────┐
  │  行级锁:                                             │
  │    FOR UPDATE      → 排他行锁 (阻塞所有其他锁)        │
  │    FOR NO KEY UPDATE → 排他行锁 (不阻塞 SELECT FOR KEY SHARE) │
  │    FOR SHARE       → 共享行锁 (允许其他 FOR SHARE)    │
  │    FOR KEY SHARE   → 最轻行锁 (仅阻塞 FOR UPDATE)     │
  │                                                      │
  │  表级锁:                                             │
  │    ACCESS SHARE    → SELECT 持有                     │
  │    ROW SHARE       → SELECT FOR UPDATE/SHARE        │
  │    ROW EXCLUSIVE   → INSERT/UPDATE/DELETE           │
  │    SHARE UPDATE EXCLUSIVE → VACUUM/CREATE INDEX CONCURRENTLY │
  │    SHARE           → CREATE INDEX                   │
  │    SHARE ROW EXCLUSIVE →                            │
  │    EXCLUSIVE       →                                │
  │    ACCESS EXCLUSIVE → ALTER TABLE / DROP TABLE      │
  │                                                      │
  │  死锁检测:                                            │
  │  - deadlock_timeout = 1s (等待超过此值开始检测)        │
  │  - 检测到死锁 → 回滚代价最小的事务                     │
  │                                                      │
  │   advisory locks (建议锁):                            │
  │  - 应用层协调锁 (pg_advisory_lock)                   │
  │  - 不自动释放, 需显式管理                             │
  │  - 用于分布式应用中的协调                             │
  └─────────────────────────────────────────────────────┘

  并发控制总结:
    - MVCC: 读写不互斥 (快照隔离)
    - 行级锁: 写写冲突时串行化
    - 谓词锁: SERIALIZABLE 隔离级别使用
    - 无读锁: SELECT 不阻塞任何写操作
    - DDL 锁: ACCESS EXCLUSIVE (最重, 阻塞所有)

### 四、Trade-offs — 权衡分析

#### 4.1 CAP 分析

| 模式 | 一致性 (C) | 可用性 (A) | 分区容忍 (P) | 说明 |
|------|-----------|-----------|-------------|------|
| **单机 PostgreSQL** | ✅ 强一致 (ACID, SERIALIZABLE) | ⚠️ 单机故障即不可用 | N/A | 单机场景，P 不适用 |
| **异步流复制** | ⚠️ 最终一致 (读从库可能读到旧数据) | ✅ 高可用 (主宕从可读) | ✅ | 主从延迟 = 不一致窗口 |
| **同步流复制** | ✅ 强一致 (等待从库确认) | ⚠️ 从库不可用时主库阻塞写 | ✅ | 可用 quorum 模式缓解 |
| **逻辑复制** | ⚠️ 最终一致 (按表异步复制) | ✅ 订阅端可独立读写 | ✅ | 应用层处理冲突 |
| **Citus (分布式扩展)** | ✅ 分片内强一致 | ⚠️ 协调节点单点 | ✅ | 分片间分布式事务 (2PC) |

#### 4.2 PACELC 分析

| 模式 | 正常态 (No Partition) | 分区态 (Partition) |
|------|----------------------|-------------------|
| **单机** | **一致性** (严格 ACID) | N/A |
| **异步流复制** | **可用性** (从库读扩展，允许延迟) | **可用性** (从库可读，可能旧数据) |
| **同步流复制** | **一致性** (等待从库 ACK) | **一致性** (主库阻塞写，等从库恢复) |
| **同步 (QUORUM)** | **一致性 + 延迟** (等 N/2+1 确认) | **一致性** (多数派不可达则阻塞) |
| **逻辑复制** | **可用性** (异步，各节点独立) | **可用性** (各分区独立运行) |

#### 4.3 核心 Trade-offs

##### MVCC 实现差异

| 权衡维度 | PostgreSQL (行内版本) | MySQL/InnoDB (Undo Log) |
|----------|----------------------|------------------------|
| 读取性能 | 扫描时跳过死元组 (可能扫描大量死数据) | 版本链遍历 (undo log 可能很长) |
| 写入性能 | 每次 UPDATE = 新插入 + 旧标记 (写放大) | 原地修改 + undo 记录 (写放大较小) |
| 空间回收 | 需要 VACUUM (额外 I/O + 后台进程) | Purge 线程自动清理 (无感) |
| 膨胀风险 | UPDATE 密集 → 表膨胀显著 | 相对稳定 |
| HOT 优化 | 无索引列更新零索引维护 | 无对应优化 |
| 事务 ID 回卷 | 需要 VACUUM FREEZE 防回卷 | 无此问题 (64 位事务 ID) |

##### 进程模型

| 权衡维度 | PostgreSQL (进程) | MySQL (线程) |
|----------|-------------------|--------------|
| 并发连接 | 数百 (受进程开销限制) | 数千 |
| 崩溃隔离 | 强 (进程崩溃不影响其他) | 弱 |
| 内存开销 | 每连接独立 work_mem | 线程共享 |
| 连接池需求 | 必须 (PgBouncer) | 推荐 (线程池插件) |
| 扩展性 | 进程模型限制了连接数 | 线程模型更利于高并发 |

##### 索引架构

| 权衡维度 | PostgreSQL (堆表+独立索引) | MySQL/InnoDB (聚簇索引) |
|----------|---------------------------|------------------------|
| 主键查询 | 索引查找 + 堆表回表 | 直接访问 (数据即索引) |
| 二级索引 | 索引条目含 ctid → 堆表回表 | 索引条目含主键 → 聚簇索引回表 |
| 多索引扫描 | 支持 Bitmap And/Or (多索引合并) | 不支持 (只能选一个索引) |
| 索引类型丰富度 | 6 种 (B-Tree/GiST/GIN/SP-GiST/BRIN/Hash) | 2 种 (B+Tree/Hash) |
| 覆盖索引 | 支持 (INCLUDE 语法) | 天然支持 (聚簇索引) |

##### 复制模式

| 权衡维度 | 流复制 | 逻辑复制 |
|----------|--------|----------|
| 延迟 | 极低 (毫秒) | 低-中 (秒级) |
| 一致性 | 异步=最终一致, 同步=强一致 | 最终一致 |
| 灵活性 | 整集群复制 | 按表过滤/选择 |
| 跨版本 | 不支持 | 支持 |
| 从库写入 | 不允许 (只读) | 允许 (可读写) |
| 冲突处理 | 不适用 | 需手动配置策略 |

#### 4.4 复杂度 vs 运维

| 运维挑战 | 说明 | 缓解方案 |
|----------|------|----------|
| VACUUM 调优 | 不当配置 → 表膨胀/事务回卷 | Autovacuum 调优 + pg_repack |
| 连接管理 | 进程模型 → 连接数受限 | PgBouncer (连接池) |
| 大事务 | 长事务阻止 VACUUM → 膨胀 | 监控长事务 + 超时设置 |
| WAL 管理 | 归档/流复制 WAL 堆积 → 磁盘满 | 监控 replication lag + wal_keep_size |
| 统计信息 | 过期统计 → 错误执行计划 | 定期 ANALYZE / Autovacuum 包含 |
| 版本升级 | 大版本升级需 pg_upgrade/dump | 逻辑复制做滚动升级 |

### 五、Influence & Legacy — 影响与遗产

#### 5.1 行业影响

**开源数据库"瑞士军刀"：**
- PostgreSQL 是功能最丰富的开源关系数据库，无商业公司控制（PostgreSQL Global Development Group 社区治理）
- BSD 许可证 → 商业友好，许多云厂商基于 PG 构建托管服务
- 兼容 Oracle 特性最多 → 企业从 Oracle 迁移的首选开源方案

**衍生生态系统：**

| 衍生项目 | 年份 | 说明 |
|----------|------|------|
| Greenplum | 2003 | MPP 数据仓库 (EMC/VMware) |
| PostGIS | 2001 | 地理空间扩展 (GIS 事实标准) |
| Citus | 2016 | 分布式 PG (微软收购) |
| TimescaleDB | 2017 | 时序数据扩展 |
| CockroachDB | 2014 | 受 PG SQL 层启发的 NewSQL |
| YugabyteDB | 2017 | 受 PG 启发的分布式 SQL |
| PolarDB (PG兼容) | 2017 | 阿里云存算分离架构 |

| AnalyticDB PG版 | - | 阿里云分析引擎 |
| TDSQL-C PG版 | - | 腾讯云云原生数据库 |

#### 5.2 技术遗产

**1. 可扩展类型系统 → 数据库即平台**

PostgreSQL 证明了"数据库应该是可扩展平台"的理念：
- 自定义类型 + 运算符 + 索引方法 → 数据库可适配任何领域
- **PostGIS** 的成功证明了这一理念：一个扩展让 PG 成为 GIS 领域的标准数据库
- 启发了 **Citus** (分片)、**TimescaleDB** (时序)、**pgvector** (向量搜索)

**2. MVCC + 堆表 → 独特的并发控制范式**

- 与 Oracle (undo segment)、MySQL (undo log) 不同的第三路径
- 行内版本存储 + VACUUM 的设计影响了后续一些系统
- HOT 优化展示了在行内版本模型上的创新空间

**3. 多索引类型 → 查询灵活性标杆**

- GiST、GIN、BRIN、SP-GiST 等索引方法使 PG 能处理从全文搜索到地理空间的广泛场景
- 无需引入外部搜索引擎 (Elasticsearch) 即可处理全文搜索
- 索引访问方法 API 成为学术研究的平台

**4. 逻辑解码 → CDC 生态的基础**

- PostgreSQL 的逻辑解码 (9.4+) 是现代 CDC (Change Data Capture) 的基石之一
- Debezium、Kafka Connect pg 等都依赖 PG 逻辑复制
- wal2json 等插件将 WAL 转换为 JSON，驱动了实时数据管道

**5. 扩展架构 → 数据库生态的模板**

- FDW 概念启发了跨数据源查询的标准化
- 过程语言支持 (PL/pgSQL, PL/Python) 使数据库内计算成为可能
- pgvector 扩展 (2023) 快速成为 AI/向量搜索的事实标准
