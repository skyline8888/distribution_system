# IBM System R (1974-1979)

## 基本信息

| 维度 | 内容 |
|------|------|
| **发布时间** | 1974-1979（研究原型） |
| **开发方** | IBM Research, San Jose Laboratory |
| **核心论文** | *"System R: Relational Approach to Database Management"* — ACM TODS 1(2), 1976; *"The History and Status of the System R Project"* — Selinger, 1979 |
| **开源/商用** | 研究原型，从未作为产品发布 |
| **关键角色** | **历史上第一个 SQL 数据库实现**；证明了 Codd 关系模型的工程可行性；奠定了现代 RDBMS 的三大基石（SQL、基于代价的优化器、ACID 并发控制） |
| **核心团队** | Don Chamberlin, Ray Boyce (SQL 创建者); Pat Selinger (优化器); Jim Gray (事务/并发); Raymond Lorie; Mario Schkolnick |

## 5-Dim 分析框架

### 一、数据模型 (Data Model)

**类型**：关系型（Relational）

- **理论基础**：E.F. Codd 1970 论文 *《A Relational Model of Data for Large Shared Data Banks》*
- **核心数据结构**：
  - **Table（表）**：关系（relation）的具体化，由行（tuple）和列（attribute）组成
  - **Row/Tuple（元组）**：一条记录，具有固定 schema
  - **Column/Attribute（属性）**：类型化的字段（INTEGER, CHAR, FLOAT 等）
- **关键创新 — SEQUEL → SQL**：
  - Chamberlin & Boyce 设计了 **SEQUEL**（Structured English Query Language），后因商标简化为 **SQL**
  - 这是**第一个声明式（declarative）数据库查询语言**
  - 用户描述"想要什么"而非"如何获取"——优化器负责决定执行路径
- **SEQUEL 语法示例**（历史形式，与现代 SQL 近似）：

```
GET NAME, SALARY
FROM EMP
WHERE DEPT = 'TOY' AND SALARY > 20000
```

等价于关系代数：

```
π_name, salary (σ_dept='TOY' ∧ salary>20000 (EMP))
```

**设计哲学**：
- 声明式接口：将"写什么"与"如何执行"彻底解耦
- 关系闭包：查询结果仍是关系，可以嵌套和组合
- 面向集合而非面向记录：一次查询处理一组元组，而非逐条遍历

```
  ┌──────────────────────────────────────────────────────┐
  │              System R Data Model                      │
  │                                                       │
  │  Database                                             │
  │  ┌────────────────────────────────────────────────┐  │
  │  │  Table: EMP (relation)                         │  │
  │  │  ┌──────┬─────────┬──────────┬──────────┐     │  │
  │  │  │ ENO  │ NAME    │ DEPT     │ SALARY   │     │  │
  │  │  ├──────┼─────────┼──────────┼──────────┤     │  │
  │  │  │ E1   │ Alice   │ TOY      │ 25000    │     │  │
  │  │  │ E2   │ Bob     │ SHOE     │ 18000    │     │  │
  │  │  │ E3   │ Carol   │ TOY      │ 30000    │     │  │
  │  │  └──────┴─────────┴──────────┴──────────┘     │  │
  │  │                                                │  │

  │  │       ────────────────────────────────────     │  │
  │  │       结果: {(Alice), (Carol)}                  │  │
  │  └────────────────────────────────────────────────┘  │
  │                                                       │

```

**数据模型要点**：

| 特性 | System R 实现 |
|------|---------------|
| 数据模型 | 关系模型（表/行/列） |
| 查询语言 | SEQUEL/SQL（声明式） |
| 数据类型 | INTEGER, CHAR, FLOAT, DATE |
| 索引 | B-tree（主/二级索引） |
| 视图 | 支持命名视图（虚拟表） |
| 完整性 | 主键约束、外键约束、CHECK 约束 |
| NULL 处理 | 三值逻辑（TRUE/FALSE/UNKNOWN） |

### 三、存储与查询引擎 (Storage & Query Engine)

#### 3.1 物理存储

**页面结构**：
- System R 将数据组织为**固定大小的页（pages）**，通过 OS/360 的文件系统持久化
- 每个页包含多条记录，页内通过**槽目录（slot directory）**管理偏移量
- 记录通过 **RID（Record Identifier）** 定位：`RID = (page_id, slot_number)`

```
  ┌─────────────────────────────────────────┐
  │              Page Layout                 │
  │                                         │
  │  ┌────────────────────────────────────┐ │
  │  │ Page Header                        │ │
  │  │  page_id | lsn | free_space_offset│ │
  │  ├────────────────────────────────────┤ │
  │  │              记录数据区              │ │
  │  │  ┌────────┐ ┌────────┐ ┌────────┐  │ │
  │  │  │Record 1│ │Record 2│ │Record 3│  │ │
  │  │  └────────┘ └────────┘ └────────┘  │ │
  │  ├────────────────────────────────────┤ │
  │  │           Slot Directory            │ │
  │  │  slot[0] → offset of Record 1      │ │
  │  │  slot[1] → offset of Record 2      │ │
  │  │  slot[2] → offset of Record 3      │ │
  │  │         ...                        │ │
  │  ├────────────────────────────────────┤ │
  │  │  Page Trailer / Checksum           │ │
  │  └────────────────────────────────────┘ │
  └─────────────────────────────────────────┘
```

**B-tree 索引**：
- System R 实现了 **B-tree 索引**作为主要的访问路径
- 支持**聚簇索引（clustering index）**和**非聚簇索引（secondary index）**
- 聚簇索引决定数据的物理排列顺序
- 非聚簇索引存储 `<key, RID>` 对，需要额外回表

```
  B-tree Index Structure:

        ┌─────────┐
        │  Root   │
        │ [10|40] │
        └─┬─┬─┬───┘
          │ │ │
    ┌─────┘ │ └─────┐
    ▼       ▼       ▼
  ┌───┐   ┌───┐   ┌───┐
  │ 5 │   │25 │   │55 │
  │ 7 │   │30 │   │50 │
  │ 9 │   │35 │   │60 │
  └─┬─┘   └─┬─┘   └─┬─┘
    │       │       │
    ▼       ▼       ▼
  [RID]   [RID]   [RID]  ← 指向数据页中的记录
```

**系统目录（System Catalog）**：
- System R 引入了**系统目录**的概念——用关系表存储元数据
- 存储表定义、列信息、索引、权限等
- 这是**自举（bootstrapping）**思想的先驱：数据库自身的数据也用数据库来管理
- 现代 RDBMS 的 `INFORMATION_SCHEMA` 和 `pg_catalog` 均源于此

#### 3.2 Selinger 查询优化器 — System R 最核心的技术创新

**问题背景**：声明式 SQL 意味着用户只说"要什么"，不说"怎么做"。同一条 SQL 可能有多种等价的关系代数表达式，每种对应的 I/O 代价差异巨大。System R 必须自动选择最优执行策略。

**Selinger 算法（1979）** — 基于代价的查询优化（Cost-Based Optimization, CBO）：

```
  ┌──────────────────────────────────────────────────────┐
  │            Selinger Cost-Based Optimizer              │
  │                                                       │
  │  输入: SQL 查询                                      │
  │  输出: 最低代价的执行计划                              │
  │                                                       │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ Step 1: 生成所有等价的关系代数表达式              │  │
  │  │         (JOIN 重排序、谓词下推、投影消除)         │  │
  │  └───────────────────┬────────────────────────────┘  │
  │                      ▼                                │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ Step 2: 对每个单表扫描，枚举可用访问路径          │  │
  │  │         • 全表扫描 (Table Scan)                 │  │
  │  │         • 聚簇索引扫描 (Clustering Index Scan)  │  │
  │  │         • 非聚簇索引扫描 (Secondary Index Scan) │  │
  │  └───────────────────┬────────────────────────────┘  │
  │                      ▼                                │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ Step 3: 代价估计                                │  │
  │  │         代价 = I/O 代价 + CPU 代价 × 权重       │  │
  │  │                                               │  │
  │  │   I/O 代价估计依赖:                            │  │
  │  │   • 表基数 (T): 表的总行数                       │  │
  │  │   • 页基数 (P): 表的总页数                       │  │
  │  │   • 选择率 (selectivity): 谓词过滤比例           │  │
  │  │   • 索引层级: B-tree 高度                        │  │
  │  │                                               │  │
  │  │   全表扫描代价:  P 页                           │  │
  │  │   索引扫描代价:  B-tree深度 + 选择行数          │  │
  │  └───────────────────┬────────────────────────────┘  │
  │                      ▼                                │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ Step 4: JOIN 顺序优化                           │  │
  │  │         • 枚举所有表连接顺序                     │  │
  │  │         • 对每种顺序，选择最优访问路径            │  │
  │  │         • 使用动态规划减少搜索空间               │  │
  │  │         • 只保留每个表子集的"帕累托最优"方案      │  │
  │  │           (对不同排序/分组需求的最优计划)         │  │
  │  └───────────────────┬────────────────────────────┘  │
  │                      ▼                                │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ Step 5: 输出最终执行计划                         │  │
  │  │         全局代价最低的方案                       │  │
  │  └────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────┘
```

**核心公式**（单表访问代价估计）：

```
设表 T 有 P 页，M 行
谓词选择率为 s (0 < s ≤ 1)

全表扫描代价:
  C_scan = P

索引扫描代价 (非聚簇):
  C_index = B + s × M    ← B = B-tree 深度, 需要 s×M 次随机 I/O 回表

索引扫描代价 (聚簇):
  C_cluster = B + ceil(s × P)  ← 聚簇使得命中的行在物理上相邻
```

**为什么这是革命性的**：
1. **首次实现基于代价的优化**：之前的数据库使用启发式规则（如"总是先做选择"），Selinger 引入了**量化代价模型**
2. **统计信息驱动**：优化器需要表的行数、列值分布、索引选择性等统计信息
3. **动态规划**：对 n 个表的 JOIN，暴力枚举 n! 种顺序不可行；Selinger 使用动态规划在 O(n × 2^n) 时间内找到最优 JOIN 顺序
4. **现代 CBO 的蓝本**：PostgreSQL、MySQL、Oracle、DB2 的优化器都是 Selinger 算法的直接继承者

```
  Selinger 优化器 vs 启发式优化器对比:

  查询: SELECT * FROM R, S, T
         WHERE R.a = S.b AND S.c = T.d AND R.e = 10

  启发式 (Heuristic):
    1. 先做所有选择谓词 (R.e = 10)
    2. 再做 JOIN
    3. 固定的执行策略，不考虑数据分布

  Selinger (Cost-Based):
    1. 收集统计: |R|=10000, |S|=500, |T|=1000
    2. 计算: R.e=10 的选择率 s=0.01 → R 过滤后剩 ~100 行
    3. 枚举 JOIN 顺序: (R⨝S)⨝T, R⨝(S⨝T), (R⨝T)⨝S, ...
    4. 对每种顺序计算 I/O 代价
    5. 选择代价最低的: 先 R⨝S (100×500), 再 ⨝T
```

### 四、事务与并发控制 (Transaction & Concurrency)

#### 4.1 两阶段锁（Two-Phase Locking, 2PL）

System R 实现了**严格两阶段锁（Strict 2PL）**作为并发控制机制：

```
  ┌──────────────────────────────────────────────────┐
  │            Strict 2PL Protocol                    │
  │                                                   │
  │  阶段一: Growing Phase (增长阶段)                 │
  │  ┌────────────────────────────────────────────┐  │
  │  │ 事务持续获取锁                              │  │
  │  │  • 读操作 → 请求 S (Shared) 锁             │  │
  │  │  • 写操作 → 请求 X (Exclusive) 锁          │  │
  │  │  • 锁可升级: S → X (但反之不行)            │  │
  │  └────────────────────────────────────────────┘  │
  │                    │                              │
  │                    ▼                              │
  │  阶段二: Shrinking Phase (收缩阶段)               │
  │  ┌────────────────────────────────────────────┐  │
  │  │ 事务提交或回滚后释放所有锁                    │  │
  │  │  • Strict 变体: 所有 X 锁直到 COMMIT/ABORT  │  │
  │  │    才释放，防止脏读和级联回滚                │  │
  │  └────────────────────────────────────────────┘  │
  │                                                   │
  │  锁兼容性矩阵:                                     │
  │  ┌─────────┬───────┬───────┐                     │
  │  │         │  S    │   X   │                     │
  │  ├─────────┼───────┼───────┤                     │
  │  │    S    │  ✓    │   ✗   │                     │
  │  │    X    │  ✗    │   ✗   │                     │
  │  └─────────┴───────┴───────┘                     │
  │                                                   │
  │  ✓ = 兼容  ✗ = 冲突                                │
  └──────────────────────────────────────────────────┘
```

**锁粒度层次**：

```
  锁粒度层次结构 (Lock Hierarchy):

  ┌─────────────────┐
  │   Database      │  ← 最粗粒度
  │   (DB lock)     │
  └───────┬─────────┘
          │
  ┌───────▼─────────┐
  │   Table         │  ← 表级锁
  │   (Relation)    │
  └───────┬─────────┘
          │
  ┌───────▼─────────┐
  │   Page          │  ← 页级锁
  │   (Buffer Page) │
  └───────┬─────────┘
          │
  ┌───────▼─────────┐
  │   Record/Row    │  ← 最细粒度
  │   (Tuple)       │
  └─────────────────┘
```

**死锁处理**：
- System R 使用 **wait-for graph（等待图）** 检测死锁
- 当检测到环路时，选择一个事务作为牺牲品（victim）并回滚
- 选择策略通常基于事务已持有的锁数量和已完成的工作量

**2PL 定理保证**：
- 严格 2PL 产生的调度是**冲突可序列化（conflict serializable）**的
- 这保证了事务的**隔离性（Isolation）**

#### 4.2 恢复管理 — WAL 日志（Write-Ahead Logging）

System R 的恢复机制奠定了现代数据库日志系统的基石：

```
  ┌──────────────────────────────────────────────────────┐
  │            Write-Ahead Logging (WAL)                  │
  │                                                       │
  │  核心规则: 日志必须先于数据写入磁盘                     │
  │  "Log before data"                                    │
  │                                                       │
  │  ┌────────────────────────────────────────────────┐  │
  │  │ 事务操作流程:                                   │  │
  │  │                                                │  │
  │  │  1. 修改内存中的数据页 (Buffer Pool)            │  │
  │  │  2. 生成日志记录 (Log Record)                   │  │
  │  │     ┌──────────────────────────────────────┐   │  │
  │  │     │ Log Record:                          │   │  │
  │  │     │  • LSN (Log Sequence Number)         │   │  │
  │  │     │  • Transaction ID                     │   │  │
  │  │     │  • Page ID                           │   │  │
  │  │     │  • Before-image (UNDO)               │   │  │
  │  │     │  • After-image (REDO)                │   │  │
  │  │     │  • Type (INSERT/UPDATE/DELETE)       │   │  │
  │  │     └──────────────────────────────────────┘   │  │
  │  │  3. 将日志记录写入日志文件 (必须 fsync)         │  │
  │  │  4. 标记数据页为脏 (Dirty)                     │  │
  │  │  5. 后续异步刷脏页到磁盘                        │  │
  │  │                                                │  │
  │  │  ⚠ 第 3 步必须在第 5 步之前完成！               │  │
  │  └────────────────────────────────────────────────┘  │
  │                                                       │
  │  崩溃恢复 (Crash Recovery):                           │
  │  ┌────────────────────────────────────────────────┐  │
  │  │  1. Analysis:  扫描日志，确定崩溃时活跃事务       │  │
  │  │  2. Redo:     重放所有日志，恢复已提交的修改      │  │
  │  │  3. Undo:     回滚未提交事务的修改               │  │
  │  └────────────────────────────────────────────────┘  │
  │                                                       │
  │  → 这一框架后来被 Jim Gray 等完善为 ARIES 算法        │
  │    (1989-1992)，成为 DB2、SQL Server、PostgreSQL     │
  │    等系统的标准恢复协议                               │
  └──────────────────────────────────────────────────────┘
```

**ARIES 的历史渊源**：
- System R 的恢复协议是 ARIES 的雏形
- ARIES（Algorithm for Recovery and Isolation Exploiting Semantics）由 IBM 在 1989-1992 年间发表
- ARIES 继承了 System R 的核心思想：WAL、UNDO/REDO 日志
- 并在此基础上增加了：Steal/No-Force 缓冲策略、LSN 追踪、补偿日志（CLRs）、模糊检查点等
- 今天绝大多数关系数据库（DB2、SQL Server、PostgreSQL、MySQL InnoDB）的恢复系统都直接或间接源于 ARIES

#### 4.3 ACID 属性实现

| 属性 | 实现机制 |
|------|----------|
| **Atomicity（原子性）** | WAL 日志 + UNDO；事务要么全部提交，要么全部回滚 |
| **Consistency（一致性）** | 约束检查 + 触发器 + 隔离性保证 |
| **Isolation（隔离性）** | Strict 2PL + 锁兼容性矩阵 |
| **Durability（持久性）** | WAL 日志 + 强制 fsync；提交后数据必定持久 |

### 五、生态系统与影响力 (Ecosystem & Impact)

#### 5.1 CAP 定位

**CAP 维度：不适用（N/A）**

System R 是**单机系统**，设计于 1974-1979 年，远早于分布式数据库时代。它运行在 IBM OS/360 大型机上，不涉及网络分区（Partition）问题。

- **C（Consistency）**：通过 2PL 实现强一致性和可串行化
- **A（Availability）**：单机系统，可用性取决于硬件可靠性
- **P（Partition Tolerance）**：无分布式部署，不存在分区容错问题

System R 的设计哲学是**单机 ACID**——在硬件可靠的前提下，通过软件机制确保事务的强一致性和持久性。这一范式主导了此后近 20 年的数据库设计。

**直接后裔**：

| 系统 | 年份 | 与 System R 的关系 |
|------|------|-------------------|
| **IBM DB2** | 1983 | System R 的直接商业继承者；继承了 Selinger 优化器和 System R 的存储架构 |
| **Oracle** | 1979 | 受 System R 论文启发；Larry Ellison 读到论文后决定实现 SQL |
| **SQL/DS** | 1981 | IBM 的第一个商用 SQL 产品，System R → SQL/DS → DB2 的中间步骤 |

**SQL 标准化历程**：

| 标准 | 年份 | 关键特性 |
|------|------|----------|
| SQL-86 (SQL1) | 1986 | 首个 ANSI/ISO 标准；基本 SQL 语法 |
| SQL-92 (SQL2) | 1992 | 完整的关系特性；JOIN 语法 |
| SQL:1999 (SQL3) | 1999 | 正则表达式、触发器、递归查询 |
| SQL:2003 | 2003 | XML 支持、窗口函数、序列 |
| SQL:2008 | 2008 | INSTEAD OF 触发器、FETCH FIRST |
| SQL:2011 | 2011 | 时间数据支持（temporal） |
| SQL:2016 | 2016 | JSON、模式匹配 |
| SQL:2023 | 2023 | 多态表函数、属性图查询 |

**所有现代 SQL 标准均可追溯至 System R 的 SEQUEL 设计**。

#### 5.3 历史意义

System R 是数据库历史上最重要的研究项目之一。它证明了：

1. **关系模型是可行的** — Codd 1970 年的理论模型可以在实际系统中高效实现
2. **声明式查询是实用的** — SQL 让非程序员也能操作数据库
3. **优化器可以自动化** — 基于代价的优化器可以比手动编码更高效
4. **ACID 可以在单机实现** — 2PL + WAL 构成了事务数据库的基石

**System R 虽然从未作为产品发布，但它是所有现代 RDBMS 的祖先。**
