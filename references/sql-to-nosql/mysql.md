# MySQL

## 5-Dim Analysis Framework

### 一、Context & Motivation — 背景与驱动力

#### 1.1 历史起点

MySQL 诞生于 **1995 年**，由 Michael "Monty" Widenius、David Axmark 和 Allan Larsson 在瑞典创立（MySQL AB）。它的名字来源于 Monty 的女儿 **My**，而不是 "My Structured Query Language"。

**时代背景：**
- 1990s 中期，关系数据库被 Oracle、DB2、Informix 等商业厂商垄断，许可证昂贵、部署复杂
- Web 正在爆发（LAMP 栈兴起），大量中小型网站需要轻量、免费、易用的数据库
- 互联网应用对 OLTP 的需求集中在**读多写少、简单查询、高并发**，不需要 Oracle 级别的重型功能

**核心设计哲学：**
> 快、简单、免费、够用 — 而不是大而全

#### 1.2 问题空间

| 问题 | 早期方案 | 痛点 |
|------|---------|------|
| 单机扩展 | 垂直加 CPU/内存/磁盘 | 硬件成本线性增长，存在天花板 |
| 高可用 | 无原生方案 | 单点故障 = 数据不可用 |
| 读扩展 | 无 | 大量读请求打满单节点 |
| 事务安全 | ISAM/MyISAM 不支持事务 | 崩溃后数据不一致 |
| 并发控制 | 表级锁 | 并发写完全串行化 |

#### 1.3 演进里程碑

| 年份 | 版本 | 关键事件 |
|------|------|----------|
| 1995 | 0.x | 首次内部发布 |

| 2023 | 8.4 (LTS) | LTS 版本策略，热备、窗口函数增强 |

#### 2.2 存储引擎插件架构 — 核心架构创新

MySQL 最关键的架构决策是 **存储引擎插件化**（Storage Engine Plugin Architecture），通过统一的 Handler API 抽象，允许不同的存储引擎在运行时被选择和替换。

```
        SQL 层 (Server)
              │
              │ 统一的 Handler API
              │ ┌─────────────────────────────────┐
              ├─│ handler::write_row()             │
              ├─│ handler::index_read()            │
              ├─│ handler::external_lock()         │
              ├─│ handler::rnd_next()              │
              ├─│ handler::delete_row()            │
              └─│ handler::start_stmt()            │
                └─────────────────────────────────┘
              │
    ┌─────────┼──────────┐
    ▼         ▼          ▼
 ┌──────┐ ┌───────┐ ┌────────┐
│InnoDB │ │MyISAM │ │MEMORY  │
│(B+树) │ │(ISAM) │ │(Hash)  │
 └──────┘ └───────┘ └────────┘
```

**设计优势：**
- **按需选型**：每张表可以独立指定存储引擎（`ENGINE=InnoDB` / `ENGINE=MyISAM`）
- **热插拔**：运行时可加载/卸载引擎插件（`INSTALL PLUGIN`）
- **生态开放**：第三方可开发自定义引擎（TokuDB、RocksDB、MyRocks、ColumnStore）
- **演进解耦**：SQL 层与存储引擎独立演进

**Handler API 核心接口：**

| 接口 | 职责 |
|------|------|
| `ha_open()` | 打开表，获取引擎句柄 |
| `ha_index_read()` | 索引查找/范围扫描 |
| `ha_rnd_init()` / `ha_rnd_next()` | 全表扫描迭代 |
| `ha_write_row()` / `ha_update_row()` / `ha_delete_row()` | 行级 DML |
| `ha_external_lock()` | 事务加锁/解锁 |
| `ha_start_stmt()` / `ha_commit()` / `ha_rollback()` | 事务生命周期 |
| `ha_index_init()` / `ha_index_end()` | 索引操作生命周期 |

#### 2.3 复制架构

MySQL 复制是其扩展性和高可用的基石，经历了三代演进：

```
┌─────────────────────────────────────────────────────────┐
│                   MySQL 复制演进                          │
│                                                          │
│  第一代：异步复制 (Async Replication) — 3.23+            │
│  ┌────────┐      binlog event      ┌──────────┐         │
│  │ Master ├──────────────────────►│ Slave    │         │
│  │        │                       │          │         │
│  │ 写完成 │                       │ 异步回放  │         │
│  │ 即返回 │                       │ SQL线程   │         │
│  └────────┘                       └──────────┘         │
│  风险：Master 宕机时 Slave 可能缺失最近事务               │
│                                                          │
│  第二代：半同步复制 (Semi-Sync) — 5.5+                   │
│  ┌────────┐      binlog event      ┌──────────┐         │
│  │ Master ├──────────────────────►│ Slave    │         │
│  │        │◄──────────────────────┤          │         │
│  │ 等待ACK│       ACK (收到)       │ 收到即回 │         │
│  │ 后返回 │                       │          │         │
│  └────────┘                       └──────────┘         │
│  保证：至少一个 Slave 收到 relay log 才返回客户端          │
│  超时降级：ACK 超时后自动降级为异步                        │
│                                                          │
│  第三代：组复制 (MGR) — 5.7.17+                          │
│  ┌────────┐     ┌────────┐     ┌────────┐               │
│  │ Node-A │◄───►│ Node-B │◄───►│ Node-C │               │
│  │        │ Paxos│        │ Paxos│        │               │
│  └────────┘     └────────┘     └────────┘               │
│  特点：分布式共识、多主/单主模式、自动故障转移               │
│  基于：XCom (Paxos 变种) + GCS (Group Communication)     │
└─────────────────────────────────────────────────────────┘
```

**异步复制数据流：**

```
Master                                Slave
  │                                     │
  │  1. 写事务                           │
  │     ↓                               │
  │  2. 写入 InnoDB redo log            │
  │     ↓                               │
  │  3. 写入 binlog (binary log)        │
  │     ↓                               │
  │  4. Commit (返回客户端) ─────────────┤
  │                                     │  5. I/O Thread 拉取 binlog
  │                                     │     写入 relay log
  │                                     │     ↓
  │                                     │  6. SQL Thread 回放 relay log
  │                                     │     写入本地 InnoDB
  │                                     │     ↓
  │                                     │  7. 应用完成
```

#### 2.4 GTID（全局事务标识符）

GTID 是 MySQL 5.6 引入的革命性复制增强：

```
GTID = source_id:transaction_id
     = server_uuid:sequence_number

示例: 3E11FA47-71CA-11E1-9E33-C80AA9429562:1-1024
```

**解决的问题：**

| 问题 | 传统 binlog position | GTID |
|------|---------------------|------|
| 故障转移定位 | 需人工找 binlog 位点 | 自动跳过已执行事务 |
| 拓扑变更 | 重新同步困难 | 自动识别执行集合 |
| 并行复制 | 无法保证一致性 | 天然幂等性 |
| 监控调试 | 位点不直观 | 事务级可追踪 |

**工作原理：**
1. 每个事务在 Master 上分配唯一 GTID
2. GTID 写入 binlog，随复制传播到 Slave
3. Slave 回放时检查 `gtid_executed` 集合，自动跳过已执行的事务
4. 幂等回放使得拓扑变更（主从切换、多源复制）变得安全简单

### 三、Core Technical Innovations — 核心技术创新

#### 3.1 InnoDB 存储引擎

##### 3.1.1 聚簇索引（Clustered Index）

InnoDB 的表存储本质上是 **索引组织表（Index-Organized Table, IOT）**。

```
聚簇索引结构 (B+ Tree)
══════════════════════════════════════════════════════════

              Root Page
           ┌──────────────┐
           │  pk=100 | P2 │
           │  pk=200 | P3 │
           └──┬───────┬───┘
              │       │
        ┌─────┘       └─────┐
        ↓                   ↓
  ┌─────────────┐     ┌─────────────┐
  │ pk=10  | R1 │     │ pk=150 | R6 │
  │ pk=25  | R2 │     │ pk=180 | R7 │
  │ pk=50  | R3 │     │ pk=190 | R8 │
  │ pk=80  | R4 │     │ pk=195 | R9 │
  │ pk=95  | R5 │     │ pk=199 | R10│
  └─────────────┘     └─────────────┘
    Leaf Pages          Leaf Pages
    (含完整行数据)        (含完整行数据)

══════════════════════════════════════════════════════════
二级索引结构
══════════════════════════════════════════════════════════

       sec_idx B+ Tree
    ┌────────────────────┐
    │ name="Alice" | 50  │──► 需回表到聚簇索引 pk=50 取完整行
    │ name="Bob"   | 25  │──► 需回表到聚簇索引 pk=25 取完整行
    │ name="Carol" | 80  │──► 需回表到聚簇索引 pk=80 取完整行
    └────────────────────┘
    叶子节点存储：(索引列值, 主键值)
    查询过程：二级索引定位 → 获取主键 → 聚簇索引回表 → 返回完整行
══════════════════════════════════════════════════════════
```

**关键特性：**
- **数据即索引**：叶子节点包含完整行数据，无需额外存储层
- **主键选择至关重要**：
  - 自增主键 → 顺序插入，页分裂少，写入高效
  - UUID 主键 → 随机插入，频繁页分裂，碎片化严重
- **无显式主键时**：InnoDB 自动生成 6 字节的 ROW_ID（全局竞争，性能差）
- **二级索引回表**：二级索引叶子节点存储 `(索引列, 主键值)`，查询需要额外回表（回表代价 = 额外一次 B+ 树搜索）
- **覆盖索引优化**：如果查询列全部在二级索引中，可避免回表

##### 3.1.2 MVCC（多版本并发控制）

MVCC 是 InnoDB 实现高并发读写的核心技术，使读操作不加锁、写操作不阻塞读。

```
MVCC 版本链
══════════════════════════════════════════════════════════

行记录的版本链 (通过 undo log 串联)
══════════════════════════════════════════════════════════

  当前最新版本 (trx_id=1050, del_flag=0)
  ┌──────────────────────────────────┐
  │ data: name="Charlie", age=30     │
  │ trx_id: 1050                     │
  │ roll_pointer ──────────────────┐ │
  └──────────────────────────────────┘ │
                                       │ undo log
                                       ▼
  ┌──────────────────────────────────┐
  │ data: name="Bob", age=28         │
  │ trx_id: 1030                     │
  │ roll_pointer ──────────────────┐ │
  └──────────────────────────────────┘ │
                                       │ undo log
                                       ▼
  ┌──────────────────────────────────┐
  │ data: name="Alice", age=25       │
  │ trx_id: 1000                     │
  │ roll_pointer ──────────────────┐ │
  └──────────────────────────────────┘ │
                                       │ undo log
                                       ▼
                               ┌──────────────┐
                               │ (初始版本)     │
                               └──────────────┘

ReadView 可见性判断 (快照读)
══════════════════════════════════════════════════════════
  ReadView {
    m_ids:       [1010, 1020, 1030, 1040]   // 活跃事务 ID 列表
    m_low_limit: 1050                        // ≥ 此值 = 未来事务，不可见
    m_up_limit:  1010                        // < 此值 = 已提交事务，可见
    m_creator:   1030                        // 创建此 ReadView 的事务 ID
  }

  判断规则 (对每行记录的 trx_id):
    1. trx_id < m_up_limit     → 可见 (已提交)
    2. trx_id ≥ m_low_limit    → 不可见 (未来事务)
    3. trx_id in m_ids         → 不可见 (活跃中)
    4. trx_id == m_creator     → 可见 (自己的修改)
    5. 否则                     → 可见 (已提交)

  遍历版本链直到找到第一个可见版本
══════════════════════════════════════════════════════════
```

**隔离级别与 MVCC 行为：**

| 隔离级别 | 快照读行为 | 幻读 |
|----------|-----------|------|
| READ UNCOMMITTED | 总是读最新版本 | 有 |
| READ COMMITTED | 每次 SELECT 创建新 ReadView | 有 |
| REPEATABLE READ (默认) | 事务内第一次 SELECT 创建 ReadView，复用 | 无 (Next-Key Lock) |
| SERIALIZABLE | 所有读加锁 (退化为 2PL) | 无 |

**核心数据结构：**
- **Undo Log**：存储旧版本数据，形成版本链
- **ReadView**：快照时刻的活跃事务快照
- **隐藏列**：每行有 `trx_id`（最后修改事务 ID）和 `roll_pointer`（指向 undo log 的回滚指针）
- **Purge Thread**：后台线程清理不再需要的 undo log 版本

##### 3.1.3 Redo Log（重做日志）— WAL 实现

Redo Log 是 InnoDB 实现 **WAL（Write-Ahead Logging）** 的核心机制，保证崩溃恢复时的持久性。

```
Redo Log 架构
══════════════════════════════════════════════════════════

  内存中的 redo log buffer ──────────► 磁盘上的 redo log file
  ┌──────────────────────┐           ┌────────────────────┐
  │                      │    fsync  │ ib_logfile0        │
  │  MLog Header         │─────────► │ (循环写入)          │
  │  LSN: 1000           │           │ ib_logfile1        │
  │                      │           │                    │
  │  [LSN 1000] INSERT   │           │ 物理页的增量修改    │
  │  [LSN 1001] UPDATE   │           │ ("以日志形式描述   │
  │  [LSN 1002] DELETE   │           │  页内字节偏移变化") │
  │  ...                 │           │                    │
  │                      │           │ 循环写入 (circular) │
  │  checkpoint_lsn      │           │ 覆盖前先刷脏页      │
  └──────────────────────┘           └────────────────────┘

  刷盘策略 (innodb_flush_log_at_trx_commit):
    0: 每秒刷盘 (最快，丢1秒数据)
    1: 每次事务提交刷盘 (默认，最安全)
    2: 每次事务提交写OS缓存，每秒fsync (折中)

  Two-Phase Commit (binlog + redo log):
    1. 写 redo log (prepare 状态)
    2. 写 binlog
    3. 写 redo log (commit 状态)
    崩溃恢复:
      - redo commit + binlog 存在 → 正常
      - redo prepare + binlog 存在 → 提交
      - redo prepare + binlog 缺失 → 回滚
══════════════════════════════════════════════════════════
```

**Redo Log 的关键设计：**
- **物理日志**：记录的是数据页的字节级修改（"page X offset Y: 将字节 Z 改为 W"），非逻辑 SQL
- **LSN（Log Sequence Number）**：全局单调递增的日志序列号，用于定位和一致性校验
- **循环写入**：redo log 是固定大小的循环缓冲区，通过 checkpoint 机制保证不会覆盖未持久化的修改
- **Group Commit**：多个事务的 redo log 合并一次 fsync，大幅降低 I/O 次数

**Two-Phase Commit（与 binlog 协调）：**
MySQL 有两份日志（InnoDB redo log 和 Server binlog），必须通过 2PC 保证一致性，否则崩溃恢复时可能出现主从不一致。

##### 3.1.4 Undo Log（回滚日志）— MVCC 与回滚

Undo Log 承担双重职责：
1. **事务回滚**：提供恢复到事务前状态的逆操作
2. **MVCC 多版本**：构建行的历史版本链，支持快照读

```
Undo Log 组织
══════════════════════════════════════════════════════════

  Segment 层次结构:
  
  Undo Tablespace (undo tablespace)
    │
    ├── Rollback Segment (回滚段, 默认128个)
    │     │
    │     └── Undo Log (回滚日志)
    │           │
    │           └── Undo Page → Undo Record → 行版本
    │
    └── Insert Undo (INSERT 专用, 无其他事务可见, 事务提交即可清理)
    └── Update Undo (UPDATE/DELETE 用, 需要保留给 MVCC 快照读)

  记录格式:
  ┌─────────────────────────────────┐
  │ Undo Log Record                 │
  │  - type (insert/update/delete)  │
  │  - table id                     │
  │  - trx id                       │
  │  - roll ptr (指向上一条 undo)    │
  │  - 旧版本数据                    │
  └─────────────────────────────────┘
══════════════════════════════════════════════════════════
```

##### 3.1.5 Doublewrite Buffer（双写缓冲）

Doublewrite Buffer 是 InnoDB 解决 **部分写（Partial Write）** 问题的精巧设计。

```
问题：操作系统页大小 (4KB) ≠ InnoDB 页大小 (16KB)
      16KB 写入不是原子操作，宕机时可能只写了部分页

解决方案：Doublewrite Buffer

  内存脏页 ──────────────────────────────────────┐
                                                 ↓
    ┌──────────────────────────────────────────────────┐
    │ Step 1: 写入 Doublewrite Buffer (顺序写)          │
    │   - 位于系统表空间 (ibdata1)                       │
    │   - 连续写入 100 页 (顺序 I/O)                     │
    │   - 此时即使崩溃，原始页仍完好                       │
    └──────────────────────────────────────────────────┘
                         ↓
    ┌──────────────────────────────────────────────────┐
    │ Step 2: 写入实际数据文件 (分散写)                   │
    │   - 写入对应的 .ibd 文件                           │
    │   - 如果中途崩溃 → 恢复时从 Doublewrite Buffer 重放│
    │   - 如果成功完成 → Doublewrite Buffer 可复用       │
    └──────────────────────────────────────────────────┘

  崩溃恢复逻辑:
    1. 检查数据页是否完整 (page checksum)
    2. 不完整 → 从 Doublewrite Buffer 恢复完整页
    3. 完整 → 通过 redo log 前滚到最新状态
══════════════════════════════════════════════════════════
```

**为什么需要：**
- Linux ext4 文件系统不保证 16KB 写原子性
- 即使 SSD 也不能保证多页写入的原子性
- 不使用 Doublewrite Buffer 的情况下，部分写的损坏页无法通过 redo log 修复（redo log 基于完整页的增量）

**性能代价：**
- 额外一次顺序写（但顺序写代价很低）
- 在某些文件系统（支持原子写）和现代 SSD 上可禁用

##### 3.2.1 基于 binlog 的异步复制

复制线程模型
══════════════════════════════════════════════════════════

  Master:                    Slave:
  ┌─────────┐               ┌──────────────────┐
  │ Dump    │               │ I/O Thread       │
  │ Thread  ├──────────────►│  - 连接 Master    │

  └─────────┘                        │
                                     ↓
                               ┌──────────────────┐
                               │ SQL Thread       │

##### 3.2.2 半同步复制 (Semi-Synchronous Replication)

```
半同步数据流
══════════════════════════════════════════════════════════

  Client          Master                     Slave
    │               │                          │
    │── INSERT ────►│                          │
    │               │ 1. 写 InnoDB (本地)       │
    │               │ 2. 写 binlog             │
    │               │ 3. 发送 binlog event ────►│
    │               │                          │ 收到后写入 relay log
    │               │◄─────── ACK ──────────────│
    │               │ 4. 返回客户端             │
    │◄── OK ────────│                          │
    │               │                          │ 5. SQL Thread 回放

  关键: Master 在步骤 3-4 之间等待至少一个 Slave 的 ACK
  超时 (rpl_semi_sync_master_timeout, 默认10s): 降级为异步
  恢复: Slave 追上后自动恢复半同步
══════════════════════════════════════════════════════════
```

##### 3.2.3 Group Replication (MGR) — 分布式共识

MySQL Group Replication (MGR) 是 5.7.17 引入的基于 **Paxos 变种**的分布式一致性复制方案。

```
MGR 架构
══════════════════════════════════════════════════════════

  ┌────────────────────────────────────────────────────────┐
  │                  MySQL Group Replication                │
  │                                                        │
  │  ┌──────────┐  XCom (Paxos)  ┌──────────┐             │
  │  │  Node-A  │◄──────────────►│  Node-B  │             │
  │  │ Primary  │                │ Secondary│             │
  │  └────┬─────┘                └────┬─────┘             │
  │       │                          │                    │
  │       └──────────┬───────────────┘                    │
  │                  │                                    │
  │                  ▼                                    │
  │          ┌──────────────┐                             │
  │          │   Node-C     │                             │
  │          │  Secondary   │                             │
  │          └──────────────┘                             │
  │                                                        │
  │  通信层 (GCS - Group Communication System):             │
  │    - 基于 XCom (eXtended Communication)                │
  │    - Paxos 协议的 MySQL 实现                            │
  │    - 负责消息排序、成员管理、故障检测                    │
  │                                                        │
  │  模式:                                                  │
  │    - 单主模式 (Single-Primary): 仅 Primary 接受写       │
  │    - 多主模式 (Multi-Primary): 所有节点可接受写          │
  └────────────────────────────────────────────────────────┘

  单主模式写入流程:
  ┌────────────────────────────────────────────────────────┐
  │ 1. 客户端写入 Primary 节点                              │
  │ 2. Primary 生成写集 (Write Set) — 行的哈希集合          │
  │ 3. 通过 XCom 将 Write Set 广播到所有节点 (Paxos propose)│
  │ 4. 所有节点对 Write Set 进行冲突检测                     │
  │    - 无冲突 → 本地应用                                  │
  │    - 有冲突 → 回滚该事务                                │
  │ 5. 多数节点确认后提交 (Paxos commit)                    │
  │ 6. 返回客户端                                          │
  └────────────────────────────────────────────────────────┘

  冲突检测机制:
    - Write Set 包含: 被修改行的哈希 (based on PK/UK)
    - 两个事务修改同一行 → 冲突
    - 先提交的赢，后提交的回滚 (certifier-based)
    - 多主模式: 跨节点冲突检测需要额外延迟
══════════════════════════════════════════════════════════
```

**XCom (Paxos 实现) 核心：**
- **日志复制**：每个提议的消息追加到 Paxos 日志，多数节点确认后提交
- **成员变更**：通过 Paxos 协议协商组成员变化（节点加入/离开/故障）
- **故障检测**：基于心跳的分布式故障检测（ suspicion 机制）
- **消息排序**：全局一致的日志顺序，保证所有节点按相同顺序应用事务

**MGR 的 CAP 定位：**
- 单主模式：CP（强一致，分区时 Primary 不可写）
- 多主模式：理论上更偏向 AP，但实际通过冲突检测保证最终一致

### 四、Trade-offs — 权衡分析

#### 4.1 CAP 分析

| 模式 | 一致性 (C) | 可用性 (A) | 分区容忍 (P) | 说明 |
|------|-----------|-----------|-------------|------|
| **单机 InnoDB** | ✅ 强一致 (ACID) | ⚠️ 单机不可用 | N/A | 单机场景，P 不适用 |
| **异步复制** | ⚠️ 最终一致 | ✅ 高可用 | ✅ | 主从延迟 = 不一致窗口 |
| **半同步复制** | ⚠️ 近强一致 | ⚠️ 超时降级异步 | ✅ | ACK 超时降级为异步 |
| **MGR 单主** | ✅ 强一致 | ⚠️ 主节点宕机期间不可写 | ✅ | Paxos 保证一致性，但主选期间写不可用 |
| **MGR 多主** | ⚠️ 最终一致 | ✅ 多节点可写 | ✅ | 冲突检测保证最终一致，但延迟冲突发现 |

#### 4.2 PACELC 分析

| 模式 | 正常态 (No Partition) | 分区态 (Partition) |
|------|----------------------|-------------------|
| **单机** | 一致性 (Latency 可接受) | N/A |
| **异步复制** | **可用性** (读扩展，允许延迟) | **可用性** (Slave 可读，可能旧数据) |
| **半同步** | **一致性** (等待 ACK) | **可用性** (超时降级异步) |
| **MGR 单主** | **一致性** (Paxos 共识) | **一致性** (主不可写) |
| **MGR 多主** | **延迟** (冲突检测增加延迟) | **可用性** (各分区独立写入，冲突后解决) |

#### 4.3 核心 Trade-offs

##### 存储引擎选择

| 权衡维度 | InnoDB | MyISAM |
|----------|--------|--------|
| 事务 | ✅ ACID | ❌ 无 |
| 锁粒度 | 行级锁 | 表级锁 |
| 崩溃恢复 | ✅ redo/undo | ❌ 需 myisamchk |
| 外键 | ✅ | ❌ |
| 全文索引 | ✅ (5.6+) | ✅ |
| COUNT(*) | 慢 (需扫描) | 快 (元数据) |
| 空间占用 | 较大 (MVCC + 事务开销) | 较小 |
| 适用场景 | OLTP 通用 | 只读/读密集、全文搜索 |

##### 复制模式选择

| 权衡维度 | 异步 | 半同步 | MGR |
|----------|------|--------|-----|
| 延迟 | 最低 | 中等 (等待 ACK) | 最高 (Paxos 共识) |
| 一致性 | 最终一致 | 近强一致 | 强一致 (单主) |
| 吞吐 | 最高 | 中等 | 最低 |
| 运维复杂度 | 低 | 中 | 高 |
| 适用场景 | 读扩展、缓存 | 关键业务高可用 | 金融级强一致 |

##### InnoDB 内部权衡

| 设计 | 收益 | 代价 |
|------|------|------|
| **聚簇索引** | 主键查询极快、范围扫描高效 | 二级索引需回表、随机主键导致页分裂 |
| **MVCC** | 读写不互斥、高并发快照读 | Undo log 存储开销、版本链遍历成本 |
| **Doublewrite Buffer** | 防止部分写损坏 | 额外一次顺序写 |
| **Redo Log (WAL)** | 崩溃恢复保证、随机写变顺序写 | 日志写入开销、Two-Phase Commit 复杂度 |
| **Change Buffer** | 非唯一索引写合并，减少随机 I/O | 读放大 (需合并后查询)、崩溃恢复时 merge |

### 五、Influence & Legacy — 影响与遗产

#### 5.1 行业影响

**开源数据库生态基石：**
- MySQL 是最成功的开源关系数据库，LAMP 栈的 "M"
- 2008 年被 Sun Microsystems 收购（$10 亿），2010 年随 Sun 被 Oracle 收购
- Oracle 收购后社区分裂出 **MariaDB**（Monty 创建），保持了开源路线
- 衍生出 **Percona Server**（性能增强分支）

**存储引擎生态：**
| 引擎 | 开发者 | 特点 |
|------|--------|------|
| InnoDB | Oracle (原 Innobase Oy) | 默认引擎、事务、MVCC |
| MyRocks | Facebook | 基于 RocksDB、LSM-Tree、高压缩比 |
| TokuDB | Percona/Tokutek | Fractal Tree 索引、高压缩 |
| ColumnStore | MariaDB | 列式存储、OLAP |
| NDB Cluster | Oracle | 内存分布式、分片 |
| Spider | MariaDB | 分布式联邦引擎 |

#### 5.2 技术遗产

**1. 存储引擎插件化 → NewSQL 的灵感**

MySQL 的存储引擎插件架构证明了 **"SQL 层与存储层解耦"** 的可行性，直接启发了：

- **TiDB**：SQL 层（TiDB）与存储层（TiKV）彻底分离
- **CockroachDB**：SQL 层与 Raft 存储层分离
- **Vitess**：在 MySQL 之上做水平分片，证明插件化的可组合性

**2. 异步复制 → 现代复制范式的基础**

MySQL 的 binlog 复制模型成为行业标杆：
- 几乎所有关系数据库都采用了类似的主从复制模型
- binlog 格式启发了 **CDC（Change Data Capture）** 生态（Debezium、Canal、Maxwell）
- Canal（阿里）→ 解析 MySQL binlog 做实时数据同步

**3. InnoDB 的 MVCC + B+ Tree → 行业标准**

- PostgreSQL 也使用 MVCC + 堆表（实现不同，理念一致）
- Oracle 使用 undo segment 构建一致性读（与 InnoDB 的 undo log 同源理念）
- 现代数据库几乎全部采用 MVCC 作为并发控制基础

**4. 半同步复制 → CAP 的实用主义**

MySQL 5.5 的半同步复制展示了一种**实用的 CAP 妥协**：
- 不强求完全一致，也不放任完全异步
- 通过超时降级机制在一致性和可用性之间动态切换
- 这种"渐进式一致性"的思路影响了后续许多系统的设计

**5. MGR → 从单机到分布式的桥梁**

- MGR 是 MySQL 向分布式共识迈出的关键一步
- 虽然性能和规模不及 TiDB/CockroachDB，但为现有 MySQL 用户提供了平滑升级路径
- XCom 的实现证明了在现有 RDBMS 中集成 Paxos 的可行性

#### 5.3 局限性

| 局限 | 说明 | 后续方案 |
|------|------|----------|
| **水平扩展** | 原生不支持自动分片，需中间件 (Vitess/MyCat) 或 MGR (有限规模) | TiDB、CockroachDB、Vitess |
| **HTAP** | 仅支持 OLTP，分析查询性能差 | TiDB (TiFlash)、PolarDB |
| **多写** | MGR 多主模式冲突检测开销大，多写场景受限 | Spanner、OceanBase |
| **全局事务** | 不支持分布式事务 (XA 有限)，跨库事务困难 | Seata、Percolator 模式 |
| **JSON/文档** | 8.0 支持 JSON 但查询性能不及 MongoDB | MongoDB、文档数据库 |
| **写放大** | B+ Tree 随机插入导致页分裂、Change Buffer merge | MyRocks (LSM-Tree) |
