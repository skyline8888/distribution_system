## 1. 概述

字节跳动作为全球规模最大的短视频平台之一（抖音 + TikTok 日活超 10 亿），其底层基础设施必须满足以下核心需求：

- **超大规模视频存储与分发**：每天新增 EB 级别视频数据
- **实时推荐**：毫秒级召回 + 排序，支撑个性化 feed 流
- **高并发写入与读取**：点赞、评论、弹幕等海量实时交互
- **AI 训练基础设施**：支持 CV/NLP/多模态大模型的分布式训练
- **全球化部署**：多区域、多 AZ 容灾，满足不同地区合规要求

字节跳动的技术栈分为两层：

| 层级 | 名称 | 定位 |
|------|------|------|
| 内部自研 | ByteStore / TOS / VePFS / ByteHouse 等 | 支撑抖音、TikTok、西瓜视频等核心业务 |
| 商业化输出 | 火山引擎 (Volcengine) | 面向企业客户的云服务，将内部能力产品化 |

## 3. 存储层

### 3.1 ByteStore — 分布式块存储

**定位**：为计算节点（ECS/虚拟机）提供低延迟、高可靠的持久化块存储。

**核心特性：**

| 维度 | 设计 |
|------|------|
| 数据模型 | 块设备抽象，支持快照、克隆、增量备份 |
| 一致性模型 | 强一致读写（单副本内），跨副本多 Paxos/Raft |
| 持久性 | 多副本跨机架分布，支持 99.999999999% (11 个 9) 数据持久性 |
| 性能 | 单卷百万级 IOPS，亚毫秒延迟（NVMe SSD 后端） |
| 容量 | 单卷最高数十 TB，支持在线扩容 |

**架构设计：**

```
┌─────────────────────────────────────────────┐
│                  ByteStore                    │
│                                               │
│  ┌─────────────┐    ┌─────────────────────┐  │
│  │  管理平面    │    │     数据平面         │  │
│  │  (Meta Mgr) │    │                     │  │
│  │             │    │  ┌───────────────┐  │  │
│  │  - 卷分配    │    │  │ Storage Node │  │  │
│  │  - 快照管理  │───▶│  │ (多副本组)    │  │  │
│  │  - 迁移调度  │    │  │  Raft/Paxos   │  │  │
│  └─────────────┘    │  └───────┬───────┘  │  │
│                     │          │          │  │
│                     │  ┌───────▼───────┐  │  │
│                     │  │ Storage Node │  │  │
│                     │  │ (多副本组)    │  │  │
│                     │  └───────┬───────┘  │  │
│                     │          │          │  │
│                     │  ┌───────▼───────┐  │  │
│                     │  │ Storage Node │  │  │
│                     │  │ (多副本组)    │  │  │
│                     │  └───────────────┘  │  │
│                     └─────────────────────┘  │
└─────────────────────────────────────────────┘
```

**关键技术点：**

- **副本管理**：采用 Raft 或类 Paxos 协议确保多副本一致性，典型 3 副本部署
- **IO 路径优化**：用户态 NVMe driver + SPDK 绕过内核栈，减少 context switch
- **快照机制**：Copy-on-Write (CoW) 快照，增量快照链
- **QoS**：按卷配置 IOPS/带宽上限，支持突发配额
- **热迁移**：存储卷可在线迁移，支持计算节点无感切换

**火山引擎对应产品**：弹性云盘 (EBS / Cloud Disk)

### 3.2 TOS — 对象存储 (Tinder Object Storage)

**定位**：海量非结构化数据存储，核心支撑抖音/TikTok 的视频、图片、音频内容。

**核心特性：**

| 维度 | 设计 |
|------|------|
| 数据模型 | S3 兼容的对象存储 (Bucket → Object) |
| 容量规模 | EB 级别，单 bucket 无对象数限制 |
| 持久性 | 多 AZ 冗余，11 个 9 持久性 |
| 可用性 | 99.99%+ SLA |
| 吞吐 | 单集群 TB/s 级吞吐，支持高并发 PUT/GET |
| 接口 | S3 API 兼容 + 自有 SDK |
| 分层 | 热/温/冷/归档 多级存储，自动生命周期管理 |

**架构设计：**

```
┌────────────────────────────────────────────────────────┐
│                        TOS 对象存储                      │
│                                                         │
│  ┌─────────────────┐  ┌──────────────────────────────┐ │
│  │   网关/接入层     │  │         元数据服务            │ │
│  │  ┌───────────┐  │  │  ┌────────────────────────┐  │ │
│  │  │ S3 API    │  │  │  │ Meta Server (KV/ETCD)  │  │ │
│  │  │ RESTful   │  │  │  │ - Object metadata      │  │ │
│  │  │ 认证/限流  │  │  │  │ - Bucket ACL           │  │ │
│  │  └─────┬─────┘  │  │  │ - Multipart tracking   │  │ │
│  │        │        │  │  └────────────┬───────────┘  │ │
│  └────────┼────────┘  └───────────────┼──────────────┘ │
│           │                           │                │
│           ▼                           ▼                │
│  ┌──────────────────────────────────────────────────┐  │
│  │                  数据分发层                        │  │
│  │  ┌────────────┐ ┌────────────┐ ┌──────────────┐  │  │
│  │  │ Chunk Svr  │ │ Chunk Svr  │ │  Chunk Svr   │  │  │
│  │  │ (分块存储)  │ │ (分块存储)  │ │  (分块存储)   │  │  │
│  │  │ Erasure    │ │ Erasure    │ │  Erasure     │  │  │
│  │  │ Coding     │ │ Coding     │ │  Coding      │  │  │
│  │  └────────────┘ └────────────┘ └──────────────┘  │  │

│  ┌──────────────────────────────────────────────────┐  │

│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**关键技术点：**

- **分块存储 + 纠删码 (Erasure Coding)**：大对象切分为 Chunk，采用 RS (Reed-Solomon) 纠删码而非纯多副本，显著降低存储成本。典型配置 N+M = 12+4，容忍 4 个 chunk 丢失
- **分层存储**：
  - 热存储 (SSD)：高频访问的热门视频
  - 标准存储 (HDD)：日常视频内容
  - 低频存储 (大容量 HDD)：历史内容
  - 归档存储 (磁带/蓝光)：合规存档
- **多版本 & 生命周期**：自动过期策略，减少无效数据占用
- **跨区域复制**：支持跨 Region 异步复制，用于全球化部署
- **CDN 回源优化**：与 CDN 深度集成，智能回源 + 预热

**火山引擎对应产品**：TOS (对象存储)，S3 兼容接口

### 3.3 VePFS — 并行文件系统 (AI 训练专用)

**定位**：面向 AI/ML 分布式训练的高性能并行文件系统，解决 GPU 训练集群的数据供给瓶颈。

**核心特性：**

| 维度 | 设计 |
|------|------|
| 数据模型 | POSIX 兼容的并行文件系统 |
| 聚合吞吐 | 单文件系统 TB/s 级聚合带宽 |
| 元数据性能 | 百万级文件/秒操作能力 |
| 客户端 | FUSE 客户端 + 内核态优化驱动 |
| 适用场景 | 大模型预训练数据集加载、Checkpoint 读写 |
| 协议 | POSIX + HDFS 兼容 |

**为什么需要 VePFS（而非 TOS）：**

```
TOS (对象存储)                     VePFS (并行文件系统)
───────────────                    ─────────────────────
✓ 海量容量, 低成本                  ✓ 高 IOPS, 低延迟
✗ 随机读写性能差                    ✓ 小文件高效访问
✗ 非 POSIX 兼容                    ✓ POSIX 语义
✗ 元数据操作慢                      ✓ 目录树/文件遍历高效
```

**关键技术点：**

- **元数据分片**：目录树按 hash 前缀分片到多个 MDS 节点，支持水平扩展
- **数据分条 (Striping)**：文件按固定大小 (如 4MB/8MB) 分条到多个 OSS 节点，实现并行读写
- **客户端缓存**：智能预读 + 本地 page cache 增强，减少网络往返
- **Checkpoint 优化**：大模型训练 Checkpoint 写入优化（异步落盘 + 并发条带化）
- **与 TOS 集成**：冷数据可自动下沉到 TOS，释放高性能存储

**火山引擎对应产品**：VePFS (并行文件系统)

## 9. CAP 理论权衡

在分布式系统中，CAP 三选二的权衡在字节跳动的不同系统中各有侧重：

### 9.1 各系统的 CAP 定位

```
CAP 三角:

        Consistency (一致性)
               ▲
              / \
             /   \
            /     \
           /       \
  ┌───────┤         ├───────┐
  │  ADB  │         │ TOS   │
  │ (CP)  │         │ (AP)  │
  │强一致 │         │最终   │
  │  OLTP │         │ 一致  │
  │       │         │       │
  ├───────┤         ├───────┤
  │       │         │       │
  │ByteStore│       │ByteHouse│
  │ (CP)  │         │ (AP)  │
  │  块存储 │         │  分析  │
  └───────┤         ├───────┘
           \       /
            \     /
             \   /
              \ /
               ▼
        Availability (可用性)

        ───────────────────────▶
           Partition Tolerance
           (所有系统都选 P)
```

### 9.2 详细分析

| 系统 | CAP 选择 | 原因 | 一致性级别 |
|------|----------|------|-----------|
| **ByteStore** | CP | 块存储不能容忍数据不一致，否则文件系统损坏 | 强一致 (Raft/Paxos) |
| **TOS** | AP (偏 CA) | 海量对象存储，容忍短暂的副本不一致以保障可用性 | 最终一致 (异步复制) |
| **VePFS** | CP | 文件系统语义要求强一致 | 强一致 (元数据 Raft) |
| **ByteHouse** | AP | OLAP 场景允许数据延迟可见，追求高吞吐查询 | 最终一致 (异步写入) |
| **ADB** | CP | 事务处理要求强一致 | 强一致 (同步/半同步复制) |
| **推荐系统** | AP | 推荐结果允许不完美，但不能不可用 | 最终一致 + 缓存降级 |
| **CDN** | AP | 边缘缓存天然最终一致 | 最终一致 (TTL 过期) |

### 9.3 实际工程中的妥协

在字节跳动的实际系统中，CAP 并非严格二选一，而是通过工程手段做更精细的权衡：

- **TOS**：热数据多副本强一致，冷数据 EC 最终一致
- **ByteHouse**：写入走同步副本保障可见性，查询走异步副本提升吞吐
- **推荐系统**：核心召回用强一致向量库，重排允许降级到缓存结果
- **ADB**：主从切换时短暂降级为最终一致，快速恢复后追赶

## 10. 关键设计模式与取舍

### 10.1 存算分离

字节跳动的核心存储系统 (ByteHouse、TOS) 均采用 **存算分离** 架构：

```
存算耦合 (传统)              存算分离 (云原生)
─────────────              ─────────────────
计算+存储绑定               计算节点无状态
扩容需同时增加两者           计算/存储独立扩展
资源利用率低                资源利用率高
故障恢复慢                  故障快速迁移
```

**优势**：
- 弹性伸缩：计算层秒级扩缩容
- 成本优化：存储按量计费，计算按需分配
- 高可用：无状态计算节点可快速迁移

### 10.2 统一存储底座

```
          统一存储底座
        ┌──────────────┐
        │   元数据服务   │

                │
    ┌───────────┼───────────┐
    │           │           │
    ▼           ▼           ▼
┌───────┐  ┌───────┐  ┌───────┐
│ByteStore│ │  TOS  │ │ VePFS │
│ 块接口  │ │对象接口│ │POSIX  │

```

底层共享统一的元数据管理和数据分发能力，降低运维复杂度。

### 10.4 核心技术取舍总结

| 决策点 | 选择 | 替代方案 | 取舍原因 |
|--------|------|----------|----------|
| 对象存储数据保护 | 纠删码 > 多副本 | 3 副本 | 容量成本降低 50%+ |
| OLAP 引擎 | ClickHouse 内核优化 | Doris/StarRocks | ClickHouse 单表查询性能极致 |
| 数据库协议 | MySQL 兼容 | PostgreSQL | 生态/人才/迁移成本 |
| 实时计算 | Flink | Spark Streaming | 真正的流处理, 低延迟 |
| AI 训练存储 | VePFS (并行 FS) | TOS / NFS | 小文件随机读性能需求 |
| CDN 协议 | QUIC/HTTP3 | TCP/HTTP2 | 弱网环境性能提升 |
| 推荐系统 | 自研 Serving 框架 | Triton/TFServing | 定制化深度优化 |
