## 一、存储基础设施层

### 1.1 CBS（Cloud Block Storage）— 云盘存储

#### 基本信息
- **类型:** 分布式块存储
- **接口:** iSCSI / NVMe-oF（内部）
- **适用场景:** 云主机系统盘/数据盘、数据库底层存储

#### Architecture

```
┌────────────────────────────────────────────────────┐
│                    CVM (云主机)                     │
│         ┌──────────────────────────────────┐        │
│         │  Block Device (/dev/vda, nvme)   │        │
│         │  Kernel I/O Stack → virtio/NVMe  │        │
│         └────────────────┬─────────────────┘        │
└──────────────────────────┼──────────────────────────┘
                           │ iSCSI / NVMe-oF
┌──────────────────────────┼──────────────────────────┐
│                CBS 接入层 (Storage Gateway)           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐             │
│  │ Gateway-1│ │ Gateway-2│ │ Gateway-N│  多路径冗余   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘             │
│       │            │            │                    │
│  ┌────┴────────────┴────────────┴──────────────────┐ │
│  │              CBS 分布式存储后端                    │ │
│  │                                                   │ │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐          │ │
│  │  │ Node-A  │  │ Node-B  │  │ Node-C  │  三副本   │ │
│  │  │ (Leader)│  │(Follower)│ │(Follower)│  Paxos   │ │
│  │  └─────────┘  └─────────┘  └─────────┘          │ │
│  │                                                   │ │
│  │  数据分布: 条带化 + 副本复制 (默认 3x)              │ │
│  │  快照: COW (Copy-on-Write) 增量快照               │ │
│  │  热迁移: 在线数据迁移无感切换                        │ │
│  └──────────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────────┘
```

#### 元数据架构
- **归类:** 集中式分区 + 分布式 Paxos
- CBS 的元数据（卷映射、快照链、QoS 策略）由集中管控服务管理，但每个卷的数据副本组内通过 Paxos 实现分布式一致性

#### 数据架构
- **分块:** 固定大小块（内部 chunk），支持条带化分布在多个存储节点
- **分布:** 基于一致性哈希映射到存储池，自动均衡
- **冗余:** 默认 3 副本同步复制，可选纠删码（冷数据）
- **I/O 路径:** 内核态 → virtio-blk/NVMe 前端 → iSCSI/NVMe-oF 后端
- **生命周期:** 热/温/冷分层，快照增量链，支持跨 AZ 复制（灾难恢复）

#### 核心技术创新
- **热迁移:** CBS 卷可以在运行时跨物理存储节点迁移，CVM 无感知
- **快照加速:** COW + 增量链，支持秒级快照创建
- **QoS 隔离:** IOPS/带宽/延迟三维度限流，防止 noisy neighbor
- **NVMe-oF 支持:** 新一代 CBS 使用 NVMe-over-Fabrics 提供更低延迟

#### CAP / PACELC 定位
- **CAP: CP** — 块存储要求强一致性，分区时宁可牺牲可用性也不返回脏数据
- **PACELC: P→C / E→L** — 分区时选一致性（C），正常时优先低延迟（L）
- **权衡:** 三副本同步写入增加延迟（~2x RTT），但保证数据零丢失

#### 性能指标（官方公开数据参考）
- 最大 IOPS: 50,000+（ESSD 类型）
- 最大吞吐量: 1,500+ MB/s
- 最低延迟: 亚毫秒级（NVMe 路径）

- **发布时间:** 2014 年正式商用

#### 元数据架构
- **归类:** 集中式分区 + 外部引擎（类 Bigtable 元数据存储）
- Bucket/Object 元数据存储在内部 KV 存储上，按 Bucket ID 哈希分片
- 元数据服务无状态，水平扩展

#### 数据架构
- **分块:** 大对象自动分片（multipart upload，默认单 chunk 20-100MB）
- **分布:** 一致性哈希映射到存储集群
- **冗余:** 多 AZ 多副本（默认 3x），冷数据使用纠删码（EC 4+2 或类似比例）
- **I/O 路径:** REST → 接入网关 → 元数据路由 → 数据节点
- **生命周期:**
  - 标准存储: 高频访问，低延迟
  - 低频存储: 较少访问，更低单价
  - 归档存储: 极少访问，取回需解冻
  - 深度归档: 合规归档，最长取回时间

#### 核心技术创新
- **海量小文件优化:** 微信/QQ 产生的海量图片/文件推动 COS 对小对象场景的优化
- **CDN 深度集成:** COS + CDN 联动，热点内容自动推送到边缘
- **智能分层:** 基于访问频率自动迁移存储层级
- **跨区域复制 (CRR):** 异步复制实现灾难恢复和就近访问

#### CAP / PACELC 定位
- **CAP: AP（同区域读）/ CP（同区域写）** — 同区域写入强一致（Paxos），跨区域复制最终一致
- **PACELC: P→A / E→L** — 分区时可降级为只读（A），正常时优先低延迟（L）
- **跨区域复制:** 典型的最终一致性（AP），接受延迟换取可用性

#### 性能指标
- 设计容量: EB 级（官方不公开上限）
- 对象大小: 0 B ~ 5 TB（单对象）
- 吞吐量: 取决于请求类型和并发，支持万级 QPS/桶

### 1.3 CFS（Cloud File Storage）— 文件存储

#### 基本信息
- **类型:** 分布式文件存储
- **接口:** NFS v3/v4, SMB/CIFS
- **适用场景:** 容器共享存储、跨主机文件共享、媒体渲染、HPC

#### Architecture

```
┌────────────────────────────────────────────────────────┐
│                      CFS 架构                           │
│                                                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐               │
│  │ Client-1  │ │ Client-2  │ │ Client-N │  NFS/SMB     │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘               │
│       │            │            │                      │
│  ┌────▼────────────▼────────────▼────────────────┐     │
│  │              协议转换层 (Protocol Gateway)       │     │
│  │  NFS Server / SMB Server                       │     │
│  │  POSIX 语义保障: 文件锁、原子操作                 │     │
│  └────────────────────┬──────────────────────────┘     │
│                       │                                 │
│  ┌────────────────────▼──────────────────────────┐     │
│  │              元数据服务 (MDS)                    │     │
│  │  ┌───────────────────────────────────────┐     │     │
│  │  │ 目录树、Inode、权限、锁                 │     │     │
│  │  │ 分布式元数据: 子树分区 + 动态迁移         │     │     │
│  │  │ Paxos 组内强一致                        │     │     │
│  │  └───────────────────────────────────────┘     │     │
│  └────────────────────┬──────────────────────────┘     │
│                       │                                 │
│  ┌────────────────────▼──────────────────────────┐     │
│  │              数据存储层 (Data Nodes)             │     │
│  │  条带化分布 + 多副本                             │     │
│  │  与 CBS/COS 共享底层存储引擎                     │     │
│  └───────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────┘
```

#### 元数据架构
- **归类:** 集中式分区（子树分割）+ 动态负载均衡
- 目录树按子树分区到不同 MDS 节点，支持热点目录自动迁移
- 文件锁（POSIX flock/fcntl）由 MDS 集中协调

#### 数据架构
- **分块:** 文件级条带化，支持大文件并行读写
- **分布:** 数据分布在存储集群，自动均衡
- **冗余:** 多副本（默认 3x）
- **I/O 路径:** NFS/SMB → 协议网关 → 元数据路由 → 数据节点（可能 bypass 元数据直接数据 I/O）

#### CAP / PACELC 定位
- **CAP: CP** — 文件存储需要强一致性（POSIX 语义），分区时牺牲可用性
- **PACELC: P→C / E→L** — 分区选一致性，正常时低延迟

## 二、数据库服务层

### 2.1 TDSQL（分布式关系型数据库）

#### 基本信息
- **类型:** 分布式 NewSQL（MySQL 兼容）

#### 元数据架构
- **归类:** 集中式管控 + 分布式 Paxos 副本组
- 分片路由信息（Schema → Set 映射）由管控层集中管理，缓存在 Proxy 层
- 每个 Set 内部使用 Paxos 保证副本间元数据/数据一致性

#### 数据架构
- **分片:** Set（分片组）为基本单位，每个 Set 包含 Master + 多个 Slave
- **分片策略:** Hash 分片（均匀分布）、Range 分片（范围查询优化）、List 分片（业务逻辑分片）
- **分布:** 应用通过 Proxy 自动路由到对应 Set
- **冗余:** Set 内 Paxos 多副本（默认 3 节点），跨 AZ 部署
- **事务:** 跨 Set 分布式事务使用 2PC/XA；单 Set 内事务由 MySQL InnoDB 保证

#### 核心技术创新
- **强同步 (Strong Sync):** 基于 Paxos 的半同步复制，确保数据不丢失
- **分布式事务:** 2PC + 优化（减少协调开销），支持跨分片 ACID
- **智能路由:** Proxy 层自动路由，对应用透明（无需修改 SQL）
- **在线 DDL:** 支持大规模表的在线 schema 变更
- **金融级特性:**
  - 两地三中心部署
  - 自动故障切换（秒级 RTO）
  - 数据强一致性保证
  - 审计日志、合规支持
- **HTAP 能力:** TDSQL 分析型版本（TDSQL-A，原 TBase）支持 OLAP 负载

#### CAP / PACELC 定位
- **CAP: CP** — 金融级数据库选择强一致性，分区时暂停写入保证数据正确
- **PACELC: P→C / E→C** — 分区时选一致性（C），正常时也选一致性（C）优先于延迟（E）
- **这是明确的 CP 系统** — 银行核心场景不允许任何数据不一致

#### 性能指标
- 单表支持: 千亿级记录
- 分片数: 水平扩展至数千分片
- 事务延迟: 同城亚毫秒级，跨 AZ 毫秒级
- 可用性: 99.999%（金融版）

#### 典型客户
- 微众银行（核心交易系统）
- 多家城商行、农商行核心系统
- 张家港农村商业银行（国内首家核心系统上云的银行）

#### 核心技术创新
- **CHDFS (Cloud HDFS):** HDFS 接口直接读写 COS，实现存算分离
- **存算分离架构:** 计算集群可独立扩展，存储使用 COS 无限扩展
- **多引擎统一数据源:** Spark/Hive/Presto/Flink 共享同一数据层
- **数据湖集成:** 支持 Delta Lake、Apache Hudi 等湖格式

#### CAP / PACELC
- **CAP: AP** — HDFS/COS 存储层分区时优先可用
- **PACELC: P→A / E→L** — 批量计算场景容忍延迟换取可用性

## 七、架构选型逻辑：为什么这样设计？

### 7.1 游戏社交场景驱动

腾讯的技术栈深深受到其核心业务的影响：

**游戏场景（王者荣耀、和平精英等）:**
- 极低延迟要求（P99 < 5ms）→ TcaplusDB、CKV
- 高并发写入（万人团战）→ 内存优先 + 异步持久化
- 不停服运维 → TcaplusDB 在线扩缩容
- 热点 Key 问题 → 自动热点分流

**社交场景（微信 13 亿+ 月活）:**
- 海量小文件 → COS 小对象优化
- 读写分离 → COS CDN 边缘缓存
- 消息海量 → CKafka/CMQ 高吞吐
- 数据合规 → COS 归档/深度归档

**金融场景（微信支付、微众银行）:**
- 数据零丢失 → TDSQL Paxos 强同步
- 两地三中心 → 跨 AZ/跨地域容灾
- ACID 事务 → 分布式 2PC
- 审计合规 → 完整审计链

### 7.2 底层统一存储引擎

腾讯云 CBS、COS、CFS 共享自研分布式存储底层，这带来了：

- **统一冗余策略:** 副本/纠删码在同一层实现
- **统一运维:** 存储池统一管理、自动均衡
- **数据流动:** CBS 快照 → COS 归档，CFS 热数据 → COS 冷存储
- **成本优化:** 根据访问频率自动分层

### 7.3 存算分离趋势

从 TDSQL-C 到 CHDFS，腾讯云正在推动存算分离架构：

| 系统 | 传统模式 | 存算分离模式 | 收益 |
|------|---------|-------------|------|
| TDSQL-C | 计算+存储耦合 | 共享分布式存储 | 秒级弹性 |
| CHDFS | HDFS 本地存储 | HDFS on COS | 存储无限扩展 |
| ClickHouse | 本地 MergeTree | COS 外表 | 存算独立 |

## 十、总结与趋势

### 当前架构特点

1. **场景驱动设计:** 游戏、社交、金融三大场景决定了腾讯云技术栈的设计优先级
2. **CP 优先的金融基因:** TDSQL 的 Paxos 强同步设计源于金融核心场景的严格要求
3. **底层统一存储:** CBS/COS/CFS 共享自研分布式存储引擎，降低运维复杂度
4. **存算分离演进:** TDSQL-C、CHDFS、ClickHouse on COS 标志着全面转向存算分离

### 未来趋势

1. **AI 训练存储优化:** 大模型训练推动存储层向 RDMA + NVMe + 高吞吐方向演进
2. **Serverless 数据库:** TDSQL-C 的 Serverless 化是云原生数据库的下一步
3. **湖仓一体:** EMR + COS + 湖格式（Delta/Hudi/Iceberg）构建统一数据平台
4. **多活架构:** 两地三中心 → 三地五中心 → 全球多活
5. **绿色计算:** 降低 PUE、智能调度、冷数据归档深度优化
