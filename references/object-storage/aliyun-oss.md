## 1. Context & Motivation（背景与动机）

### 解决的问题

在 2010 年前后，阿里巴巴面临几个关键挑战：

1. **电商爆发式增长**：淘宝/天猫平台上的海量商品图片、视频、日志文件需要可靠、低成本的持久化存储。传统 NAS/SAN 方案在容量和成本上无法满足 EB 级需求。
2. **云化战略**：阿里云从 2009 年开始构建，需要一个类似 Amazon S3 的云原生对象存储服务，作为 IaaS 层的核心组件之一。
3. **多业务统一存储**：阿里巴巴内部存在大量异构存储系统（日志、图片、备份、归档），缺乏统一入口和标准 API。

### 前代系统的局限性

| 前代系统 | 局限性 |
|----------|--------|
| 本地文件系统/NAS | 无法水平扩展，单点容量有限，运维成本高 |
| HDFS | 面向 MapReduce 批量处理设计，不适合海量小文件和高并发随机读写 |
| 传统 SAN | 成本极高，无法支持云化的按需付费模型 |
| Ceph（同期） | 开源生态早期，CRUSH 算法在小文件和高 QPS 场景下存在性能瓶颈 |

### 硬件/网络背景

- **磁盘**：2010 年 SATA 磁盘成为主流，成本大幅下降（$/GB 持续降低），使大规模廉价存储成为可能
- **网络**：万兆以太网 (10GbE) 在数据中心内部普及，为分布式存储提供足够带宽
- **x86 集群**：廉价 x86 服务器集群替代专用存储阵列，软件定义存储成为趋势
- **中国 IDC 环境**：多地域、多可用区的网络基础设施逐步成熟，为跨区域部署提供基础

### 数据模型与命名空间

OSS 采用 **Bucket + Object** 两层模型，与 Amazon S3 高度一致：

```
oss://<bucket-name>.<endpoint>/<object-key>

示例:
oss://my-bucket.oss-cn-hangzhou.aliyuncs.com/photos/2024/image.jpg
```

| 概念 | 说明 |
|------|------|
| **Bucket** | 顶级命名容器，全局唯一名称（全球级别），归属特定 Region |
| **Object** | 存储实体，由 Key（路径）+ Data（内容）+ Metadata（元数据）组成 |
| **Key** | UTF-8 字符串，最大 1023 字节，支持 `/` 模拟目录结构 |
| **Object Size** | 0 ~ 48.8 TB（单对象上限） |
| **Part Size** | 分片上传：100 KB ~ 5 GB |
| **Metadata** | 系统元数据（Content-Type 等）+ 用户自定义元数据（x-oss-meta-*） |

### 元数据架构归类：**集中式分区 (Concentrated/Partitioned)**

OSS 的元数据管理采用女娲 (Nvwa) 分布式协调服务：

- **Paxos 协议**保证元数据操作的一致性
- **分片 (Sharding)**：Bucket 级别元数据按哈希分区到多个 Nvwa Master
- **高可用**：每个元数据分片通过 Paxos 多副本保证容错
- 这属于「集中式分区」模式：每个分区内部集中管理，但元数据整体是分区化的

### 数据架构分析

| 维度 | 设计 |
|------|------|
| **分块方式** | 对象按固定大小分块（Chunk），类似 GFS 但块更小（盘古默认 ~64MB~256MB Chunk） |
| **分布策略** | Chunk 通过盘古的分布式调度算法分配到多个 ChunkServer，支持自动再平衡 |
| **冗余方式** | **纠删码 (Erasure Coding)** 为主，同可用区内 4+2 或 8+2 编码；跨可用区采用多副本 + 纠删码混合策略 |
| **I/O 路径** | 客户端 REST API → 协议网关 → 元数据层 → 数据层 → 磁盘 |
| **生命周期管理** | 自动分层：Standard → IA → Archive → Cold Archive，基于规则引擎按时间/前缀/标签触发 |

### 存储类别

| 存储类别 | 适用场景 | 可用性 SLA | 最低存储期限 | 取回费用 | 首字节延迟 |

| **标准 (Standard)** | 热数据，频繁访问 | 99.995% | 无 | 无 | ms 级 |
| **低频 (IA)** | 较少访问（每月几次） | 99.9% | 30 天 | 按量收取 | ms 级 |

| **深度冷归档 (Deep Cold Archive)** | 极长期归档 | 99.9% | 180 天 | 按量收取 | 小时级（需解冻） |

## 3. Core Technical Innovations（核心技术创新）

### 一致性模型

| 操作类型 | 一致性保证 | 说明 |
|----------|-----------|------|
| **同一 Region 内的读/写** | **强一致性 (Read-After-Write)** | 2021 年起全面支持，覆盖新对象写入、覆盖写入、删除后读取 |
| **跨区域复制 (CRR)** | **最终一致性** | 异步复制，存在秒级延迟 |
| **多版本控制** | **强一致性** | 版本索引强一致，各版本内容独立 |

> **历史背景**：2021 年之前，OSS 对部分场景（如覆盖写后读）存在最终一致性窗口。2021 年公告全面升级为强一致性模型，对齐 S3 的 2020 年一致性升级。

### 纠删码与持久性

- **持久性指标**：**99.999999999%（11 个 9）**
- **纠删码方案**：盘古底层采用灵活的 EC 策略
  - 同 AZ 内：典型配置为 `4+2`（4 个数据块 + 2 个校验块）或 `8+2`
  - 跨 AZ 场景：多副本与纠删码混合
- **容忍度**：`k+m` 配置下可同时容忍最多 `m` 个 ChunkServer 或磁盘故障
- **自动修复**：数据损坏时自动从剩余块中重建，后台后台再平衡

### 容错机制

| 机制 | 说明 |
|------|------|
| **盘古多副本/EC 冗余** | 数据级容错，磁盘/服务器故障不丢数据 |
| **女娲 Paxos 元数据** | 元数据级容错，多数派存活即可服务 |
| **自动故障转移** | ChunkServer 故障后自动重新分配数据到其他节点 |
| **数据完整性校验** | 全链路 CRC/Checksum 校验，端到端数据一致性验证 |
| **跨区域容灾** | CRR 提供地域级灾备能力 |

### 性能优化技术

| 技术 | 说明 |
|------|------|
| **分片上传 (Multipart Upload)** | 大对象并行上传，支持断点续传，最高提升数十倍吞吐 |
| **CDN 集成** | 全球 2800+ 节点就近缓存，回源自动处理 |
| **传输加速** | 利用阿里云全球骨干网优化跨地域传输 |
| **图片/视频处理** | 服务端即时处理（裁剪、水印、转码），避免下载→处理→上传的往返 |
| **智能分层 (Intelligent Tiering)** | 自动检测访问模式，在 Standard 和 IA 之间迁移 |
| **Select 查询** | 直接对 OSS 中的 CSV/JSON/Parquet 执行 SQL 查询，减少数据传输量 |

### 扩展方式

- **Scale-out**：全分布式架构，无单点瓶颈
- **Bucket 级隔离**：每个 Bucket 独立资源池，互不影响
- **弹性容量**：用户无需预分配容量，按需使用
- **QPS 上限**：单 Bucket 默认 10,000 QPS（可通过工单提升）

## 4. Trade-offs (CAP / PACELC)

### CAP 定位

```
        Consistency
            ▲
            │
            │      ● OSS 元数据 (Bucket/Object 索引)
            │      (强一致, CP)
            │
            │
            │           ● OSS 数据 (同 Region)
            │           (强一致读后写, CP)
            │
            │                    ● OSS 跨区域复制
            │                    (最终一致, AP)
            │
            └─────────────────────────────► Availability
```

| 组件 | CAP 选择 | 说明 |
|------|----------|------|
| **元数据服务 (Nvwa)** | **CP** | Paxos 保证元数据强一致，分区时牺牲可用性（少数派不可写） |
| **同 Region 数据读写** | **CP** | 2021 年后统一为强一致性模型 |
| **跨区域复制 (CRR)** | **AP** | 异步复制，网络分区时不阻塞本地写入 |
| **CDN 边缘缓存** | **AP** | 边缘节点最终一致，源站为准 |

### PACELC 定位

| 场景 | 选择 | 说明 |
|------|------|------|
| **分区时 (P)** | **C** | 同 Region 内优先保证一致性，元数据层 Paxos 多数派原则 |
| **正常时 (E)** | **L** | 无分区时优化延迟，读操作直接返回，无需跨节点协调（本地强一致） |
| **跨区域复制** | **A (分区时) / L (正常时)** | CRR 异步模式，分区时不影响本地服务 |

> **PACELC 总结**：同 Region 场景属于 **PC/EL** — 分区时选 C（一致性），正常时选 L（低延迟）。跨区域复制属于 **PA/EL** — 分区时选 A（可用性），正常时选 L（低延迟）。

### 复杂度 vs 运维

| 维度 | 权衡 |
|------|------|
| **对用户的复杂度** | 极低 — REST API，零运维，按需付费 |
| **内部复杂度** | 极高 — 盘古 + 女娲 + 网关 + CDN + 增值服务的多层系统 |
| **运维模式** | 全托管 — 用户不接触底层基础设施 |
| **API 兼容性** | 高度 S3 兼容 — 降低迁移成本，但非 100% 兼容 |

### 成本 vs 性能

| 维度 | 权衡 |
|------|------|
| **存储成本** | 分层定价：Standard (最高) → IA → Archive → Cold Archive (最低)，可降一个数量级 |
| **取回成本** | 冷数据取回费用递增，引导用户合理分层 |
| **流量成本** | 同 Region 内网免费，跨 Region/公网收费 |
| **EC vs 副本** | 纠删码存储效率更高（~66%~80% 利用率 vs 3 副本的 33%），但写入/修复计算开销更大 |
| **CDN 加速** | 额外费用，但大幅降低回源延迟和流量成本 |

### 成为行业标准的设计模式

1. **S3 兼容 API**：OSS 的 API 设计与 S3 高度兼容，成为中国云对象存储的事实标准接口
2. **存储分层模型**：Standard → IA → Archive 的三级（后扩展为五级）分层被国内所有云厂商采纳
3. **CDN + 对象存储深度集成**：OSS 与阿里云 CDN 的无缝集成模式成为行业标配
4. **服务端数据处理**：图片处理、视频转码等内置增值服务，避免了数据搬运
5. **生命周期规则引擎**：基于前缀/标签/时间的自动化数据迁移策略

### 与 Amazon S3 的对比总结

| 维度 | Amazon S3 | 阿里云 OSS |
