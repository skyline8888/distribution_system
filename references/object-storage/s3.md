# Amazon S3 — Simple Storage Service

## 基本信息

| 维度 | 内容 |
|------|------|
| **发布时间** | 2006 年 3 月 14 日 |
| **开发方** | Amazon Web Services (AWS) |
| **类型** | 商用托管、闭源 |
| **核心论文** | 无独立论文（设计通过 AWS 文档/博客/演讲披露） |
| **定位** | 云原生对象存储——定义了"对象存储"这一品类本身 |

---

## 1. Context & Motivation（背景与动机）

### 1.1 解决了什么问题？

2006 年前，企业数据存储主要依赖三类方案：

| 方案 | 痛点 |
|------|------|
| 本地 SAN/NAS | 容量固定、扩容慢、资本支出高、运维复杂 |
| 自建 HTTP 文件服务 | 需要自己处理冗余、分发、CDN 集成、计费 |
| CDN 分发网络 | 只能做缓存，不能作为源存储，写入受限 |

Amazon S3 的目标是：**提供一个无限可扩展、按需付费、通过简单 HTTP API 访问的持久化存储服务**。用户无需关心硬件采购、容量规划、数据冗余——上传即存，下载即取。

### 1.2 前代系统的局限性

- **GFS (2003)**：面向 Google 内部 MapReduce 批处理场景，不对外暴露，API 非 HTTP，不具备多租户和按需计费能力。
- **传统块存储 (EBS/SAN)**：块存储需要文件系统层，复杂度高，不适合海量小对象或静态资源分发。
- **FTP/HTTP 文件服务器**：无内置冗余、无多租户隔离、无编程接口。

### 1.3 硬件与网络趋势

- **廉价 x86 硬件 + 商品 HDD**：Amazon 已在电商和 AWS 基础设施中积累了大规模运营经验。
- **HTTP/REST 成熟**：RESTful API 成为 Web 服务标准，使得通过 HTTP PUT/GET/DELETE 操作存储成为自然选择。
- **互联网带宽提升**：宽带普及使得通过互联网上传 GB 级数据成为可行方案。

## 2. Architecture（架构设计）

### 2.1 系统拓扑

S3 是**高度分布式、多 AZ（Availability Zone）部署**的托管服务。用户不部署 S3——AWS 运营它。

```
┌─────────────────────────────────────────────────────────────────┐
│                        Amazon S3 (Regional)                     │
│                                                                 │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐               │
│  │   AZ-1    │    │   AZ-2    │    │   AZ-3    │    ← 至少 3 AZ│
│  │           │    │           │    │           │               │
│  │ ┌───────┐ │    │ ┌───────┐ │    │ ┌───────┐ │               │
│  │ │Storage│ │    │ │Storage│ │    │ │Storage│ │               │
│  │ │Nodes  │ │    │ │Nodes  │ │    │ │Nodes  │ │               │
│  │ └───────┘ │    │ └───────┘ │    │ └───────┘ │               │
│  │ ┌───────┐ │    │ ┌───────┐ │    │ ┌───────┐ │               │
│  │ │Index  │ │    │ │Index  │ │    │ │Index  │ │               │
│  │ │Layer  │ │    │ │Layer  │ │    │ │Index  │ │               │
│  │ └───────┘ │    │ └───────┘ │    │ └───────┘ │               │
```

### 2.2 数据模型与命名空间

```
┌──────────────────────────────────────────────┐
│  Account                                      │
│  ┌────────────────────────────────────────┐   │
│  │  Bucket (全局唯一名称, DNS 可解析)       │   │
│  │  ┌──────────────────────────────────┐  │   │
│  │  │  Object                           │  │   │
│  │  │  ├── Key (UTF-8 字符串, 最大 1024B)│  │   │
│  │  │  ├── Value (数据, 最大 5TB)        │  │   │
│  │  │  ├── Metadata (系统 + 自定义 KV)    │  │   │
│  │  │  ├── ETag (内容 MD5 / 哈希)        │  │   │
│  │  │  ├── Version ID (版本化时)         │  │   │
│  │  │  └── Storage Class (存储层级)       │  │   │
│  │  └──────────────────────────────────┘  │   │
│  │  ┌──────────────────────────────────┐  │   │
│  │  │  Object (key="photos/cat.jpg")    │  │   │
│  │  └──────────────────────────────────┘  │   │
│  └────────────────────────────────────────┘   │
│  ┌────────────────────────────────────────┐   │
│  │  Bucket (my-backups)                    │   │
│  │  ...                                    │   │
│  └────────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

**核心设计选择**：

| 设计 | 说明 |
|------|------|
| **扁平命名空间** | 不存在真正的"目录"。`photos/cat.jpg` 是一个 Key，不是路径。`/` 分隔符仅为 UI 约定。 |
| **桶全局唯一** | 桶名在所有 AWS 账户间唯一，可作为 DNS 子域名 (`bucket.s3.amazonaws.com`)。 |
| **不可变对象** | 对象一旦写入即不可修改。更新 = 写入新对象覆盖旧对象（版本化时可保留历史）。 |
| **单对象最大 5TB** | 通过 Multipart Upload 分块上传（每块最小 5MB，最大 5GB）。 |

### 2.3 元数据架构归类：**集中式分区（AWS 内部管理）**

S3 的元数据架构对外不可见，但可归类为 **集中式分区** 模式：

- **桶级元数据**（名称、策略、ACL、区域）由区域级控制平面集中管理。
- **对象级元数据**（Key、ETag、版本、存储类、大小）由 S3 内部索引层分布管理，通过 Key 哈希映射到具体存储节点。
- 用户无需关心元数据分布——这是 S3 作为托管服务的核心优势，也是其黑盒特性。

### 2.4 数据架构分析

#### 数据分块 (Chunking)

- 单对象 ≤ 5GB：单 PUT 直接上传。
- 单对象 > 5GB 或需要并发：使用 **Multipart Upload**，将对象分为 5MB–5GB 的块独立上传，最后 Complete。
- 内部存储引擎将大对象进一步拆分为固定大小的段 (segment)，分布到多个存储节点。

#### 数据分布

- **桶级路由**：桶创建于特定 AWS 区域，桶内所有数据驻留在该区域。
- **对象级路由**：对象 Key 经过一致性哈希/分区映射到区域内的多个 AZ。
- 跨区域复制 (Cross-Region Replication) 是可选功能，非自动。

#### 数据冗余

- **多 AZ 自动复制**：标准存储类下，每个对象自动复制到区域内 **至少 3 个 AZ**。
- **耐久性设计目标**：99.999999999%（11 个 9），即年均对象丢失概率 ≤ 0.000000001%。
- **校验和**：写入时计算 ETag，读取时校验完整性。MD5SUM 验证可选。
- **位衰减保护 (Bit Rot Protection)**：后台扫描检测并修复静默数据损坏。

#### I/O 路径

```
Client ──HTTPS/REST──▶ S3 Endpoint (DNS)
                           │
                      Load Balancer
                           │
                    Request Router ──▶ 定位桶 & 对象所在 AZ
                           │
                    ┌────────┴────────┐
                    ▼                 ▼
               Index Layer      Storage Nodes
              (元数据查找)        (数据读写)
                    │                 │
                    └────────┬────────┘
                             ▼
                      返回数据 / 确认
```

- 全部通过 **HTTPS + REST API**（PUT/GET/HEAD/DELETE/LIST）。
- 无内核态组件暴露给用户——纯网络 I/O。

#### 生命周期管理

| 功能 | 说明 |
|------|------|
| **版本控制 (Versioning)** | 每个 PUT 保留一个版本，可恢复历史版本。支持 MFA Delete 防误删。 |
| **生命周期策略 (Lifecycle)** | 基于时间/前缀自动转换存储类或过期删除。 |
| **跨区域复制 (CRR)** | 桶级策略，异步将对象复制到目标区域桶。 |
| **对象锁定 (Object Lock)** | WORM（Write Once Read Many），满足合规要求（SEC 17a-4 等）。 |
| **事件通知** | S3 Event Notifications → SQS/SNS/Lambda，触发下游处理。 |

## 3. Core Technical Innovations（核心技术创新）

### 3.1 一致性模型演进

| 阶段 | 时间 | 行为 |
|------|------|------|
| **最终一致** | 2006–2020 | PUT 后 LIST 可能不立即返回新对象；DELETE 后 GET 可能仍返回旧对象（典型延迟 < 1s）。 |
| **强一致** | 2020.12 | 全面改为**强一致性**（strongly consistent）。PUT 后立即可读；DELETE 后立即不可读。无性能惩罚。 |

**2020 年强一致化**是 S3 历史上最重要的架构变更之一。此前，跨 AZ 复制导致的异步传播使 S3 属于**最终一致**系统。AWS 通过重构内部复制管道，在保持性能和耐久性不变的前提下实现了强一致。

### 3.2 容错机制

| 机制 | 实现 |
|------|------|
| **多 AZ 部署** | 至少 3 个物理隔离的 AZ，每个 AZ 独立电源/网络。 |
| **自动修复** | 后台进程持续扫描副本一致性，检测到不一致/损坏后自动重建。 |
| **跨区域容灾** | CRR + S3 Replication Time Control (RTC) 提供分钟级 RPO。 |
| **DDoS 防护** | AWS Shield 集成，自动缓解大规模 DDoS 攻击。 |

### 3.3 存储层级 (Storage Classes)

S3 通过存储层级实现**数据生命周期成本优化**，这是对象存储区别于块/文件存储的关键创新：

| 存储类 | 适用场景 | 可用性 SLA | 最低存储期 | 特点 |
|--------|----------|-----------|-----------|------|
| **S3 Standard** | 热数据、频繁访问 | 99.99% | 无 | 多 AZ，ms 级延迟 |
| **S3 Intelligent-Tiering** | 访问模式未知/变化 | 99.9% | 30 天 | ML 自动分层，无需策略 |
| **S3 Standard-IA** | 低频访问 | 99.9% | 30 天 | 低存储费 + 取回费 |
| **S3 One Zone-IA** | 可重建数据 | 99.5% | 30 天 | 单 AZ，成本更低 |
| **S3 Glacier Instant Retrieval** | 归档，需即时访问 | 99.9% | 90 天 | ms 级取回 |
| **S3 Glacier Flexible** | 长期归档 | 99.99% | 90 天 | 分钟–小时取回 |
| **S3 Glacier Deep Archive** | 合规/长期保留 | 99.99% | 180 天 | 12 小时取回，最低成本 |

**Intelligent-Tiering** 是 S3 最具创新性的层级之一：内置 ML 算法持续分析访问模式，自动在 Standard ↔ IA ↔ Archive 间迁移，无需用户编写生命周期规则。

### 3.4 S3 Select / Glacier Select

传统模式：下载整个对象 → 客户端过滤。
S3 Select：在服务端用 SQL 过滤对象内容，仅返回匹配行/列。

```sql
SELECT s._1, s._3 FROM S3Object s
WHERE CAST(s._4 AS INT) > 100
```

- 支持 CSV、JSON、Parquet 格式。
- 可减少 60–90% 的网络传输和计算开销。
- 是**"计算靠近数据"**理念的体现，与 Athena/Redshift Spectrum 一脉相承。

### 3.5 扩展方式

S3 采用 **Scale-Out** 架构：

- 桶数量和对象数量**无上限**（实践中已证明单桶可存数十亿对象）。
- 请求速率：S3 自动扩展，无需预分区。前缀设计优化（已不再必要，但曾推荐在 Key 前加哈希前缀以分散负载）。
- 全球传输加速 (S3 Transfer Acceleration) 利用 CloudFront 边缘节点加速上传。

## 4. Trade-offs (CAP / PACELC)

### 4.1 CAP 三角定位

```
        Consistency (C)
              ▲
             / \
            /   \
           /  S3  \    ← 2020 年后：CP
          / (当前)  \
         /           \
        /             \
       ◄───────►       \
  Partition (P)       Availability (A)

  2006–2020: CA (正常时), CP (分区时) — 最终一致
  2020–至今: CP — 强一致读后写
```

| 阶段 | CAP 定位 | 说明 |
|------|----------|------|
| **2006–2020** | **CA / 弱 C** | 正常时高可用；分区时通过最终一致性维持服务，但读可能返回旧数据。 |
| **2020–至今** | **CP** | 强一致性优先。分区时部分请求可能延迟或不可用以保证一致性。实际中 AWS 通过多 AZ 冗余使分区极罕见。 |

### 4.2 PACELC 定位

```
PACELC: Partition 时选 A 还是 C?  Else(正常)时选 L 还是 E?

  2006–2020: PA/EL  — 分区时选 A(可用), 正常时选 L(低延迟, 最终一致)
  2020–至今: PC/EC  — 分区时选 C(一致), 正常时选 C(一致, 强一致读后写)
```

| 阶段 | 分区时 | 正常时 | 说明 |
|------|--------|--------|------|
| **2006–2020** | **PA** — 选可用性 | **EL** — 选低延迟（异步复制） | 适合 Web 静态资源、备份等容忍最终一致的场景。 |
| **2020–至今** | **PC** — 选一致性 | **EC** — 选一致性（同步复制路径） | 强一致代价极低——AWS 优化后几乎无感知延迟增加。 |

### 4.3 核心权衡总结

| 权衡 | 选择 | 代价 |
|------|------|------|
| **一致性 vs 可用性** | 当前选 C（强一致） | 分区时部分请求延迟增加，但 AWS 多 AZ 架构使分区极罕见 |
| **延迟 vs 一致性** | 当前选 C（一致性） | 强一致引入的延迟增量 < ms 级，对绝大多数场景不可感知 |
| **复杂度 vs 运维** | AWS 托管，用户零运维 | 用户无法调优内部参数，无法自行部署，供应商锁定 |
| **成本 vs 性能** | 分层存储提供精细选择 | 频繁跨层取回会产生额外费用；Deep Archive 取回成本高 |
| **灵活性 vs 不可变性** | 对象不可变 | 不支持随机写/追加，不适合数据库等场景 |

## 5. Influence & Legacy（影响与遗产）

### 5.1 S3 API 成为事实标准

S3 最伟大的遗产不是存储引擎本身，而是它的 **REST API 规范**：

```
PUT    /{bucket}/{key}     — 上传对象
GET    /{bucket}/{key}     — 下载对象
HEAD   /{bucket}/{key}     — 获取元数据
DELETE /{bucket}/{key}     — 删除对象
GET    /{bucket}?list-type=2  — 列举对象
PUT    /{bucket}            — 创建桶
DELETE /{bucket}            — 删除桶
```

**几乎所有**现代对象存储系统都提供 S3 兼容 API：

| 系统 | S3 兼容程度 |
|------|------------|
| MinIO | 原生 S3 兼容（设计目标） |
| Ceph RGW | S3 + Swift 双协议 |
| OpenStack Swift | 通过 swift3 中间件 |
| 阿里云 OSS | S3 兼容 API |
| 腾讯云 COS | S3 兼容 API |
| Google Cloud Storage | 部分 S3 兼容 |
| Cloudflare R2 | 完全 S3 兼容 |
| Backblaze B2 | S3 兼容 |
| Wasabi | S3 兼容 |
| SeaweedFS | S3 兼容网关 |
| JuiceFS | S3 作为后端 |
| Apache Ozone | S3 网关 |

这意味着：**S3 API = 对象存储的 POSIX**。

### 5.2 启发的设计模式

| 设计模式 | 来源 | 影响 |
|----------|------|------|
| 桶 + 对象模型 | S3 | 所有对象存储的基础抽象 |
| 不可变对象 | S3 | WORM、版本控制、数据湖不可变层 |
| 存储层级 | S3 | 冷热分层成为对象存储标配 |
| 生命周期策略 | S3 | 数据自动归档/过期 |
| Presigned URL | S3 | 临时凭证访问模式 |
| Multipart Upload | S3 | 大文件分块上传标准 |
| 事件驱动通知 | S3 → Lambda | 云原生事件处理范式 |

### 5.4 当前相关性与使用场景

| 场景 | 说明 |
|------|------|
| **数据湖/湖仓一体底座** | S3 是数据湖事实上的默认存储层，配合 Iceberg/Hudi/Delta 提供 ACID 语义。 |
| **Web 静态资源托管** | S3 + CloudFront 是全球最常见的静态网站架构。 |
| **备份与归档** | Glacier 系列提供最低成本的长期保留方案。 |
| **AI/ML 训练数据** | 大规模 ML 训练数据集存储在 S3 上，通过 S3 Select / Athena 预处理。 |
| **应用数据存储** | 日志、媒体文件、用户生成内容、导出文件等半结构化/非结构化数据。 |
| **合规归档** | Object Lock + Glacier Deep Archive 满足 SEC/FINRA 等合规要求。 |
