# 分布式对象存储演进总览

## 什么是对象存储

对象存储 = 扁平命名空间 + HTTP/REST API + 桶（Bucket）+ 对象（Object）

与传统文件系统对比：
| 维度 | 文件系统 | 对象存储 |
|------|----------|----------|
| 命名空间 | 层次目录树 | 扁平桶 + 对象 Key |
| 访问协议 | POSIX/NFS/SMB | HTTP/REST (S3 API) |
| 修改方式 | 随机读写 | 整体覆盖（不可变对象） |
| 元数据 | 丰富（权限/时间戳/属性） | 有限（自定义 KV 标签） |
| 一致性 | 强一致 | 最终一致（早期）→ 强一致（现代） |
| 扩展性 | 受限于元数据 | 几乎无限 |

## 演进脉络

```
Amazon S3 (2006) → Ceph RGW (2010) → OpenStack Swift (2010)
→ MinIO (2015) → SeaweedFS 对象模式 (2015) → Cloudflare R2 (2022)
```

## 核心驱动力

1. **云计算崛起**：按需付费、无限扩展的存储需求
2. **数据湖底座**：对象存储成为数据湖/湖仓一体默认存储层
3. **S3 成为事实标准**：几乎所有对象存储都提供 S3 兼容 API
4. **成本优势**：相比块存储/文件存储，对象存储成本最低

## 技术演进主线

### 一致性
- 最终一致 (S3 早期) → 强一致读后写 (S3 2020, MinIO)

### 数据保护
- 多副本 → 纠删码 (Erasure Coding) → 比特旋转保护 (Bit Rot Protection)

### 性能优化
- 单线程 HTTP → 多连接并发 → RDMA 加速 → NVMe 后端

### 多租户
- 简单隔离 → IAM 策略 → 细粒度 ACL → 存储桶策略

## 时代划分

### 第一代：对象存储概念确立 (2006-2010)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| Amazon S3 | 2006 | 定义对象存储标准：桶+对象、REST API、无限扩展 |

### 第二代：开源对象存储 (2010-2015)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| Ceph RGW | 2010 | RADOS 上层、S3/Swift 双协议、CRUSH 数据分布 |
| OpenStack Swift | 2010 | Ring 架构、Account/Container/Object 三层、多租户 |

### 第三代：高性能对象存储 (2015-2022)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| MinIO | 2015 | 高性能纠删码、SIMD 优化、强一致、S3 兼容标杆 |
| SeaweedFS | 2015 | Volume Server 架构、O(1) 对象定位 |

### 第四代：云原生对象存储 (2022+)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| Cloudflare R2 | 2022 | 零出口流量费、边缘加速 |
| Backblaze B2 | 持续 | 低成本、S3 兼容 |

## 核心技术对比

| 维度 | S3 | Ceph RGW | MinIO | Swift |
|------|-----|----------|-------|-------|
| 一致性 | 强一致(现) | 强一致 | 强一致 | 最终一致 |
| 数据保护 | 多 AZ 副本 | CRUSH+EC | 纠删码 | 副本 |
| 扩展性 | 无限 | 数千 OSD | 线性 | 数百节点 |
| S3 兼容 | 原生 | 高 | 原生 | 部分 |
| 多租户 | IAM | RGW Realm | 内置 | Account |
| 部署 | 托管 | 复杂 | 简单 | 中等 |
| 典型延迟 | ms 级 | ms 级 | 亚 ms | ms 级 |

## 对象存储与数据湖的关系

对象存储是现代数据湖的默认存储层：
```
计算引擎 (Spark/Presto/Trino)
         ↕
   表格式 (Iceberg/Hudi/Delta)
         ↕
   对象存储 (S3/MinIO)
         ↕
   物理存储 (HDD/SSD)
```
