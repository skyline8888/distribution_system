## 基本信息

| 维度 | 内容 |
|------|------|

| **核心论文** | 无独立公开论文；设计通过火山引擎官方文档、架构白皮书、技术大会演讲披露 |

## 1. Context & Motivation（背景与动机）

### 1.1 解决的问题

TOS 的诞生由两条并行的驱动线推动：

**内部驱动：抖音/TikTok 海量内容存储需求**

- **短视频爆发**：抖音/TikTok 用户每天上传数千万条短视频，每条视频需要存储原始素材、转码后的多分辨率版本、封面图、审核素材等，对象数量以百亿/千亿计。
- **热/温/冷分层**：用户最近观看的内容需极低延迟（Standard），历史内容可归档（Archive），分层策略必须自动执行且对业务透明。
- **全球化多 Region**：TikTok 覆盖 150+ 国家/地区，需要多 Region 部署 + 跨区域复制，满足数据本地化合规与就近访问。
- **CDN 深度集成**：短视频内容天然需要全球 CDN 分发，对象存储与 CDN 的联动（回源加速、预热、边缘缓存刷新）是核心链路。

**外部驱动：火山引擎公有云商业化**

- 2020 年字节跳动成立火山引擎，将内部技术能力对外输出，需要一个对标 AWS S3、阿里云 OSS、腾讯云 COS 的对象存储产品。
- 企业客户希望迁移存量数据到火山引擎云，S3 兼容 API 是最低迁移成本的入口。

### 1.2 前代系统的局限性

| 前代方案 | 痛点 |
|----------|------|
| 自建 HDFS / Ceph | 运维复杂度高，无法支撑抖音级别的 QPS 和容量扩展 |
| 使用公有云竞品 | 数据在第三方云平台，延迟不可控、成本不透明、与字节内部生态割裂 |
| 早期内部存储方案 | 缺乏标准化 API、分层管理不完善、跨 Region 复制能力弱 |

### 1.3 硬件与网络趋势

- **NVMe SSD + 大容量 HDD 混合**：热数据走 NVMe 层，温/冷数据下探到大容量 HDD，分层粒度自动化。
- **25/100 Gbps RDMA 网络**：字节内部数据中心网络已全面升级到 RDMA (RoCE v2)，跨 AZ 复制和数据重平衡的延迟与带宽不再成为瓶颈。
- **纠删码大规模落地**：相比 3 副本，纠删码 (如 4+2、8+4) 显著降低存储成本，11 个 9 的持久化率成为行业标准。
- **边缘 CDN 节点爆发**：抖音/ TikTok 全球 CDN 节点数以万计，TOS 天然成为这些 CDN 的最佳源站。
- **GPU 集群训练需求**：AI 大模型训练需要从对象存储高速读取海量训练数据，TOS 需要支持高吞吐顺序读。

### 2.2 数据模型与命名空间

| 概念 | 描述 |
|------|------|
| **Bucket** | 顶层命名空间，全局唯一名称（Region 内），类似 S3 Bucket |
| **Object** | 存储单元，由 Key + Data + Metadata 组成，Key 为 UTF-8 字符串，最大 5TB |
| **Prefix** | 对象 Key 的前缀，用于模拟目录层次（扁平命名空间，无真实目录） |
| **Metadata** | 系统元数据（Content-Type、ETag、Last-Modified）+ 用户自定义元数据（x-tos-meta-*） |
| **Multipart Upload** | 大对象分片上传，每片 5MB~5GB，支持断点续传 |
| **Versioning** | 对象版本管理，同一 Key 可保留多个版本，支持版本级操作 |

### 2.3 元数据架构归类

**归类：原生分布式元数据服务（第 6 大类）**

TOS 元数据架构的核心特征：

- **分布式元数据集群**：元数据节点以 Raft Group 形式组织，每个 Group 管理一组 Bucket/前缀的元数据。
- **元数据分片**：按 Bucket + 前缀范围划分元数据分片（Shard），不同分片分配到不同 Raft Group，实现水平扩展。
- **热/冷元数据分层**：高频访问的元数据（Bucket 信息、热点对象）缓存在内存/SSD 层，低频元数据（归档对象索引）存储在 HDD 层。
- **Raft 共识**：每个 Raft Group 内通过 Raft 协议保证元数据一致性，跨 AZ 同步写入多数派确认后返回成功。

### 2.4 数据架构分析

**分块方式 (Chunking)**

- 对象上传后按固定大小切分为 Chunk（典型值 4MB~64MB，根据对象大小自适应）。
- Multipart Upload 的每个 Part 在内部对应一个或多个 Chunk。
- 追加写和覆盖写均以 Chunk 粒度执行，避免全对象重写。

**分布策略 (Distribution)**

- Chunk 通过一致性哈希 + 负载均衡算法分布到多个 AZ 的 Chunk Server。
- 跨 AZ 放置策略确保任意单 AZ 故障不影响数据可用性。
- 热点自动检测：高频访问的 Chunk 被主动复制/缓存到更多节点。

**冗余方式 (Redundancy)**

| 存储级别 | 冗余策略 | 说明 |
|----------|----------|------|
| **Standard** | 跨 AZ 多副本（3 副本）或纠删码（如 4+2） | 同一 Region 内跨 3 个 AZ 部署，9 个 9 以上持久化 |
| **Infrequent Access (IA)** | 纠删码（如 8+4） | 降低冗余度，适合低频访问，持久化率 11 个 9 |
| **Archive** | 纠删码 + 深度冷存储 | 最低成本，取回需等待分钟~小时级 |

- **Standard 级别**：同城多 AZ（3 AZ）部署，对象写入后通过同步复制在 3 个 AZ 各存一份，或通过纠删码（如 4+2）跨 AZ 分布。容忍任意单 AZ 故障。
- **IA 级别**：采用纠删码（典型 8+4），12 份数据+校验片分布在多个节点，允许最多 4 份同时丢失而不影响数据恢复。
- **Archive 级别**：纠删码 + 大容量 HDD 存储池，数据可能被离线/近线存储，取回延迟较高但成本最低。

**I/O 路径 (IO Path)**

- **热数据 (Standard/NVMe)**：客户端 → TOS Gateway → 路由到对应 Chunk Server → NVMe SSD 读取 → 返回。利用 RDMA (RoCE v2) 降低跨 AZ 传输延迟。
- **温数据 (IA/HDD)**：客户端 → TOS Gateway → 路由到 HDD Chunk Server → 读取。延迟略高，但吞吐足够。
- **冷数据 (Archive)**：客户端请求 → 触发取回任务 → 数据从冷存储迁移到热层 → 返回客户端。异步过程，有等待时间。

**生命周期管理 (Data Lifecycle)**

- **自动分层**：支持基于访问频率、时间的自动迁移策略（Standard → IA → Archive）。
- **生命周期规则**：可配置按前缀/标签的过期删除、存储级别转换策略。
- **合规保留**：支持 Object Lock（合规模式/治理模式），防止对象在保留期内被删除或修改。
- **版本清理**：旧版本对象可自动过期清理。

## 3. Core Technical Innovations（核心技术创新）

### 3.1 一致性模型

TOS 的一致性模型根据操作类型和场景有所不同：

| 操作 | 一致性级别 | 说明 |
|------|-----------|------|
| 同一 Region 内读/写新对象 | **强一致 (Read-after-Write)** | 写入成功后的读请求一定能读到最新数据 |
| 同一 Region 内覆盖写 | **强一致** | 写入成功后后续读请求读到新版本 |
| 跨 Region 复制 (CRR) | **最终一致** | 异步复制，目标 Region 有延迟（秒~分钟级） |
| 同 Region 内 List 操作 | **最终一致** | 大规模 List 可能有短暂延迟 |

### 3.2 共识协议

- **元数据 Raft**：每个元数据 Raft Group 运行 Raft 协议，保证元数据在 AZ 间的强一致。Leader 处理读写请求，Follower 同步日志。
- **数据路径**：数据写入时不依赖 Raft，而是通过主-从同步或直接写多副本（3 副本）或纠删码编码后分布写入。数据层的一致性由存储引擎 ByteStore 保证。

### 3.3 容错机制

| 容错层面 | 机制 |
|----------|------|
| **节点级故障** | Chunk Server 宕机后自动检测并触发数据重建，利用剩余副本/纠删码片恢复丢失数据 |
| **AZ 级故障** | Standard 级别跨 3 AZ 部署，任意单 AZ 宕机仍可读可写 |
| **网络分区** | Raft 多数派可继续服务；少数派 AZ 不可读不可写，但不会脑裂 |
| **磁盘故障** | 单盘故障不影响数据，自动从副本重建 |
| **Region 级故障** | 跨 Region 异步复制（CRR），手动或自动故障切换到目标 Region |
| **数据完整性** | 端到端校验和（上传时计算，存储时校验，读取时验证），定期后台巡检 |

### 3.4 性能优化技术

| 优化 | 描述 |
|------|------|
| **RDMA (RoCE v2)** | 字节内部数据中心网络已全面 RDMA 化，跨 AZ 数据传输绕过内核协议栈，延迟降低至微秒级 |
| **智能预热/热点检测** | 高频访问的 Chunk 自动预热到缓存层（SSD/NVMe），CDN 节点主动预热热门内容 |
| **Multipart Upload + 并发** | 客户端支持并发分片上传，服务端并行接收并编码 |
| **分层存储自动调度** | 基于访问模式的 AI 预测模型，自动将冷数据下移、热数据上移，降低总成本 |
| **纠删码加速** | 针对 CPU/GPU 优化的纠删码编码/解码库（如基于 ISA-L、SIMD 指令），降低计算开销 |
| **CDN 深度集成** | TOS 与字节全球 CDN 深度集成，支持 CDN 回源、预热、边缘缓存刷新，短视频内容分发延迟毫秒级 |
| **媒体处理 Pipeline** | 内置图片处理（缩放、裁剪、水印、格式转换）和视频处理（转码、截图、拼接），减少外部系统调用 |
| **大对象优化** | 针对 GB/TB 级大对象的流式分块处理，避免整对象加载到内存 |

### 3.5 扩展方式

- **Scale-out（水平扩展）**：通过增加 Chunk Server 和元数据节点实现容量和吞吐的线性扩展。
- **元数据分片**：按前缀范围自动分裂元数据 Shard，Raft Group 数量可动态增长。
- **跨 Region 扩展**：新增 Region 时自动同步复制策略，数据按需迁移。
- **多租户隔离**：Bucket 级别的 QoS 限流和配额管理，防止租户间相互影响。

## 4. Trade-offs (CAP / PACELC)

### 4.1 CAP 定位

```
        Consistency (C)
             ▲
            / \
           /   \
          /     \
         /   ●   \      ← TOS (同 Region 操作)
        /         \        选择 C + A（无分区时）
       /    ●      \     ← TOS (跨 Region 复制)
      /_____________\        选择 A + 最终一致性（分区时）
Availability (A)   Partition Tolerance (P)
```

| 场景 | CAP 选择 | 说明 |
|------|----------|------|
| 同 Region 正常操作 | **CA**（无分区）| 强一致 + 高可用，Raft 多数派保证 |
| 同 Region 网络分区 | **CP** | 少数派 AZ 不可用，保证一致性，牺牲可用性 |
| 跨 Region 复制 | **AP** | 异步复制，容忍网络延迟，最终一致性 |

**总结：TOS 在同 Region 内优先选择 CP（一致性 + 分区容错性），跨 Region 复制选择 AP（可用性 + 分区容错性 + 最终一致性）。**

### 4.2 PACELC 定位

| 条件 | 选择 | 说明 |
|------|------|------|
| **When Partition (P)** | **A** | 网络分区时优先保证可用性（多数派 AZ 继续服务） |
| **Else (no partition)** | **L**（部分场景）/ **E**（部分场景） | 同 Region 写操作选 E（低延迟，多数派写入即可返回）；跨 Region 读操作选 L（可能读到旧版本，避免跨 Region 延迟） |

更精确地说：

- **写入路径**：PACELC → **EL**（低延迟：多数派确认即返回，不等待全局同步）
- **同 Region 读取**：PACELC → **E**（强一致，读取本地最新数据）
- **跨 Region 读取**：PACELC → **L**（允许最终一致，避免跨 Region 延迟）

### 4.3 复杂度 vs 运维简单性

| 选择 | 权衡 |
|------|------|
| 多 Raft Group 管理 | 元数据运维复杂，但获得高可用和高扩展性 |
| 纠删码 vs 副本 | 纠删码降低存储成本（30%~50% 节省），但增加 CPU 编解码开销和故障恢复复杂度 |
| 分层存储自动化 | 用户无需手动管理数据生命周期，但系统内部调度策略复杂 |
| 闭源托管服务 | 用户无需运维底层集群，但无法自定制或审计内部实现 |

### 4.4 成本 vs 性能

| 存储级别 | 成本 | 性能 | 适用场景 |
|----------|------|------|----------|
| **Standard** | 高 | 毫秒级延迟，最高吞吐 | 热数据、短视频在线播放 |
| **IA** | 中 | 亚秒~秒级延迟 | 低频访问的历史内容、日志 |
| **Archive** | 低 | 分钟~小时级取回 | 合规归档、长期备份 |

- 纠删码比 3 副本节省 30%~50% 存储成本。
- RDMA 网络降低跨 AZ 传输成本（减少 CPU 开销和延迟）。
- CDN 缓存减少回源流量，降低出口带宽成本。

## 5. Influence & Legacy（影响与遗产）

### 5.1 业务影响力

- **抖音/TikTok 核心存储底座**：承载了抖音/ TikTok 全部短视频内容的持久化存储，是字节跳动最核心的基础设施之一。日增 EB 级数据量的处理能力证明了其规模和可靠性。
- **火山引擎对外服务**：作为火山引擎公有云的核心 IaaS 服务，TOS 为外部企业客户提供对象存储能力，与火山引擎其他服务（计算、大数据、AI）深度集成。

### 5.2 技术影响力

| 影响领域 | 具体表现 |
|----------|----------|
| **S3 兼容生态** | TOS 的 S3 兼容 API 使得基于 S3 的工具链（SDK、CLI、备份工具、数据湖框架）可以直接使用，降低了迁移门槛 |
| **CDN + 对象存储深度集成** | TOS 与字节全球 CDN 的联动模式（回源加速、边缘缓存、智能预热）成为行业参考架构 |
| **媒体处理流水线** | TOS 内置的图片/视频处理能力，为短视频行业提供了"存储+处理+分发"一体化解决方案 |
| **AI 训练数据湖** | TOS 作为 AI 大模型训练的数据湖底座，支持高吞吐顺序读，满足 GPU 集群的数据供给需求 |

### 5.3 技术谱系位置

```
Amazon S3 (2006)
    │
    ├── 定义对象存储标准 (Bucket + Object + S3 API)
    │       │
    │       ├── Aliyun OSS (2012) ── Pangu 底层
    │       │
    │       ├── Tencent COS (2014) ── CFS 底层
    │       │
    │       ├── MinIO (2014) ── 开源 S3 兼容
    │       │
    │       ├── Ceph RGW (~2012) ── RADOS 上层
    │       │
    │       └── ByteDance TOS (2020+) ── ByteStore 底层
    │               │
    │               ├── S3 兼容 API (低迁移成本)
    │               ├── 纠删码 + 多 AZ (持久化 11 个 9)
    │               ├── RDMA 内部网络 (微秒级延迟)
    │               ├── 分层存储 (Standard/IA/Archive)
    │               ├── CDN 深度集成 (短视频分发)
    │               ├── 媒体处理 Pipeline (图片/视频)
    │               └── 跨 Region 复制 (全球化合规)
```
