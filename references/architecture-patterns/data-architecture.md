## 1. 共享存储 (Shared Storage / SAN-NAS)

### 1.1 架构概述

共享存储是最传统的分布式数据存储方式：所有计算节点通过专用存储网络（SAN）或文件共享协议（NAS）访问同一组物理磁盘阵列。数据物理上集中存放，逻辑上被多个节点共享。

```
┌─────────────────────────────────────────────────────┐
│                    Application Layer                 │
├──────────┬──────────┬──────────┬─────────────────────┤
│  Node A  │  Node B  │  Node C  │  Node D  (Compute)  │
└────┬─────┴────┬─────┴────┬─────┴────┬────────────────┘
     │          │          │          │
     └──────────┴────┬─────┴──────────┘
              Storage Network (SAN / NAS)
        ┌───────┴───────┐  ┌──────────┐
        │ Fibre Channel │  │  NFS/SMB │
        │ / iSCSI / FC  │  │ Protocol │
        └───────┬───────┘  └────┬─────┘
                │               │
        ┌───────▼───────────────▼───────┐
        │     Shared Disk Array         │
        │  ┌─────┬─────┬─────┬─────┐   │
        │  │Disk1│Disk2│Disk3│Disk4│   │
        │  └─────┴─────┴─────┴─────┘   │
        │     RAID Controller(s)        │
        └───────────────────────────────┘
```

### 1.2 详细分析

#### 数据访问模型

- **块级共享（SAN）**：通过光纤通道（FC）或 iSCSI 协议，将远程磁盘映射为本地的块设备。多个主机可同时访问同一 LUN，但文件系统需要集群文件系统（如 GFS2、OCFS2）来协调并发。
- **文件级共享（NAS）**：通过 NFS、SMB/CIFS 等文件协议共享，文件系统由存储设备端管理，客户端看到的是文件级接口。

#### 一致性保障

共享存储的一致性由存储阵列本身保证（RAID 冗余、写缓存同步、电池保护），上层应用无需关心数据复制逻辑。但多节点并发写需要分布式锁管理器（DLM）或集群文件系统协调。

#### 典型 I/O 路径

```
Application → VFS → Local FS → SCSI/iSCSI Stack → SAN Switch → Storage Array
```

每次 I/O 都经过完整的内核协议栈和网络栈，延迟通常 > 0.5ms（SAN）或 > 1ms（NAS）。

### 1.3 代表系统

| 系统 | 类型 | 说明 |
|------|------|------|
| EMC VMAX / PowerMax | SAN 阵列 | 企业级高端存储，支持主机自动分层、远程复制 |
| NetApp FAS | NAS/SAN | 统一存储，ONTAP 操作系统 |
| IBM GPFS (早期) | 集群文件系统 | NSD 模式通过共享磁盘实现并行文件系统 |
| Oracle RAC + ASM | 数据库 + 共享存储 | RAC 多实例共享 ASM 磁盘组 |
| GFS2 / OCFS2 | 集群文件系统 | 基于共享块设备的 Linux 集群文件系统 |

### 1.4 优缺点

**优点：**
- 集中管理：所有数据在一个位置，备份、快照、复制策略统一
- 经过验证的可靠性：企业级阵列提供 99.999% 可用性，RAID + 热备盘 + 双控制器
- 应用透明：对现有应用几乎零改造，挂载即可使用
- 高性能阵列缓存：高端阵列配备大容量 DRAM/NVRAM 缓存，热点数据加速

**缺点：**
- 成本高昂：专有硬件（Fibre Channel HBA、交换机、磁盘柜），许可证费用
- 网络瓶颈：所有 I/O 汇聚到存储网络带宽上限，难以线性扩展
- 规模有限：单个阵列的控制器性能和容量有硬性天花板
- 供应商锁定：专有协议和管理接口，迁移成本高
- 单点风险：虽然阵列本身有高可用设计，但集中式架构在极端故障时影响面大

### 1.5 适用场景

- 传统企业核心数据库（Oracle RAC、SQL Server Always On）
- 对数据一致性和可靠性要求极高的场景
- 中小规模（< 100 节点）且预算充足的企业环境
- 需要统一存储管理、快照、远程复制等企业级功能的场景

## 2. 本地分布式存储 (Local Distributed Storage)

### 2.1 架构概述

每个计算节点使用本地磁盘存储数据，节点之间通过网络协调，形成分布式存储集群。这是"无共享（Shared-Nothing）"架构的典型实现。

```
┌─────────────────────────────────────────────────────────────┐
│                      Cluster Coordination                    │
│                   (Metadata / Gossip / Raft)                 │
├────────────┬────────────┬────────────┬───────────────────────┤
│   Node A   │   Node B   │   Node C   │   Node D              │
│ ┌────────┐ │ ┌────────┐ │ ┌────────┐ │ ┌────────┐           │
│ │Local FS│ │ │Local FS│ │ │Local FS│ │ │Local FS│           │
│ ├────────┤ │ ├────────┤ │ ├────────┤ │ ├────────┤           │
│ │SSD/HDD │ │ │SSD/HDD │ │ │SSD/HDD │ │ │SSD/HDD │           │
│ │Data:   │ │ │Data:   │ │ │Data:   │ │ │Data:   │           │
│ │Shard 1 │ │ │Shard 2 │ │ │Shard 3 │ │ │Shard 4 │           │
│ │Shard 2*│ │ │Shard 3*│ │ │Shard 4*│ │ │Shard 1*│           │
│ └────────┘ │ └────────┘ │ └────────┘ │ └────────┘           │
│            │            │            │                       │
│  (* = replica from another node)                              │
└────────────┴────────────┴────────────┴───────────────────────┘

         Replication over Cluster Network (TCP/IP)
```

### 2.2 详细分析

#### 数据分布

- **一致性哈希**：Cassandra、Dynamo 采用虚拟节点环，将数据映射到哈希环上的位置
- **范围分区**：HBase、TiKV 使用有序的 RowKey 范围划分 Region/Tablet
- **固定分片**：Elasticsearch、MongoDB 通过 shard 键将文档分配到固定分片

#### 复制策略

- **同步复制**：主从同步写入（TiKV Raft、HDFS 写管道），保证强一致性，但延迟受最慢节点影响
- **异步复制**：主节点本地确认后异步复制（Cassandra tunable consistency），延迟低但可能丢失数据
- **Quorum 读写**：Dynamo 风格，W+R > N 保证一致性，可灵活权衡

#### 容错与重建

节点故障时，其他节点上的副本继续服务。新节点加入或故障节点恢复后，需要数据重建（rebuild/streaming），这个过程消耗大量网络和磁盘 I/O。

#### 数据本地性

计算与数据在同一节点，本地读零网络开销。但跨节点查询需要网络传输，且调度器应尽量将计算任务分配到数据所在节点（data locality scheduling）。

### 2.3 代表系统

| 系统 | 数据模型 | 复制机制 | 典型规模 |
|------|----------|----------|----------|
| HDFS | 文件（块级） | 3 副本，写管道 | 千节点，PB 级 |
| Apache Cassandra | 宽列（键值） | 异步，Tunable Consistency | 万节点 |
| TiKV | 键值（Raft Group） | Raft 同步复制 | 数百节点 |
| Ceph OSD | 对象/块/文件 | CRUSH + 多副本/EC | 万节点 |
| RocksDB | 键值（嵌入式） | 上层复制（Raft/Replica） | 单机嵌入式 |
| Elasticsearch | 文档（倒排索引） | 主从副本 | 百节点 |
| MongoDB | 文档 | Replica Set + Sharding | 百节点 |

### 2.4 优缺点

**优点：**
- 使用 commodity hardware：标准 x86 服务器 + SATA/SAS/NVMe 磁盘
- 线性扩展：增加节点即增加容量和吞吐，理论无上限
- 无共享瓶颈：不依赖中心化存储设备，消除单点
- 数据本地性：计算与数据同节点，本地读零网络延迟
- 开源生态丰富：大量成熟项目可供选择

**缺点：**
- 运维复杂：集群管理、容量规划、节点扩缩容均需专业运维
- 数据重建开销：节点故障后副本重建消耗大量网络和磁盘带宽
- 存储利用率：多副本方案（通常 3 副本）存储利用率仅 33%
- 扩容需要数据重平衡：增加节点后需迁移大量数据
- 跨节点查询性能差：需要网络传输数据或中间结果

### 2.5 适用场景

- 大数据处理（Hadoop 生态、Spark）
- 分布式 NoSQL 数据库（高并发读写）
- 需要 PB 级存储且预算有限的场景
- 微服务/容器化环境中的分布式状态管理

## 3. 存算分离 (Compute-Storage Separation)

### 3.1 架构概述

将计算层和存储层解耦为独立的组件，各自独立扩展。计算节点无状态，存储层独立持久化数据。

```
┌──────────────────────────────────────────────────────────┐
│                    Stateless Compute Layer                 │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │Compute A│ │Compute B│ │Compute C│ │Compute D│  ← 可弹性扩缩容
│  │(Query/  │ │(Query/  │ │(Query/  │ │(Query/  │        │
│  │ Process)│ │ Process)│ │ Process)│ │ Process)│        │
│  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘        │
│       │           │           │           │              │
│       └───────────┴─────┬─────┴───────────┘              │
│                    │                                      │
├────────────────────┼──────────────────────────────────────┤
│              Network (TCP/RDMA/S3 API)                     │
│                    │                                      │
├────────────────────┼──────────────────────────────────────┤
│              Persistent Storage Layer                      │
│  ┌─────────────────┴─────────────────┐                   │
│  │  ┌───────┐ ┌───────┐ ┌───────┐   │                   │
│  │  │Store A│ │Store B│ │Store C│   │  ← 独立扩展存储    │
│  │  │(Block/│ │(Object│ │(Block/│   │                    │
│  │  │ File) │ │ /File)│ │ File) │   │                    │
│  │  └───────┘ └───────┘ └───────┘   │                   │
│  │       Distributed / RAID          │                   │
│  └───────────────────────────────────┘                   │
└──────────────────────────────────────────────────────────┘
```

### 3.2 详细分析

#### 核心设计理念

存算分离的核心是将 **状态（数据）** 和 **无状态（计算）** 彻底分开：

1. **存储层**：负责数据持久化、可靠性、多副本/EC 冗余。提供标准的访问接口（块设备、文件系统、对象 API）。
2. **计算层**：完全无状态，可以从后端存储读取数据，执行计算后写回。计算节点可以随时启动/销毁，不影响数据持久性。

#### 数据一致性挑战

- **并发写协调**：多个计算节点同时写入同一数据块时需要分布式锁或乐观并发控制
- **缓存一致性**：计算节点本地缓存与后端存储的一致性
- **事务支持**：跨计算节点的事务需要分布式事务协议（如两阶段提交、Percolator 模型）

#### 实现模式

| 模式 | 说明 | 代表 |
|------|------|------|
| 共享磁盘 | 计算节点共享同一块存储设备（网络块设备） | PolarDB, Aurora |
| 共享对象 | 计算节点通过对象 API（S3）读写数据 | Snowflake, JuiceFS |
| 共享日志 | 存储层是预写日志（WAL），计算节点消费日志 | Pravega |

### 3.3 代表系统

| 系统 | 计算层 | 存储层 | 说明 |
|------|--------|--------|------|
| Snowflake | 虚拟仓库（无状态 EC2） | 内部 S3 兼容存储 | 数据仓库存算分离典范 |
| Amazon Aurora | RDS 实例 | Aurora 分布式存储（6 副本跨 3 AZ） | 数据库存算分离 |
| PolarDB | PolarDB 计算节点 | PolarStore (RDMA + 分布式存储) | 阿里云数据库 |
| JuiceFS | POSIX/FUSE 客户端 | 元数据引擎 + 对象存储 | 云原生分布式文件系统 |
| Alluxio | 内存计算缓存层 | 底层文件系统（HDFS/S3） | 数据编排层 |
| Databricks Delta Lake | Spark 计算引擎 | S3/ADLS 对象存储 | 数据湖计算引擎 |

### 3.4 优缺点

**优点：**
- 独立弹性扩展：计算和存储按需独立扩缩，互不影响
- 成本效率：计算按需启停（spot 实例、serverless），存储按实际使用量计费
- 多租户隔离：不同计算集群可以安全访问同一份数据
- 灵活的计算框架：同一份数据可被 SQL 引擎、Spark、自定义应用并发使用
- 灾备简单：存储层天然多副本/跨区域复制

**缺点：**
- 网络延迟：每次数据访问都需要网络往返，延迟高于本地磁盘
- 缓存管理：需要智能缓存策略减少重复网络请求
- 并发写入复杂：多计算节点写同一数据需要复杂的协调机制
- 依赖后端存储性能：计算性能上限受存储层吞吐和延迟制约
- 最终一致性：对象存储通常是最终一致性模型，强一致性场景需要额外设计

### 3.5 适用场景

- 云原生数据仓库/数据湖（Snowflake、Databricks 模式）
- 数据库服务的弹性扩展（Aurora、PolarDB 模式）
- 多计算引擎共享同一数据源的场景
- 需要按实际使用量付费、弹性伸缩的云端应用

## 4. 对象存储后端 (Object Storage Backend)

### 4.1 架构概述

将对象存储（通常是 S3 兼容）作为底层数据持久化层，上层系统通过对象 API 进行数据读写。对象存储本身提供高持久性和可扩展性。

```
┌────────────────────────────────────────────────────────┐
│                  Application / Service                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ JuiceFS  │  │Alluxio   │  │Delta Lake│  ← 上层系统   │
│  │  Client  │  │  Client  │  │  Reader  │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       └──────┬──────┴──────┬──────┘                     │
│              │  S3 API (REST over HTTP)                  │
├──────────────┼─────────────┼────────────────────────────┤
│              ▼             ▼                            │
│  ┌─────────────────────────────────────────────┐       │
│  │          S3-Compatible Object Storage         │       │
│  │                                              │       │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐      │       │
│  │  │ Bucket A│  │ Bucket B│  │ Bucket C│      │       │
│  │  │  obj-1  │  │  obj-1  │  │  obj-1  │      │       │
│  │  │  obj-2  │  │  obj-2  │  │  obj-2  │      │       │
│  │  │  obj-N  │  │  obj-N  │  │  obj-N  │      │       │
│  │  └─────────┘  └─────────┘  └─────────┘      │       │
│  │                                              │       │
│  │  ┌────────────────────────────────────────┐  │       │
│  │  │  Erasure Coding / Replication Layer     │  │       │
│  │  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐   │  │       │
│  │  │  │AZ-1│ │AZ-2│ │AZ-3│ │AZ-4│ │AZ-5│   │  │       │
│  │  │  └────┘ └────┘ └────┘ └────┘ └────┘   │  │       │
│  │  └────────────────────────────────────────┘  │       │
│  └──────────────────────────────────────────────┘       │
└────────────────────────────────────────────────────────┘
```

### 4.2 详细分析

#### 对象存储的本质

对象存储将数据作为不透明的"对象"存储，每个对象包含：
- **数据体**：任意大小的二进制数据
- **元数据**：用户自定义键值对
- **全局唯一标识**：Key 或 Object ID

与传统文件系统不同，对象存储：
- 没有目录层级（扁平命名空间，前缀模拟目录）
- 不支持原地修改（写操作是覆盖整个对象）
- 通常是最终一致性模型（部分厂商已支持强一致性）

#### 上层系统如何使用对象存储

**JuiceFS 模式**：
```
用户文件 → JuiceFS Client → 分块 → 对象存储（数据块）
                           → 元数据引擎（目录树、权限、分块映射）
```

**Delta Lake 模式**：
```
DataFrame 写入 → 数据写为 Parquet 文件 → S3 对象存储
              → 事务日志（_delta_log）→ S3 对象存储
```

**Alluxio 模式**：
```
应用请求 → Alluxio（内存缓存命中？）→ 是：直接返回
                              → 否：从 S3 拉取 → 缓存 → 返回
```

#### 数据切分策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 大对象直存 | 一个文件对应一个对象 | 日志、备份、大文件 |
| 分块存储 | 大文件切分为固定大小块（如 64MB） | 文件系统、需要随机读的场景 |
| 追加写 | 以追加方式写入不可变对象 | 日志流、时序数据 |

### 4.3 代表系统

| 系统 | 角色 | 说明 |
|------|------|------|
| Amazon S3 | 对象存储 | 行业标准，11 个 9 持久性 |
| MinIO | 对象存储 | 开源 S3 兼容，可自部署 |
| Ceph RGW | 对象存储 | Ceph 的对象网关层 |
| JuiceFS | 文件系统层 | 元数据引擎 + 对象存储数据层 |
| Alluxio | 数据编排层 | 内存缓存 + 对象存储作为底层 |
| Delta Lake | 数据湖格式 | 基于对象存储的事务性数据湖 |
| Apache Iceberg | 表格式 | 对象存储上的表级 ACID |
| Apache Hudi | 数据湖格式 | 对象存储上的增量数据处理 |

### 4.4 优缺点

**优点：**
- 极致耐久性：S3 提供 99.999999999%（11 个 9）持久性
- 无限扩展：理论上对象数量和容量无上限
- 按需付费：用多少付多少，无预置容量限制
- 跨区域复制：原生支持多区域数据复制
- 生态丰富：几乎所有云服务和开源工具都支持 S3 API

**缺点：**
- 延迟较高：HTTP REST API 往返延迟通常 > 10ms（对比本地 SSD < 0.1ms）
- 不支持原地修改：每次写操作需要重写整个对象
- 最终一致性：部分场景下读取可能读到旧数据
- 小文件性能差：大量小对象导致元数据操作成为瓶颈
- 有限随机访问：虽然有 Range GET，但不如块存储灵活

### 4.5 适用场景

- 数据湖（Data Lake）存储底座
- 备份和归档（冷数据、合规存储）
- 静态资源分发（图片、视频、软件包）
- 云原生文件系统的后端存储
- 多区域数据分发和灾难恢复

## 5. 内存+持久化分层 (Tiered Memory+Persistence)

### 5.1 架构概述

将数据按照访问频率分层存储：热数据保存在内存中获得极致性能，温/冷数据下沉到磁盘/SSD 以降低成本。

```
┌──────────────────────────────────────────────────────────┐
│                    Application Layer                       │
└────────────────────────┬─────────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────────┐
│                 Memory Tier (Hot Data)                     │
│  ┌─────────────────────────────────────────────────────┐ │
│  │                   DRAM (GB-TB)                       │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │ │
│  │  │ Key-Val │ │  Cache  │ │ Working │ │  Index  │   │ │
│  │  │  Store  │ │ Entries │ │  Set    │ │  Data   │   │ │
│  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘   │ │
│  │               Access: ~100ns                         │ │
│  └─────────────────────────────────────────────────────┘ │
│                         │                                 │
│              Eviction Policy (LRU/LFU/TTL)                │
├─────────────────────────┼─────────────────────────────────┤
│                 SSD Tier (Warm Data)                       │
│  ┌─────────────────────────────────────────────────────┐ │
│  │              NVMe SSD (TB-PB)                         │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │ │
│  │  │Evicted  │ │  Index  │ │  Recent │ │ Prefetch│   │ │
│  │  │  Data   │ │  Pages  │ │  Logs   │ │  Data   │   │ │
│  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘   │ │
│  │               Access: ~10-100μs                      │ │
│  └─────────────────────────────────────────────────────┘ │
│                         │                                 │
│             Aging / Compaction / Persistence              │
├─────────────────────────┼─────────────────────────────────┤
│                 Disk/Cloud Tier (Cold Data)                │
│  ┌─────────────────────────────────────────────────────┐ │
│  │           HDD / Object Storage (PB-EB)               │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │ │
│  │  │ Archive │ │ Backups │ │  Snap-  │ │Historical│   │ │
│  │  │  Data   │ │  & AOF  │ │  shots  │ │  Data   │   │ │
│  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘   │ │
│  │               Access: ~1-10ms (HDD) / 10ms+ (S3)    │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 5.2 详细分析

#### 分层策略

**Redis 模式**：
```
热数据：内存中的 Hash/Set/ZSet 等数据结构
持久化：AOF（追加日志）+ RDB（定期快照）
扩展：Redis on Flash — 将 Value 存储到 SSD，Key 仍在内存
```

**Alluxio 模式**：
```
L1（内存）：RAMDisk，热数据缓存，LRU 淘汰
L2（SSD）：本地 SSD，温数据缓存
Under FS：HDFS/S3，冷数据持久层
```

**Tiered Cache 通用模型**：
```
读路径：L1 命中 → 返回
       L1 未命中 → L2 命中 → 提升到 L1 → 返回
       L1/L2 均未命中 → 底层存储 → 写入 L1+L2 → 返回

写路径：写入 L1 → 异步刷到 L2 → 异步持久化到底层
       或：同步写 L1 + WAL → 后台刷盘
```

#### 淘汰算法

| 算法 | 策略 | 适用场景 |
|------|------|----------|
| LRU | 淘汰最久未访问的数据 | 通用场景，时间局部性好 |
| LFU | 淘汰访问频率最低的数据 | 热点数据稳定的场景 |
| ARC | 自适应替换缓存 | 访问模式变化的场景 |
| TinyLFU | 概率性频率计数 | 高吞吐缓存（Caffeine） |
| TTL | 基于过期时间 | 会话数据、临时缓存 |
| W-TinyLFU | Window + Main 分区 | 现代缓存系统（Caffeine/Redis） |

#### 数据持久化方式

- **AOF（Append-Only File）**：每次写操作追加到日志文件，恢复时重放。数据安全性高，但文件可能很大
- **RDB（Snapshot）**：定期全量快照。恢复快，但可能丢失上次快照后的数据
- **混合持久化**：AOF + RDB 混合（Redis 4.0+），兼顾安全性和恢复速度

### 5.3 代表系统

| 系统 | 内存层 | 持久层 | 说明 |
|------|--------|--------|------|
| Redis | 内存数据结构 | AOF/RDB + Redis on Flash | 内存数据库标杆 |
| Alluxio | RAMDisk | SSD + Under FS | 分布式内存缓存 |
| Memcached | 内存 LRU | 无（纯缓存） | 简单高性能缓存 |
| Apache Ignite | 内存数据网格 | 磁盘持久化 | 内存计算平台 |
| Dragonfly | 内存 + 线程模型优化 | AOF | Redis 兼容高性能替代 |
| KeyDB | 多线程内存 | AOF/RDB | Redis 多线程分支 |
| Caffeine | JVM 内存 | 无（进程内） | Java 高性能缓存库 |

### 5.4 优缺点

**优点：**
- 极致性能：内存访问延迟 ~100ns，比 SSD 快 1000 倍
- 成本效率：热数据用内存，冷数据用廉价存储，性价比最优
- 自动分层：对用户透明，系统自动管理数据迁移
- 灵活持久化：可根据业务需求选择持久化级别（无、异步、同步）

**缺点：**
- 内存昂贵：DRAM 成本约为 SSD 的 10-20 倍
- 驱逐策略复杂：需要精细调优缓存大小、淘汰算法、TTL
- 冷启动/预热问题：系统重启后缓存为空，需要预热期
- 数据一致性：多层缓存间的同步和一致性维护复杂
- 内存碎片：长期运行后内存碎片可能导致实际可用容量下降

### 5.5 适用场景

- 高频读/低延迟要求的缓存层（Session、API 响应、查询结果）
- 实时分析系统的热数据加速
- 数据库前置缓存（Read-Through / Write-Through）
- 需要极致吞吐量的场景（广告竞价、实时风控）

## 6. 裸设备/用户态 I/O (Bare Device / Userspace I/O)

### 6.1 架构概述

绕过操作系统内核 I/O 栈，直接在用户态访问存储设备。通过轮询模式、零拷贝、无锁数据结构等技术，实现最低延迟和最高吞吐。

```
┌──────────────────────────────────────────────────────────┐
│                    Application Layer                       │
└────────────────────────┬─────────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────────┐
│              Traditional Kernel I/O Path (对比)             │
│                                                          │
│  App → syscall → VFS → Page Cache → Block Layer →       │
│        → SCSI/NVMe Driver → Interrupt → Context Switch   │
│        Latency: 每个 I/O 经历多次上下文切换和内存拷贝       │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│              Userspace I/O Path (SPDK/DPDK)                │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │              Application (Userspace)                  │ │
│  │                  │                                   │ │
│  │  ┌───────────────▼───────────────┐                   │ │
│  │  │     SPDK Framework            │                   │ │
│  │  │  ┌─────────────────────────┐  │                   │ │
│  │  │  │  Poll Mode Driver       │  │  ← 轮询替代中断    │ │
│  │  │  │  (无中断, 忙等待)        │  │                   │ │
│  │  │  └─────────────────────────┘  │                   │ │
│  │  │  ┌─────────────────────────┐  │                   │ │
│  │  │  │  Zero-Copy Buffers      │  │  ← 零拷贝 DMA      │ │
│  │  │  │  (共享内存, 无拷贝)      │  │                   │ │
│  │  │  └─────────────────────────┘  │                   │ │
│  │  │  ┌─────────────────────────┐  │                   │ │
│  │  │  │  Lock-Free Queues       │  │  ← 无锁数据结构    │ │
│  │  │  │  (SPSC/MPMC)            │  │                   │ │
│  │  │  └─────────────────────────┘  │                   │ │
│  │  └───────────────┬───────────────┘                   │ │
│  │                  │                                   │ │
│  │     PCI Express / NVMe Direct                        │ │
│  └──────────────────┼───────────────────────────────────┘ │
│                     │                                     │
│              ┌──────▼──────┐                              │
│              │  NVMe SSD   │  ← 内核旁路, 直接 MMIO        │
│              └─────────────┘                              │
└──────────────────────────────────────────────────────────┘
```

### 6.2 详细分析

#### 内核 I/O vs 用户态 I/O

**内核 I/O 路径的开销**：
```
1. 系统调用 (syscall)         → ~50-100ns 上下文切换
2. VFS 层查找                 → 路径解析、权限检查
3. 页缓存管理                 → 内存分配、拷贝
4. 块设备层                   → I/O 调度、合并
5. 设备驱动                   → DMA 映射
6. 硬件中断                   → 中断处理、唤醒进程
7. 再次上下文切换             → 返回用户态
总延迟：通常 > 1μs（不含硬件延迟）
```

**用户态 I/O 的优化**：
```
1. 无系统调用                 → 零上下文切换
2. 轮询模式 (Poll Mode)       → 无中断开销
3. 零拷贝 DMA                 → 数据直接从设备到用户缓冲区
4. 无锁队列                   → 无锁竞争
5. CPU 亲和性绑定             → 无跨核迁移
6. HugePages                  → 减少 TLB miss
总延迟：可低至 10μs（含 NVMe 硬件延迟）
```

#### 核心技术

**SPDK（Storage Performance Development Kit）**：
- 将 NVMe、iSCSI、vhost 等驱动移到用户态
- 使用轮询模式替代中断驱动
- 提供线程安全的无锁数据结构
- 支持 NVMe-oF（NVMe over Fabrics）

**NVMe-oF（NVMe over Fabrics）**：
```
计算节点                          存储节点
┌──────────┐                  ┌──────────┐
│ SPDK     │   NVMe-oF (RDMA) │ SPDK     │
│ NVMe-oF  │◄────────────────►│ Target   │
│ Initiator│   TCP/RDMA/TCP   │ (NVMe    │
│          │                  │  SSD)    │
└──────────┘                  └──────────┘

传输方式：
- RDMA (RoCE v2 / InfiniBand): < 5μs RTT
- TCP: ~10-20μs RTT
- FC: 传统光纤通道
```

**Ceph BlueStore**：
- 绕过文件系统，直接操作裸设备
- WAL（预写日志）和 DB（元数据）可放到独立 SSD
- BlueFS（轻量级内嵌文件系统）管理数据文件
- 直接通过 RocksDB 管理对象元数据

### 6.3 代表系统

| 系统 | 技术 | 性能指标 |
|------|------|----------|
| SPDK | NVMe 用户态驱动 | 单核 1M+ IOPS, < 10μs 延迟 |
| Ceph BlueStore | 裸设备直写 | 对比 FileStore 提升 2-5x |
| NVMe-oF Target | 远程 NVMe 访问 | RDMA 传输 < 5μs 延迟 |
| DPDK | 网络用户态驱动 | 网络包处理 10M+ pps/核 |
| io_uring | 异步 I/O 接口 | Linux 内核优化, 减少 syscall |
| eBPF Storage | 内核可编程 | 灵活的 I/O 路径定制 |

### 6.4 优缺点

**优点：**
- 最低延迟：消除内核上下文切换和中断开销
- 最高吞吐：轮询模式 + 零拷贝 + 无锁队列
- CPU 效率高：每个 CPU 核可处理百万级 IOPS
- 确定性延迟：无内核调度不确定性

**缺点：**
- 实现复杂：需要自行管理内存、I/O 队列、错误处理
- CPU 占用：轮询模式持续占用 CPU，空闲时也消耗
- 硬件要求：需要 NVMe SSD、RDMA 网卡等高性能硬件
- 运维难度：用户态驱动调试困难，缺乏内核工具支持
- 安全性：绕过内核安全机制，需要自行实现访问控制
- 生态局限：并非所有存储协议都支持用户态实现

### 6.5 适用场景

- 超低延迟交易系统（金融高频交易）
- NVMe SSD 极致性能释放
- 分布式存储的后端引擎（Ceph OSD 优化）
- 高性能网络存储网关（NVMe-oF target）
- 需要确定性延迟的实时系统

## 7. 架构演进趋势

### 7.1 演进路线

```
1990s          2000s              2010s              2020s
  │              │                  │                  │
  ▼              ▼                  ▼                  ▼
┌──────┐   ┌──────────┐      ┌────────────┐     ┌────────────┐
│共享存储│──→│本地分布式│ ──→  │ 存算分离    │──→  │ Serverless  │

└──────┘   └──────────┘      └────────────┘     └────────────┘
   │             │                  │                  │
   • 专有硬件    • Commodity HW     • 云原生           • 按需付费
   • 集中式      • Shared-Nothing   • 弹性伸缩         • 完全托管
   • 强一致性    • 最终一致性       • 多引擎共享       • 无服务器
   • 高成本      • 运维复杂         • 成本优化         • 零运维
   • 有限扩展    • 线性扩展         • 独立扩缩         • 秒级弹性
```

### 7.2 详细演进分析

#### 第一阶段：共享存储 (1990s - 2000s)

**时代背景**：互联网尚未普及，企业 IT 以传统应用为主，数据量在 GB 级别。

**核心特征**：
- 昂贵的专有硬件（EMC、NetApp）
- 光纤通道 SAN 网络
- 集群文件系统解决并发访问
- 数据量：GB ~ 低 TB 级
- 节点数：< 100

**代表架构**：
- Oracle RAC + ASM + SAN
- IBM DB2 DPF + GPFS
- 传统虚拟化 + NAS

#### 第二阶段：本地分布式存储 (2000s - 2010s)

**时代背景**：Google 三论文发表（GFS、MapReduce、BigTable），互联网数据爆炸。

**核心特征**：
- Google 三论文开源化（Hadoop 生态）
- Commodity hardware 替代专有硬件
- 最终一致性取代强一致性
- 数据量：TB ~ PB 级
- 节点数：百 ~ 万级

**代表架构**：
- Hadoop (HDFS + MapReduce)
- Cassandra / Dynamo 风格的 NoSQL
- 开源分布式数据库 (MySQL 分片、MongoDB)
- Ceph 统一存储平台

**关键转折**：从"买更好的硬件"转向"用更多便宜的硬件"。

#### 第三阶段：存算分离 (2010s - 2020s)

**时代背景**：云计算成熟，对象存储（S3）成为事实标准，弹性需求驱动架构变革。

**核心特征**：
- 计算与存储彻底解耦
- 对象存储成为通用数据底座
- 多计算引擎共享同一数据
- Serverless 计算兴起
- 数据量：PB ~ EB 级

**代表架构**：
- Snowflake（数据仓库存算分离）
- Amazon Aurora（数据库存算分离）
- Databricks Delta Lake（数据湖计算）
- JuiceFS / Alluxio（云原生文件系统）

**关键转折**：从"数据跟着计算走"转向"计算去找数据"。

#### 第四阶段：Serverless (2020s - 未来)

**时代背景**：云原生成熟，开发者希望完全不关心基础设施。

**核心特征**：
- 完全托管，零运维
- 秒级自动弹性扩缩
- 按实际使用量精确计费
- 开发者的视角只有 API

**代表架构**：
- AWS Aurora Serverless v2
- Snowflake Serverless
- Google BigQuery（完全 serverless）
- Cloudflare R2 + Workers（边缘计算 + 存储）
- Neon（Postgres serverless，存算分离 + 分支）

### 7.3 演进驱动力

| 维度 | 共享存储 → 本地分布式 | 本地分布式 → 存算分离 | 存算分离 → Serverless |
|------|----------------------|----------------------|----------------------|
| 核心驱动 | 成本 + 规模 | 弹性 + 多租户 | 运维简化 + 按需付费 |
| 数据规模 | GB → PB | PB → EB | EB → 无上限 |
| 硬件依赖 | 专有硬件 | 标准服务器 | 云基础设施 |
| 运维成本 | 高（厂商支持） | 极高（专业团队） | 低（完全托管） |
| 一致性 | 强一致 | 最终一致 | 可配置 |
| 扩展粒度 | 垂直（升级硬件） | 水平（加节点） | 自动（毫秒级） |

### 7.4 未来趋势

```
                    ┌──────────────────┐
```

### 8.3 选型决策树

```
需要存储数据？
│
├─ 数据量 < 10TB，预算充足，需要企业级 SLA？
│  └─→ 共享存储 (SAN/NAS)
│
├─ 数据量 > 100TB，预算有限，有运维团队？
│  └─→ 本地分布式存储 (HDFS/Ceph/Cassandra)
│
├─ 部署在云上，需要弹性伸缩，多引擎共享数据？
│  └─→ 存算分离 (Snowflake/JuiceFS/Aurora)
│
├─ 数据湖场景，需要 11 个 9 持久性，多区域复制？
│  └─→ 对象存储后端 (S3 + Delta Lake/Iceberg)
│
├─ 读多写少，延迟敏感（< 1ms），热点数据明显？
│  └─→ 内存+持久化分层 (Redis/Alluxio)
│
└─ 极致性能要求（< 10μs），有高性能硬件，有底层开发能力？
   └─→ 裸设备/用户态 I/O (SPDK/NVMe-oF/BlueStore)
```
