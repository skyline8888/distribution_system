# Lakehouse（湖仓一体）

## 基本信息

| 维度 | 详情 |
|------|------|
| **概念提出** | ~2020 年（Databricks 发表《What is a Lakehouse?》白皮） |
| **核心推动者** | Databricks（Matei Zaharia、Reynold Xin、Matei Zaharia 等） |
| **核心论文** | "Lakehouse: A New Generation of Open Platforms that Unify Data Warehousing and Advanced Analytics"（CIDR 2021） |
| **架构定义** | Data Lake + Data Warehouse = 统一数据平台 |
| **关键技术** | 开放表格式（Delta Lake / Iceberg / Hudi）+ 计算存储分离 + 多引擎访问 |
| **代表实现** | Databricks Lakehouse Platform、ByteHouse LAS、阿里云 DLF + MaxCompute |

**一句话定义：** Lakehouse 是在廉价对象存储之上，通过表格式层赋予数据湖以数据仓库的 ACID 事务、Schema 治理、高性能查询能力，使 BI 和 ML 工作流在**同一份数据**上运行的统一架构。

---

## 1. Context & Motivation（背景与动机）

### 1.1 解决的问题

2010s 末，企业数据架构普遍陷入 **"双系统困境"**：

```
┌──────────────┐         ┌──────────────┐
│  数据仓库     │         │   数据湖      │
│  (Warehouse) │         │   (Lake)      │
│              │         │               │
│ ✅ BI/SQL    │         │ ✅ ML/AI       │
│ ✅ ACID      │         │ ✅ 任意格式    │
│ ✅ 高性能     │         │ ✅ 低成本      │
│ ❌ 高成本     │         │ ❌ 无 ACID     │
│ ❌ 仅限结构化  │         │ ❌ 低质量      │
│ ❌ 厂商锁定   │         │ ❌ 数据沼泽    │
└──────────────┘         └──────────────┘
```

**双系统困境的典型症状：**

| 问题 | 表现 |
|------|------|
| **数据冗余** | 同一份数据在数仓和湖中各存一份，ETL 同步，成本翻倍 |
| **数据不一致** | 数仓和湖中的数据版本不同步，分析师和 ML 工程师看到不同结果 |
| **ETL 复杂性** | 需要维护数条从湖到数仓的 ETL 管道，延迟 T+1 甚至更长 |
| **新鲜度低** | 数仓数据是湖数据的"昨日快照"，无法支持实时分析 |
| **非结构化数据无法利用** | 数仓无法存图片/日志/JSON，ML 需要的原始数据只能放在湖中 |

### 1.2 前代系统的局限

| 架构 | 局限 |
|------|------|
| **纯数据仓库** | 昂贵（专有存储）、仅限结构化、Schema-on-Write 变更困难、不适合 ML |
| **纯数据湖** | 无 ACID 导致数据不可靠、并发写入产生脏数据、无 Schema 治理、查询性能差 |
| **Lambda 架构** | 批流双路径，两套代码，维护噩梦，结果不一致 |

### 1.3 硬件/网络趋势使能

| 趋势 | 影响 |
|------|------|
| **对象存储成熟且廉价** | S3/ADLS/OSS 提供 EB 级、99.999999999% 持久性、$0.023/GB/月 |
| **计算存储分离成为范式** | Snowflake 证明了存算分离的商业可行性 |
| **列式存储标准化** | Parquet 成为事实标准，压缩比高、谓词下推、跨引擎兼容 |
| **2020 年 S3 强一致性** | AWS S3 改为强一致，消除了 Lakehouse ACID 的最后一道障碍 |
| **云原生 SQL 引擎成熟** | Spark SQL / Trino / Presto 提供高性能分布式查询 |
| **ML 工作流数据需求增长** | 深度学习需要海量原始数据，推动 Lake 与 Warehouse 融合 |

### 2.2 核心设计原则

Lakehouse 架构有 **9 个设计原则**（来自 CIDR 2021 论文）：

| # | 原则 | 说明 |
|---|------|------|
| 1 | **可靠的事务** | ACID 语义，多 Writer 并发安全 |
| 2 | **Schema 治理** | Schema enforcement + evolution |
| 3 | **统一的存储** | 单一数据副本服务于 BI + ML + 流处理 |
| 4 | **开放的格式** | Parquet 等开放格式，无厂商锁定 |
| 5 | **多引擎支持** | SQL / Python / R / ML 框架直接访问 |
| 6 | **支持各种数据类型** | 结构化、半结构化、非结构化 |
| 7 | **直接访问原始数据** | ML 框架读取原始文件，无需 ETL 到中间格式 |
| 8 | **端到端优化** | 缓存、索引、物化视图、查询优化 |
| 9 | **存算分离** | 独立扩展计算和存储资源 |

## 3. Core Technical Innovations（核心技术创新）

### 3.1 开放表格式（Open Table Format）

表格式是 Lakehouse 的**核心技术层**，它在对象存储的"哑存储"之上叠加了数据库语义：

```
传统数据库:  存储引擎 + 事务管理 + 查询优化 → 一体化
数据湖:      对象存储（哑存储） → 无事务、无 Schema
Lakehouse:   对象存储 + 表格式层 → 事务 + Schema + 优化
```

**三大表格式对比矩阵：**

| 维度 | Apache Iceberg | Delta Lake | Apache Hudi |
|------|---------------|------------|-------------|
| **发起方** | Netflix (2018) | Databricks (2019) | Uber (2016) |
| **元数据模型** | 四层树形 (metadata.json → manifest list → manifest → data) | 事务日志 (_delta_log JSON commits + checkpoint) | 时间线 + Metadata Table (HBase/RocksDB) |
| **快照模型** | 不可变快照树 | 版本号递增 + Time Travel | Timeline (Instant-based) |
| **写入模型** | 快照隔离 + Manifest 管理 | CoW (默认) | CoW + MoR (原生) |
| **行级更新** | v2: Position + Equality Deletes | ✅ 原生 MERGE INTO | ✅ 原生 UPSERT |
| **CDC** | 有限 | Change Data Feed (3.0+) | ✅ 原生 |
| **流式写入** | 有限 (v3 增强中) | ✅ Structured Streaming 集成 | ✅ 最强 |
| **Schema 演化** | ✅ 最完善 (列 ID 机制) | ✅ autoMerge | 有限 |
| **引擎支持** | 最广 (Spark/Flink/Trino/Presto/StarRocks/Doris/ClickHouse) | Spark/Databricks/Trino | Spark/Flink/Trino |
| **厂商中立性** | ✅ 最高 | 较低 (Databricks 主导) | ✅ 高 |
| **中国生态** | 阿里云/火山引擎原生 | 较少直接使用 | 字节跳动内部使用 |

### 3.2 ACID 事务 — 在对象存储上实现事务

Lakehouse 的 ACID 实现基于 **OCC（乐观并发控制）** + **底层存储的原子写入**：

```
┌──────────────────────────────────────────────────────────┐
│           Lakehouse 事务协议 (OCC)                        │
│                                                          │
│  Writer:                                                 │
│  1. 读取当前表快照（metadata.json / _last_checkpoint）    │
│     → 记录版本号 v                                        │
│  2. 写入数据文件到对象存储                                 │
│     → Parquet 文件原子写入（S3 PUT 是原子的）               │
│  3. 准备新的元数据文件（commit / snapshot）                │
│  4. CAS: 将表指针从 v 更新到 v+1                          │
│     → 成功: 事务提交                                      │
│     → 失败: 冲突! 重新读取 v+1，合并变更，重试             │
│                                                          │
│  Reader:                                                 │
│  1. 读取当前表快照指针                                    │
│  2. 解析快照到文件集合                                     │
│  3. 读取数据文件（并发、无需协调）                          │
│                                                          │
│  保证:                                                   │
│  - 原子性: 要么整个 commit 存在，要么不存在                 │
│  - 一致性: 读者始终看到完整的快照                          │
│  - 隔离性: Serializable / Snapshot Isolation              │
│  - 持久性: 对象存储保证（11 个 9 的持久性）                 │
└──────────────────────────────────────────────────────────┘
```

**关键洞察：** Lakehouse 不需要分布式锁服务或共识协议。事务的原子性由 **对象存储的单文件原子写入** 保证，一致性由 **不可变快照 + CAS 元数据更新** 保证。这是一种**极简的事务模型**，与 Spanner 的 TrueTime / TiDB 的 Raft + Percolator 形成鲜明对比。

### 3.3 存算分离（Compute-Storage Separation）

```
┌──────────────────────────────────────────────────────────────┐
│                  Lakehouse 存算分离架构                        │
│                                                              │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐             │
│  │ Compute A  │  │ Compute B  │  │ Compute C  │             │
│  │ (SQL/BI)   │  │ (ML/Python)│  │ (Streaming)│             │
│  │ 弹性伸缩    │  │ GPU 集群    │  │ Flink 集群  │             │
│  │ 按需付费    │  │ 独立调度    │  │ 实时处理    │             │
│  └──────┬─────┘  └──────┬─────┘  └──────┬─────┘             │
│         │               │               │                    │
│         └───────┬───────┴───────┬───────┘                    │
│                 ▼               ▼                            │
│         ┌───────────────────────────────┐                   │
│         │      共享对象存储 (S3)         │                   │
│         │                               │                   │
│         │  /lakehouse/                  │                   │
│         │  ├── bronze/ (原始层)          │                   │
│         │  ├── silver/ (清洗层)          │                   │
│         │  ├── gold/   (聚合层)          │                   │
│         │  └── _delta_log/              │                   │
│         └───────────────────────────────┘                   │
│                                                              │
│  优势:                                                       │
│  - 计算资源独立扩展（BI 高峰时加计算节点，不影响存储成本）      │
│  - 存储成本极低（对象存储 $0.023/GB/月 vs 数仓 $0.1-0.5）    │
│  - 多工作负载共享同一份数据                                    │
│  - 零复制克隆、时间点恢复                                     │
└──────────────────────────────────────────────────────────────┘
```

**与传统架构的对比：**

| 架构 | 存储 | 计算 | 扩展性 | 成本 |
|------|------|------|--------|------|
| 传统数仓 (Teradata) | 本地磁盘 | 本地 CPU | 垂直 | 极高 |
| MPP 数仓 (Greenplum) | 本地磁盘 | 本地 CPU | 水平（有限） | 高 |
| 云数仓 (Redshift) | 本地 SSD | 本地 CPU | 手动调整节点 | 中高 |
| 存算分离 (Snowflake) | 云存储 | 虚拟仓库 | 自动 | 中 |
| **Lakehouse** | **对象存储** | **任意引擎** | **完全独立** | **低** |

### 3.4 多引擎访问（Multi-Engine Access）

Lakehouse 的核心承诺是：**同一份数据，多种计算引擎直接访问**：

```
                    ┌─────────────────────┐
                    │   同一份数据 (S3)    │
                    │   Parquet + 元数据   │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
          ▼                    ▼                    ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
   │  SQL 查询     │    │  ML/AI 训练   │    │  流处理       │
   │  Trino/      │    │  Spark MLlib │    │  Flink       │
   │  Presto/     │    │  PyTorch     │    │  Spark SS    │
   │  StarRocks   │    │  TensorFlow  │    │  Kafka Streams│
   └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
          │                   │                   │
          ▼                   ▼                   ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
   │  BI 仪表盘    │    │  特征工程     │    │  实时告警     │
   │  Tableau     │    │  模型训练     │    │  实时监控     │
   └──────────────┘    └──────────────┘    └──────────────┘
```

**实现机制：**
- 开放表格式确保**元数据可被任何引擎解析**
- Parquet 是**跨引擎兼容的数据格式**
- Catalog 提供**统一的表发现接口**（Hive Metastore / REST Catalog）
- 查询引擎只需实现对应的表格式 Reader 插件

### 3.5 数据跳过与索引（Data Skipping & Indexing）

Lakehouse 通过表格式层实现查询优化，无需读取全部数据：

```
Data Skipping（数据跳过）：

  查询: WHERE amount > 100 AND country = 'CN'

  Manifest 中的统计信息：
  ┌──────────────────────────────────────────────────────┐
  │ 文件    │ partition    │ amount {min, max} │ country  │
  ├─────────┼──────────────┼───────────────────┼──────────┤
  │ file1   │ dt=2024-01   │ {10, 90}          │ {CN, US} │ ← max=90 < 100, 跳过
  │ file2   │ dt=2024-01   │ {5, 200}          │ {CN}     │ ← 需要扫描
  │ file3   │ dt=2024-02   │ {100, 500}        │ {US, JP} │ ← country≠CN, 跳过
  │ file4   │ dt=2024-02   │ {120, 300}        │ {CN}     │ ← 需要扫描
  └──────────────────────────────────────────────────────┘
  结果: 只扫描 file2 和 file4，跳过 50% 的数据

  进阶优化：
  - Delta: Z-ORDER / Liquid Clustering（多维数据聚类）
  - Iceberg: 隐藏分区 + 排序表 + v3 索引（实验性）
  - Hudi: 文件组索引 + Bloom Filter 索引
```

### 3.6 分层架构（Medallion Architecture）

Lakehouse 实践中最常用的数据组织模式：

```
┌──────────────────────────────────────────────────────────────┐
│                    Medallion 架构                             │
│                                                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐               │
│  │  Bronze  │───▶│  Silver  │───▶│   Gold   │               │
│  │  (原始层) │    │  (清洗层) │    │  (聚合层) │               │
│  │          │    │          │    │          │               │
│  │ 原始数据  │    │ 清洗/     │    │ 业务指标   │               │
│  │ 无 Schema │    │ 去重/     │    │ 数据模型   │               │
│  │ 追加写入  │    │ 合并/     │    │ 星型模型   │               │
│  │          │    │ Schema 化 │    │ 聚合视图   │               │
│  └──────────┘    └──────────┘    └──────────┘               │
│                                                              │
│  数据流:                                                     │
│  API/日志/CDC → Bronze → ETL/ELT → Silver → 聚合 → Gold     │
│                                                              │
│  特点:                                                       │
│  - 同一存储层，不同"质量层级"                                   │
│  - 每层都是独立的 Lakehouse 表（有 ACID、有时间旅行）           │
│  - 支持双向追溯：从 Gold 指标追溯到 Bronze 原始数据              │
│  - 增量更新：每层只处理增量变更                                 │
└──────────────────────────────────────────────────────────────┘
```

### 3.7 统一批流（Unified Batch & Streaming）

Lakehouse 消除了 Lambda 架构的批流双路径：

```
Lambda 架构 (已过时):
  ┌─────────────────┐    ┌─────────────────┐
  │   批处理路径     │    │   流处理路径      │
  │   (Hadoop/Spark) │    │   (Storm/Flink)  │
  │   T+1 结果       │    │   实时结果        │
  └────────┬────────┘    └────────┬────────┘
           │                      │
           └──────────┬───────────┘
                      ▼
              ┌──────────────┐
              │  结果合并层   │  ← 两套代码，结果可能不一致
              └──────────────┘

Lakehouse Kappa 架构:
  ┌─────────────────────────────────────────────┐
  │          统一存储层 (Lakehouse Table)        │
  │  ┌──────────────────────────────────────┐   │
  │  │  同一张表，同时接受:                    │   │
  │  │  - 批量写入 (Spark batch)              │   │

  │  │  - CDC 摄入 (Debezium → Flink)        │   │

```

## 4. Trade-offs（CAP / PACELC）

### 4.1 CAP 定位

Lakehouse **本身不是分布式系统**（没有独立的节点集群）。CAP 属性取决于 **底层存储 + 表格式** 的组合：

| 组件层级 | CAP 行为 | 说明 |
|----------|---------|------|
| **对象存储层** | 现代云存储：**CP** | S3（2020+ 强一致）、ADLS、GCS 均提供强一致性；旧 S3 为 AP（最终一致） |
| **表格式层** | **CP** | 通过不可变快照 + OCC 保证一致性；写入冲突时重试而非返回过期数据 |
| **Catalog 层** | 取决于实现 | Hive Metastore：单点 C；REST Catalog + 分布式后端：可配置 |
| **计算引擎层** | **AP** | Spark/Trino 是无状态计算节点，天然 AP |

**综合 CAP 定位：**

```
         C (一致性)
          ▲
          │
          │  ★ Lakehouse
          │  (存储=CP + 表格式=CP)
          │
          │
A (可用性) ─┼──────────────────
          │
          │
          ▼
```

**关键约束：** Lakehouse 的 ACID 保证 **仅当底层存储提供原子单文件写入和强一致性时才成立**。在早期 S3（最终一致）上，需要额外的 CAS 机制（如 DynamoDB 锁表）。

### 4.2 PACELC 定位

| 场景 | Lakehouse 的选择 | 说明 |
|------|-----------------|------|
| **Partition（网络分区时）** | **C（一致性）** | 写入通过 OCC + CAS 保证原子性，分区期间无法提交则写入失败/重试 |
| **Else（正常运行时）** | **E（一致性）优先** | 快照隔离确保所有读者看到一致的版本；但读延迟取决于对象存储 LIST 开销 |

**本质取舍：**
- Lakehouse 在 PACELC 中偏向 **CE**（一致性优先）
- 代价是 **写入延迟**（OCC 重试）和 **读取延迟**（对象存储 LIST/GET 延迟 ~100ms 级）

### 4.3 复杂度 vs 运维简洁性

| 优势（降低复杂度） | 代价（增加复杂度） |
|-------------------|-------------------|
| 无服务器组件（表格式是 Library 而非 Service） | 需要定期维护：OPTIMIZE、VACUUM、expireSnapshots |
| 对象存储自动扩缩容，零运维 | 元数据膨胀管理（大表可能 GB 级元数据） |
| 多引擎访问无需数据复制 | 多写入者并发冲突时 OCC 重试影响吞吐 |
| 统一存储层简化数据治理 | 表格式选型（Delta vs Iceberg vs Hudi）需要长期承诺 |
| 存算分离简化资源规划 | 网络 I/O 成为瓶颈时需引入缓存层（Alluxio） |

### 4.4 成本 vs 性能

| 维度 | 低成本模式 | 高性能模式 |
|------|-----------|-----------|
| 存储介质 | 对象存储 ($0.023/GB/月) | 本地 SSD / 内存缓存 |
| 数据格式 | Parquet（高压缩） | Parquet + Z-ORDER / Liquid Clustering |
| 计算引擎 | 按需 Spot 实例 | 预留实例 + 自动扩缩容 |
| 查询优化 | 基础 Data Skipping | 物化视图 + 结果缓存 + 谓词下推 |
| 元数据管理 | 定期清理 | REST Catalog + 元数据缓存 |

**典型成本对比（以 1 PB 数据为例）：**

| 架构 | 存储成本/月 | 计算成本/月 | 总月成本 |
|------|------------|------------|---------|
| 传统数仓 (Teradata) | ~$50,000 | 包含在硬件中 | ~$50,000+ |
| 云数仓 (Redshift) | ~$25,000 | ~$15,000 | ~$40,000 |
| 存算分离 (Snowflake) | ~$23,000 | ~$20,000 | ~$43,000 |
| **Lakehouse (S3 + Spark)** | **~$23,000** | **~$5,000-15,000** | **~$28,000-38,000** |

> 注：计算成本取决于查询量。Lakehouse 在低查询量场景下优势最大（按需计算）；高查询量场景下，缓存和物化视图可显著降低成本。

### 5.3 成为标准的设计模式

1. **表格式作为独立层**：在"哑存储"和"智能计算"之间增加表格式层，成为数据平台标准架构
2. **不可变快照**：所有写入产生新快照，读取始终在一致性快照上执行
3. **乐观并发控制**：用 CAS + 重试替代分布式锁，简化并发控制
4. **数据即文件 + 元数据即日志**：数据文件不可变，元数据记录所有变更
5. **存算完全分离**：计算无状态、存储共享，成为云原生数据平台标准模式
6. **Medallion 分层**：Bronze → Silver → Gold 成为 Lakehouse 数据组织的标准模式
7. **多引擎统一访问**：BI + ML + 流处理在单一数据副本上运行

**思想传承路径：**
- **数据仓库的 ACID 传统** → 通过 OCC 适配到对象存储场景
- **Hive 的 Schema-on-Read** → 保留灵活性，增加 Schema enforcement
- **Git 的版本控制** → 快照隔离 + 时间旅行
- **微服务的解耦思想** → 存算分离、表格式与计算引擎解耦
- **Lambda 架构的教训** → 统一批流，消除双路径

### 5.5 Data Warehouse vs Data Lake vs Lakehouse 对比矩阵

| 维度 | 数据仓库 (Data Warehouse) | 数据湖 (Data Lake) | 湖仓一体 (Lakehouse) |
|------|--------------------------|-------------------|---------------------|
| **数据类型** | 仅结构化 | 任意格式 | 任意格式 |
| **Schema** | Schema-on-Write | Schema-on-Read | Schema-on-Read + Schema Enforcement |
| **ACID 事务** | ✅ 原生 | ❌ 无 | ✅ 通过表格式 |
| **查询性能** | 优（预优化、索引） | 差（全表扫描） | 接近数仓（Data Skipping + 索引） |
| **存储成本** | 高（专有存储） | 低（对象存储） | 低（对象存储） |
| **数据质量** | 高（ETL 后入库） | 低（原始数据） | 中高（Schema 治理） |
| **BI 支持** | ✅ 原生 | ❌ 差 | ✅ 通过 SQL 引擎 |
| **ML 支持** | ❌ 差 | ✅ 原生 | ✅ 直接访问原始数据 |
| **流处理** | 有限 | ✅ 支持 | ✅ 统一批流 |
| **灵活性** | 低 | 高 | 中高 |
| **运维复杂度** | 中（托管） | 低（无服务器组件） | 中（需维护表格式） |
| **厂商锁定** | 高（专有格式） | 低（开放格式） | 低（开放表格式） |
| **代表系统** | Snowflake, BigQuery, Redshift | S3+Presto, Hadoop+Hive | Databricks, Iceberg+Trino |

### 5.6 当前相关性与使用场景

**Lakehouse 在以下场景是首选：**

1. **统一数据平台**：企业需要同时支持 BI、ML、流处理，不想维护多套系统
2. **成本敏感 + 数据量大**：PB 级数据，对象存储成本远低于数仓专有存储
3. **多引擎需求**：不同团队使用不同工具（SQL / Python / Spark / Flink）
4. **数据新鲜度要求高**：需要近实时分析，消除 T+1 延迟
5. **开放生态**：避免厂商锁定，需要跨云/跨引擎兼容

**正在演化的方向：**

| 方向 | 说明 |
|------|------|
| **湖仓融合加速** | Snowflake 支持 Iceberg、Databricks 支持 Iceberg → 格式壁垒消退 |
| **流式湖仓** | Apache Paimon 将湖仓能力扩展到秒级流式场景 |
| **AI/ML 集成** | Lakehouse 直接作为 Feature Store 和模型训练数据源 |
| **治理与合规** | Unity Catalog、Apache Ranger、数据血缘追踪 |
| **Serverless Lakehouse** | Databricks Serverless、AWS Athena + Iceberg → 零运维 |
| **边缘/混合云** | Lakehouse 表格式扩展到边缘节点和混合云场景 |
