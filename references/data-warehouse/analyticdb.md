## 1. Context & Motivation（背景与动机）

### 解决的问题

阿里巴巴在 2013-2015 年间面临一个明确痛点：**离线数仓（MaxCompute/ODPS）无法满足实时分析需求**。MaxCompute 作为 EB 级离线数仓，擅长批量 ETL 和历史数据分析，但查询延迟通常在分钟到小时级别。

业务场景的驱动因素：
- **电商实时大屏**：双 11 交易大屏需要秒级刷新的聚合指标
- **运营实时监控**：库存、流量、转化率的实时看板
- **用户行为分析**：A/B 测试、漏斗分析需要亚秒级响应
- **金融级报表**：风险控制、反欺诈需要实时多维分析

### 前代系统局限

| 前代系统 | 局限 |
|----------|------|
| MaxCompute (ODPS) | 批处理模式，查询延迟分钟级；不适合交互式查询 |
| 自建 Hadoop/Spark | 运维复杂，延迟 10s 级别，MySQL 兼容性差 |
| MySQL 主库直查 | OLTP 引擎无法承受分析型扫描，拖垮线上业务 |
| 传统 MPP（Greenplum/Teradata） | 存算耦合，扩容需停机，成本高，弹性差 |

### 硬件/网络趋势

AnalyticDB 的架构设计受益于同期硬件趋势：
- **NVMe SSD 普及**：本地 SSD IOPS 突破百万，使得本地列存索引成为可能
- **RDMA/RoCE 网络**：25/100Gbps 网络降低节点间通信延迟
- **大内存节点**：单节点 256GB-1TB 内存，支持大规模向量计算
- **多核 CPU 扩展**：AVX-512 指令集为向量化执行提供硬件基础

## 2. Architecture（架构设计）

### 系统拓扑：存算分离 + 三层架构

```
                     ┌─────────────────────────────────────────┐
                     │          MySQL Protocol Endpoint         │
                     │       (JDBC / ODBC / MySQL CLI)          │
                     └────────────────┬────────────────────────┘
                                      │
                    ┌─────────────────▼─────────────────┐
                    │          Access / Proxy 层          │
                    │  ┌──────────┐ ┌──────────┐        │
                    │  │  Proxy   │ │  Proxy   │ ...    │  ← 无状态代理
                    │  │  Node    │ │  Node    │        │    负载均衡 &

                    └─────────────────┬─────────────────┘

                    │         └──────────────┘          │
                    └───────────────────────────────────┘
```

### 三层架构详解

**第一层：接入层（Access/Proxy Layer）**
- 无状态代理节点集群，水平扩展
- 实现 MySQL 协议兼容，对客户端完全透明
- 负载均衡、连接池管理、SQL 路由
- 支持标准 JDBC/ODBC 连接，无需客户端改造

**第二层：计算层（Compute Layer）**
- **Coordinator Node (CN)**：负责 SQL 解析、查询优化、分布式执行计划生成。采用基于代价的优化器（CBO），针对列存和索引特性做了深度优化。
- **Worker Node (WN)**：真正的计算执行单元。每个 Worker 独立执行其负责的数据分片的扫描、过滤、聚合操作。**计算节点无状态**——不持久化数据，可以从存储层按需拉取数据。
- Worker 之间通过 shuffle 操作交换中间结果。

**第三层：存储层（Storage Layer）**
- 底层依赖阿里云**盘古（Pangu）**分布式文件系统
- 数据按 Shard 分布，每个 Shard 包含多个列存文件（Column File）
- 每个列存文件包含多个 **Row Group**（类似 Parquet 的 Row Group 概念）
- 副本机制：通过 **Paxos 协议**保证多副本强一致性
- 支持热数据（SSD）与冷数据（HDD/OSS）分层存储

### 数据模型与命名空间

```
Database → Table → Shard → Column File → Row Group → Column Chunk
```

- **Database**：逻辑隔离单元，对应 MySQL 的 Database
- **Table**：支持主键表、明细表、分区表
- **Shard**：数据分片单位，默认按主键哈希分布到多个计算节点
- **Column File**：列式存储文件，按列物理存储
- **Row Group**：列文件中的行组（~10K-100K 行），是索引和过滤的基本单位
- **Column Chunk**：Row Group 内的单列数据块

### 元数据架构归类

**归类：外部引擎 + 原生分布式元数据混合模式**

- 全局元数据（Schema、Shard 映射、节点状态）存储在独立的**元数据服务**（Meta Service）中
- 元数据服务本身是高可用的分布式系统
- 本地元数据（索引结构、统计信息）内嵌在存储层
- 查询执行期间，CN 从元数据服务获取数据分布信息，生成最优执行计划

### 数据架构分析

#### 分块方式 (Chunking)
- **按 Shard 分片**：数据按主键哈希分布，默认 Shard 数 = 计算节点数 × 每节点分片数
- **列存 Row Group**：每个 Row Group ~10K-100K 行，是向量化执行和索引裁剪的粒度
- **压缩块**：列数据按块压缩（LZ4/ZSTD），减少 I/O 和内存占用

#### 分布策略 (Distribution)
- **Hash 分布**（默认）：按主键哈希，保证相同主键落在同一 Shard
- **Range 分布**（可选）：按主键范围分布，适合范围查询场景
- 分区表支持按时间/日期分区，加速时序查询

#### 冗余方式 (Redundancy)
- **Paxos 多副本**：每个数据 Shard 有 3 个副本（或更多），通过 Paxos 协议保证强一致
- 副本分布在不同物理机架，容忍单点/单机架故障
- 自动故障切换，副本重建对用户透明

#### I/O 路径 (I/O Path)
- 计算节点通过内部高速网络从存储层拉取列数据
- 支持**谓词下推（Predicate Pushdown）**：在存储层完成部分过滤，减少网络传输
- 本地 SSD 缓存热点数据（Buffer Pool），降低远程 I/O 延迟
- 零拷贝和 SIMD 指令优化列数据反序列化

#### 数据生命周期 (Lifecycle)
- **热/温/冷分层**：热数据存 SSD，温数据存 HDD，冷数据存 OSS
- **自动 TTL**：支持按分区自动过期清理
- **列级压缩**：不同列采用不同压缩算法（字典编码、RLE、Delta 编码等）
- **自动统计信息收集**：支持查询优化器使用实时统计信息

## 3. Core Technical Innovations（核心技术创新）

### 3.1 一致性模型：强一致性（Strong Consistency）

AnalyticDB 在 CAP 中选择 **CP**（Consistency + Partition Tolerance）：

- **写入强一致**：数据写入需通过 Paxos 多数派确认后返回成功
- **读取强一致**：读请求路由到最新副本，保证读己写（Read-Your-Writes）
- **事务支持**：支持 ACID 事务（单表事务和多表事务），保证数据更新的原子性

这与 ClickHouse 等最终一致系统形成鲜明对比。AnalyticDB 的设计目标是让业务系统**无需担心数据一致性问题**——写入立即可查。

### 3.2 共识协议：Paxos

- 存储层使用 **Multi-Paxos** 协议进行副本间数据同步
- 每个 Shard Group 有自己的 Paxos Group
- 写入流程：客户端 → Leader 副本 → Paxos 日志同步 → 多数派确认 → 返回成功
- 故障恢复：Follower 自动选举为 Leader，数据自动补齐

### 3.3 容错机制

| 故障类型 | 应对策略 |
|----------|----------|
| 单节点宕机 | Paxos 副本自动切换，计算节点无状态可重建 |
| 磁盘故障 | 副本自动在其他节点重建 |
| 网络分区 | Paxos 多数派继续服务，少数派暂停写入 |
| 计算节点扩容 | 无状态 Worker 可随时添加，存储自动重新均衡 |

### 3.4 性能优化技术

#### 3.4.1 多段自适应索引（BMAD — Block Multi-dimensional Adaptive Dictionary）

BMAD 是 AnalyticDB 的核心索引创新，是一种**多维自适应块级索引**：

```
┌──────────────────────────────────────────────────────────┐
│                    BMAD 索引层级结构                       │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer 0: 全局统计信息（Min/Max, NDV, Histogram）  │  │  ← 粗粒度裁剪
│  └──────────────────────┬─────────────────────────────┘  │
│                         ▼                                │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer 1: Shard 级索引（每个 Shard 的列范围索引）    │  │  ← 中粒度裁剪
│  └──────────────────────┬─────────────────────────────┘  │
│                         ▼                                │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer 2: Row Group 级索引                          │  │  ← 细粒度裁剪
│  │  - Min/Max 值索引（每 Row Group 的列值范围）        │  │
│  │  - Bloom Filter（高基数列的成员检测）                │  │
│  │  - 字典编码索引（低基数列的值映射）                   │  │
│  └──────────────────────┬─────────────────────────────┘  │
│                         ▼                                │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Layer 3: 稀疏索引（Sparse Index）                   │  │  ← 精确定位
│  │  - 采样点索引：每 N 行记录一个物理偏移量             │  │
│  │  - 类似 LSM-Tree 的 SSTable 索引策略                 │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**BMAD 核心特性**：

1. **自适应粒度**：根据数据分布特征（基数、倾斜度）自动选择最优索引层级
2. **多维联合索引**：支持多列联合裁剪，例如 `WHERE date = '2024-01-01' AND city = 'Beijing' AND category = 'Electronics'` 可同时利用三列索引
3. **块级操作**：索引粒度对齐 Row Group，直接跳过不匹配的整个 Row Group
4. **动态更新**：写入时自动维护索引，无需离线重建

#### 3.4.2 稀疏索引（Sparse Index）

- 稀疏索引是 BMAD 的底层索引策略之一
- 每 N 行（例如每 1024 行）记录一个采样点的物理偏移量
- 查询时通过二分查找快速定位到目标 Row Group
- 存储空间开销极小（约为数据量的 0.1%-1%）
- 特别适合大范围扫描场景：先通过稀疏索引定位，再在局部精确扫描

#### 3.4.3 向量化执行引擎（Vectorized Execution）

```
┌──────────────────────────────────────────────────────┐
│              向量化执行 vs. 逐行执行                     │
│                                                      │
│  传统逐行执行（Tuple-at-a-Time）：                     │
│    row_1 → filter → project → aggregate → result     │
│    row_2 → filter → project → aggregate → result     │
│    row_3 → filter → project → aggregate → result     │
│    ...  (高 CPU 分支预测失败率)                        │
│                                                      │
│  向量化执行（Batch-at-a-Time）：                       │
│    [row_1..row_1024] → SIMD filter  → vector_batch   │
│    vector_batch  → SIMD project → vector_batch       │
│    vector_batch  → SIMD aggregate → result            │
│    ...  (CPU 流水线利用率 > 80%)                      │
└──────────────────────────────────────────────────────┘
```

**向量化执行的核心优势**：
- **SIMD 指令利用**：使用 AVX-512 指令集对 512-bit 寄存器同时处理 8 个 64 位值或 16 个 32 位值
- **CPU 缓存友好**：列式存储天然适合缓存行预取
- **低分支预测失败**：批量处理减少条件分支
- **流水线优化**：算子之间传递 Vector Batch 而非单行元组，减少函数调用开销
- **MPP 并行**：每个 Worker 内部多线程并行执行向量算子

#### 3.4.4 列式存储与压缩

- **纯列式存储**：每列独立物理存储，分析查询只读取所需列
- **智能压缩**：
  - 字典编码（Dictionary Encoding）：低基数字符串列
  - RLE（Run-Length Encoding）：有序或重复值列
  - Delta Encoding：单调递增的数值列（如时间戳、ID）
  - LZ4/ZSTD：通用压缩
- **压缩比**：典型场景下原始数据的 3-10 倍压缩比

#### 3.4.5 CBO 查询优化器

- **基于代价的优化器（Cost-Based Optimizer）**：
  - 利用列级统计信息（Min/Max/NDV/直方图）估算扫描代价
  - 根据索引选择率选择最优执行路径
  - 支持 Join 重排序、谓词下推、聚合下推
- **自适应执行**：运行期间根据实际数据分布动态调整执行策略

### 3.5 扩展方式（Scaling）

#### 弹性计算扩展（Elastic Compute Scaling）

```
┌────────────────────────────────────────────────────────┐
│              弹性扩缩容示意                               │
│                                                        │
│  计算层：                                               │
│    [WN] [WN] [WN]         ← 初始 3 个 Worker           │
│         ▼ 扩容事件                                      │
│    [WN] [WN] [WN] [WN] [WN] [WN] [WN]                  │
│         ▲ 缩容事件                                      │
│    [WN] [WN] [WN] [WN]                                  │
│                                                        │
│  存储层：保持不变                                        │
│    [Shard0] [Shard1] [Shard2] [Shard3] [Shard4]        │
│                                                        │
│  ★ 计算与存储独立扩展，互不影响                           │
└────────────────────────────────────────────────────────┘
```

- **计算节点独立伸缩**：增加 Worker 不触发数据重分布（数据在存储层）
- **按需扩容**：业务高峰期临时增加计算节点，低峰期缩容节省成本
- **秒级弹性**：新增计算节点无需数据搬迁，立即可用
- **存储自动扩展**：存储容量随数据增长自动扩展，无需人工干预
- **规格灵活**：支持不同规格的计算节点混合使用

## 4. Trade-offs (CAP / PACELC)

### 4.1 CAP 定位：CP（强一致性 + 分区容忍性）

```
         C（一致性）
          ▲
          │
          │    ★ AnalyticDB
          │    (CP 系统)
          │
          │
  ────────┼──────────────→ P（分区容忍性）
          │
          │
          │
```

- **选择 C（Consistency）**：写入必须经过 Paxos 多数派确认，保证强一致
- **牺牲部分 A（Availability）**：网络分区期间，少数派副本暂停服务
- **选择 P（Partition Tolerance）**：分布式系统必须容忍网络分区

**与竞品的 CAP 对比**：

| 系统 | CAP 定位 | 一致性级别 |
|------|----------|-----------|
| AnalyticDB | **CP** | 强一致（Paxos） |
| ClickHouse | **AP** | 最终一致（异步复制） |
| Snowflake | **CP** | 强一致 |
| Amazon Redshift | **CP** | 强一致 |
| Apache Doris | **CP** | 强一致（Raft） |

### 4.2 PACELC 定位

```
PACELC 决策树：

  网络分区发生时 (P)？
    ├─ 选 C（Consistency）→ AnalyticDB 的选择 ✅
    │   写入暂停直到多数派恢复
    │
  正常运行时 (E vs L)？
    ├─ 选 L（Latency）→ 部分查询路径优化延迟 ✅
    │   但写入仍需 Paxos 同步
```

- **P 时选 C**：网络分区时优先保证一致性，少数派暂停写入
- **EL 时选 L**：正常运行时优化查询延迟，但写入仍需等待 Paxos 同步

**这意味着**：AnalyticDB 在写入场景下存在**同步延迟开销**（通常 < 10ms，取决于网络 RTT）。这是为强一致性付出的代价。

### 4.3 复杂度 vs 运维简易性

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| **架构复杂度** | 高 | 三层架构 + Paxos + BMAD 索引 + 向量化引擎 |
| **运维复杂度** | 低（对用） | SaaS 托管，自动扩缩容、备份、监控 |
| **接入复杂度** | 极低 | MySQL 协议兼容，零代码改造 |

AnalyticDB 的核心设计哲学是：**将复杂性封装在平台内部，对外提供极简的 MySQL 接口**。用户只需写标准 SQL，无需关心分布式执行细节。

### 4.4 成本 vs 性能

| 场景 | 成本特点 | 性能特点 |
|------|----------|----------|
| **按量付费（Serverless）** | 按计算节点运行时间 + 存储容量计费 | 弹性最佳，按需付费 |
| **包年包月（Reserved）** | 预付费用，单价更低 | 稳定计算资源，适合可预测负载 |
| **冷热分层** | 热数据贵但快，冷数据便宜但慢 | 自动分层优化性价比 |

**成本优势**：存算分离使得计算和存储可以独立优化。冷数据不需要高性能计算资源，只需存储成本。

**性能代价**：相比 ClickHouse 等 AP 系统，AnalyticDB 的写入性能（Paxos 同步）和查询性能（强一致读取）有一定开销，但换来了一致性保证。

## 5. Influence & Legacy（影响与遗产）

### 5.1 启发的后续系统

- **Hologres**：同样基于阿里巴巴内部的实时分析引擎演进，但采用 HSAP（Hybrid Serving/Analytical Processing）架构，与 AnalyticDB 形成互补
- **阿里云其他 OLAP 产品**：AnalyticDB 的技术积累直接影响了阿里云的实时数仓产品线
- **StarRocks / Doris**：虽然技术路线不同，但在向量化执行和 MPP 架构上受到类似趋势驱动

### 5.2 成为行业标准的设计模式

1. **存算分离架构**：从 AnalyticDB 到 Snowflake，存算分离已成为云数仓的标准模式
2. **向量化执行**：ClickHouse、Doris、StarRocks 全部采用向量化执行引擎
3. **多维自适应索引**：BMAD 的思想（多层索引、自适应裁剪）影响了后续列存数据库的索引设计
4. **MySQL 协议兼容**：降低迁移成本，成为云 OLAP 的事实标准
