# 数据仓库（Data Warehouse）演进总览

## 什么是数据仓库

数据仓库 = 面向分析的集中式数据存储 + SQL 查询接口 + 预优化 + 历史数据

```
OLTP 系统 (MySQL/PostgreSQL)
         ↓ ETL/ELT
    ┌──────────┐
    │ 数据仓库  │  ← 列式存储 + MPP 执行 + 预建模
    └────┬─────┘
         ↓
   BI / 报表 / 分析
   (Tableau/Superset)
```

**数据仓库 vs 数据湖 vs 湖仓一体**

| 维度 | 数据仓库 | 数据湖 | 湖仓一体 |
|------|----------|--------|----------|
| 数据格式 | 结构化 | 任意格式 | 任意格式 |
| Schema | Schema-on-Write | Schema-on-Read | Schema-on-Read + 治理 |
| 查询性能 | 优 | 取决于引擎 | 接近数仓 |
| 数据质量 | 高 | 低 | 中高 |
| 成本 | 高 | 低 | 中 |
| 灵活性 | 低 | 高 | 中高 |
| 适用场景 | BI/报表 | 探索/ML | 统一分析平台 |

## 演进脉络

```
传统数仓 (1990s) → MPP 数仓 (2000s) → 云数仓 (2010s)
→ 存算分离 (2015) → 实时数仓 (2018) → Serverless (2020)
```

## 核心驱动力

1. **分析需求增长**：OLTP 不适合复杂分析，需要专用分析系统
2. **数据量增长**：TB → PB 级，需要 MPP 架构
3. **云计算**：从买硬件到按需付费
4. **实时分析**：T+1 → 分钟级 → 秒级

## 技术演进主线

### 架构
- 单机 → MPP (大规模并行) → 云托管 → 存算分离 → Serverless

### 存储格式
- 行式 → 列式 (Parquet/ORC) → 压缩编码 → 向量化

### 执行引擎
- 火山模型 → 向量化执行 → MPP → GPU 加速

## 时代划分

### 第一代：传统数据仓库 (1990s-2008)

| 系统 | 核心特征 |
|------|----------|
| Teradata | MPP 开创者、专用硬件、并行优化器 |
| Oracle Exadata | 一体机、In-Memory 列缓存 |
| Greenplum | 基于 PostgreSQL、MPP、开源版 |
| Netezza | 硬件加速、FPGA 过滤 |

**特点**：专用硬件、价格昂贵、垂直扩展为主

### 第二代：云数据仓库 (2008-2015)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| Amazon Redshift | 2012 | 基于 PostgreSQL、列式存储、云托管 |
| Google BigQuery | 2010 | Serverless、Dremel 引擎、按需付费 |

**特点**：云托管、降低运维、但存算耦合

### 第三代：存算分离云数仓 (2015-2020)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| Snowflake | 2015 | 存算分离开创者、多集群共享、零复制克隆 |
| Databricks | 2013+ | Delta Lake + Spark、湖仓一体 |

**特点**：存算独立扩展、按需计费、多租户隔离

### 第四代：实时数仓 / 开源替代 (2018+)

| 系统 | 年份 | 核心创新 |
|------|------|----------|
| ClickHouse | 2016 | 极致 OLAP 性能、列式+向量化+稀疏索引 |
| Apache Doris | 2018 | MPP、FE/BE 架构、高并发点查 |
| StarRocks | 2020 | Doris 分支、向量化+CBO+湖仓分析 |

**特点**：开源、实时、高性能、可自建

## 核心技术对比

| 维度 | Redshift | Snowflake | BigQuery | ClickHouse | Doris/StarRocks |
|------|----------|-----------|----------|------------|-----------------|
| 架构 | 存算耦合 | 存算分离 | Serverless | 存算一体 | FE/BE MPP |
| 存储 | 列式 | 列式 (微分区) | 列式 (Capacitor) | 列式 | 列式 |
| 扩展性 | 手动 | 自动 | 完全自动 | 手动 | 在线扩展 |
| 并发 | 中 | 高 (多集群) | 高 | 中 | 高 |
| 延迟 | 秒级 | 秒级 | 秒级 | 亚秒 | 亚秒 |
| 成本 | 中 | 中-高 | 按查询量 | 低 (自建) | 低 (自建) |
| SQL 兼容 | PostgreSQL | 定制 SQL | BigQuery SQL | SQL 子集 | MySQL 协议 |
| 实时写入 | 批量 | 流式 (Snowpipe) | 流式 | 批量+流式 | 批量+流式 |

```
┌──────────────────────────────────────────┐
│            应用层                         │
│  BI │ 报表 │ 数据科学 │ 实时大屏           │
├──────────────────────────────────────────┤
│           计算层 (Compute)                │
│  MPP 集群 │ 多虚拟仓库 │ Serverless        │

│  列式存储 │ 云对象存储 │ 本地 SSD          │
```

## 关键技术概念

### 列式存储 (Columnar Storage)
- 按列存储，而非按行
- 分析查询只需读取相关列 → I/O 大幅减少
- 同列数据相似性高 → 压缩效果好

### MPP (Massively Parallel Processing)
- 查询被拆分到多个节点并行执行
- Shuffle 阶段是性能关键
- 数据分布键（Distribution Key）影响性能

### 向量化执行 (Vectorized Execution)
- 批量处理数据（SIMD 指令）
- 比逐行处理快 5-10x
- ClickHouse/StarRocks 极致性能的核心

### 物化视图 (Materialized View)
- 预计算结果存储
- 加速常见查询
- 需要维护一致性

## 湖仓一体趋势

```
传统演进：数据仓库 ←→ 数据湖 (各自独立)

现代趋势：湖仓一体 (Lakehouse)
  - 数据湖的灵活性 + 数仓的性能/治理
  - 统一存储层（对象存储）
  - 统一表格式（Iceberg/Delta/Hudi）
  - 多引擎访问（SQL/ML/流处理）
```
